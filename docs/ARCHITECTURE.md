# Architecture

RunTypes is a compile-time **type resolver** targeting **TypeScript 7 / tsgo**. It provides a native side-channel into tsgo's type checker for tools (Vite plugin, codegen, test harness) that need to know a TypeScript type at a specific call site without relying on the legacy custom-transformer API (which the Go port does not expose; see [microsoft/typescript-go#516](https://github.com/microsoft/typescript-go/issues/516)).

## Big picture

```
  .ts source                   Go resolver                  JSON type table
  ┌─────────┐    scanFiles    ┌──────────────┐   TypeNode   ┌───────────────┐
  │  app.ts │ ─────────────▶ │  ts-runtypes │ ───────────▶ │  site → id    │
  └─────────┘                │  (typescript- │              │  id   → node  │
       ▲                     │   go checker) │              └───────────────┘
       │ rewrites calls      └──────┬───────┘                       │
       │                            │                               │
  ┌─────────┐                       │ spawns (stdio JSON)           │ virtual
  │  Vite   │ ◀─────────────────────┘                               │ module
  │  plugin │ ────────────────────────────────────────────────────▶ │ import
  └─────────┘                                                       ▼
                                                             runtime / RT
```

### Three lifecycles

1. **Build lifecycle** (Vite, Rollup, CI codegen): Go resolver is spawned once, receives one `scanFiles` query per source file, dumps the full table at end-of-build, is torn down.
2. **Query lifecycle**: every source file with calls to `InjectRunTypeId<T>`-marked functions is sent to `scanFiles`. The resolver walks every CallExpression, asks tsgo for the resolved signature, and returns one site per call whose trailing parameter type is the sentinel marker.
3. **Runtime lifecycle**: the rewritten source passes the site id as the trailing argument; the library's runtime helper does a `Map.get(id)`. No reflection work happens at runtime.

## Execution model — what we replace and what we don't

In the **bundler path** we **do not replace `tsc`**, and we do not implement a compiler. The Vite plugin is a normal `transform()` plugin running inside Vite/Rollup's pipeline, and the Go binary is a separate process used solely for _type resolution + the call-site rewrite_ — the bundler emits the JavaScript.

> **The `--compile` command is the exception (and the only one).** For a host with no bundler plugin, `ts-runtypes --compile` is a batch, tsc-like build that DOES emit `.js`: it applies the rewrite, then drives tsgo's own `Emit` on the rewritten source (honoring the tsconfig `outDir`/`target`/`module`/`sourceMap`), and composes the two source maps back to the original TypeScript. It reuses the same rewrite; only the emit + map-composition are extra ([ts-go-runtypes/internal/compiler/batchcompile](../ts-go-runtypes/internal/compiler/batchcompile/), [compose.go](../ts-go-runtypes/internal/compiler/sourcerewrite/compose.go)). The table below describes the bundler path.

Concretely, at build time (bundler path):

| -------------------------------------- | --------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

### Why a separate Go binary

`tsc` (the JavaScript build) shipped a [custom-transformer API](https://github.com/itsdouges/typescript-transformer-handbook) that runtime-type libraries (`ts-runtime`, `typia`, `deepkit`, and others) hooked into. That API let a Node-side program walk the AST and rewrite source _with access to the checker_ before `tsc` emitted JS.

`tsgo` — the Go port that will become TypeScript 7 — **does not currently expose any plugin or transformer API to user code**. The compiler is now a single statically-linked Go binary that runs as a CLI; you cannot load a Node plugin into it, and the checker's symbol-resolution machinery lives behind Go's `internal/` visibility wall (which Go enforces at the compiler level — see the "Shims" section below).

Tracking issue: [microsoft/typescript-go#516 — Plugin support](https://github.com/microsoft/typescript-go/issues/516). As of this writing, no roadmap.

That leaves two choices for type-aware tooling against TypeScript 7:

1. **Wait** for an official API. Unbounded timeline.
2. **Reach into tsgo's checker at the Go level** via a side-channel. That's what this project does, using the [oxc-project/tsgolint](https://github.com/oxc-project/tsgolint) shim layer (also used by oxc's TypeScript linter for the same reason). When #516 lands, we swap the shim for the official API and leave the Vite plugin, marker, cache, and runtime untouched.

### Where each process runs

```
┌───────────────────────────────────────────────────────────────────────┐
│  Vite dev server  /  rollup build  (Node.js)                          │
│                                                                       │
│  ┌────────────────┐   transform(file)    ┌────────────────────────┐   │
│  │  Vite core     │ ───────────────────▶ │  @ts-runtypes/devtools │   │
│  │  (esbuild)     │                      │  (Node)                │   │
│  │                │   ◀── rewritten src  └─────────┬──────────────┘   │
│  └────────────────┘                                │ stdio JSON       │
│                                                    ▼                  │
└───────────────────────────────────────────────┬───────────────────────┘
                                                │
                            spawn() one child   │
                                                ▼
                                  ┌──────────────────────────┐
                                  │  bin/ts-runtypes  (Go)  │
                                  │  ─────────────────────   │
                                  │  tsgo Program + Checker  │
                                  │  via shim/* imports      │
                                  └──────────────────────────┘
```

The Go binary is a long-lived child process during a build session: spawned once, fed one `scanFiles` request per `.ts` file, dumped at the end of the build, then torn down. The plugin can also run it as a Unix-socket daemon for HMR scenarios where the build session outlives a single Vite invocation.

The user's tsconfig drives both worlds: the binary parses it to bootstrap the same `Program` view tsgo would use, so what we resolve as type `T` is exactly what tsgo resolves.

### Configuration surface

The same tsconfig is also the canonical config surface for the compiler's project options. On the build path (`ts-go-runtypes/cmd/ts-runtypes/main.go`) the binary reads the `compilerOptions.plugins[name=ts-runtypes]` entry (`resolveBuildPlugin` in `config.go`) and merges it under the CLI flags with **tsc-style precedence** — an explicitly-set flag (tracked via `flag.Visit`) wins over the tsconfig entry, which wins over the built-in default (`mergeBuildOptions` in `buildconfig.go`). The host plugins therefore forward a `--flag` only for an option the user set explicitly, so an unset host option falls through to tsconfig. The inline / server test modes carry no tsconfig, so they run on flags + defaults alone. Recognised keys: `emitMode`, `moduleMode`, `inlineMode`, `hashLength`, `runTypesGenDir` (where the `--compile` batch command writes its generated cache modules — resolved flag > tsconfig > `<cwd>/__runtypes`, `resolveRunTypesGenDir` in `buildconfig.go`), `singleThreaded`, `parallelScan`, `parallelRender`, `genDir`, `i18n` (the enrichment mirror root and the FriendlyText translation config, consumed by the out-of-band `describe`/`gen`/`check` verbs — see [docs/AI_ENRICHMENT.md](./AI_ENRICHMENT.md) and [docs/done/friendly-type-i18n.md](./done/friendly-type-i18n.md)); an unknown key is ignored with a stderr warning (the known set is derived from the struct's json tags by reflection, so it can't drift). Newly-exposed options need no disk-fingerprint bump: they change site detection, module grouping, or the cache location rather than a cached entry's body (the body-affecting `hashLength` / `emitMode` / `inlineMode` were already folded in).

There is deliberately **no `cacheDir` key** (nor a `--cache-dir` flag). The on-disk RT artifact cache follows TypeScript's own `incremental` / `composite` switch rather than a knob of ours: the resolver reads `program.IsIncremental()` (surfaced from the parsed `CompilerOptions`, so an `incremental`/`composite` inherited through `extends` counts) and, when true, caches under the canonical `<cwd>/node_modules/.cache/ts-runtypes`; when false it doesn't cache at all — matching the mental model users already have from `tsc`, and matching how the inline / server modes (no tsconfig) stay cache-off. The enable+locate decision lives in `resolver.cacheLocation` (fed by `Options.CacheFollowsIncremental`); the internal `RT_CACHE_DIR` env var is the only override (a path forces the cache on there, an empty string forces it off) — retained for the disk-cache end-to-end tests and direct-binary power users, and forwarded per-spawn as child env by `ResolverClient` so parallel test spawns stay isolated. Staleness needs no active pruning: entries are content-addressed (the binary version folds into every typeID hash and non-version options fingerprint into a subdir), so a changed type simply writes a new entry and the old one is never read again — the same non-pruning behaviour as `tsc`'s `.tsbuildinfo`, bounded by `node_modules/.cache` being wiped by standard `clean` recipes.

## The sentinel marker

Detection is anchored on a single TypeScript type alias exported from the `@ts-runtypes/core` package:

```ts
export type InjectRunTypeId<T> = string & {readonly __rtInjectRunTypeIdBrand?: T};
```

A function opts into compile-time id injection by declaring `id?: InjectRunTypeId<T>` as its **trailing parameter**. The transformer rewrites every call site of such a function, injecting the resolved hash id at that slot. This includes:

- The static helper `getRunTypeId<T>(id?)` shipped from `@ts-runtypes/core` — explicit type, no value.
- The reflection helper `getRunTypeId<T>(value, id?)` — `T` inferred from a runtime value.
- Any user-defined wrapper that propagates the marker — `function validate<T>(v, id?: InjectRunTypeId<T>)`.

Detection requires both:

1. The trailing parameter type alias must be named `InjectRunTypeId`.
2. The alias must be declared in `@ts-runtypes/core`. Either inside `declare module "@ts-runtypes/core" { ... }` or in a file whose enclosing on-disk `package.json` has `"name": "@ts-runtypes/core"`. This rules out accidental collisions with same-named user types declared elsewhere. The marker set is fixed at the binary's defaults — there is no CLI override for either knob.

A call inside a generic body where the marker's `T` is the wrapper's own free type parameter is **skipped** — there's no concrete `T` to assign an id to yet. The wrapper must propagate the marker via its own signature, and the injection happens at the wrapper's call sites instead.

### The second marker — `InjectTypeFnArgs<T, Fn>` and demand-driven caches

`InjectRunTypeId<T>` covers **reflection-only** sites (`getRunTypeId` static + value-first forms, value-first RT builders, `createMockData`) and injects a bare `"<typeId>"` string. The `createX<T>()` factory family (`createValidate`, `createGetValidationErrors`, the unknown-keys group, `createJsonEncoder`/`createJsonDecoder`, `createBinaryEncoder`/`createBinaryDecoder`) instead declares a trailing `InjectTypeFnArgs<T, Fn>` marker, whose second type argument `Fn` names the function family (`'val'`, `'verr'`, `'jsonEncoder'`, …). For those sites the transformer injects a `["<typeId>", "<fnHash>"]` **tuple**:

- `typeId` — the structural-id hash of `T`, same as the reflection cache.
- `fnHash` — an **opaque precomputed hash** (length 3, `hash("op|" + operationName + sorted comptime-args)`) the scanner computes from `Fn` plus the relevant `CompTimeFnArgs` literal (the `ValidateOptions` bag for `it`/`te`; the JSON strategy for the encoder/decoder), via the operations registry in [`ts-go-runtypes/internal/cachegen/operations`](../ts-go-runtypes/internal/cachegen/operations/) (`FnHashFor`/`PlainHash`/`Canonical`). Go computes every fnHash — the runtime treats it as an opaque lookup-key prefix and never hashes anything. The injected tuple is the complete demand: it tells the backend exactly what to emit and gives the runtime the exact lookup key, so the runtime no longer re-derives a variant key. The cache key is `<fnHash>_<typeId>`. The canonicalizer is property-order-independent (like the structural type-id, which already sorts object members).

  The fnHash salt is **version-independent** — it namespaces operation hashes away from structural type-ids with a constant `"op|"` infix but folds in NO binary version (unlike the `typeId`, which folds `constants.Version` into every structural hash). Cross-version cache invalidation rides entirely on the `typeId` half of the composite key (and the version-folded typeID disk directory); folding the version into the fnHash too would be redundant and would make the per-family fnHash **prefix move on every release**, forcing any consumer that maps `family → prefix` (e.g. mion's `JIT_FUNCTION_IDS`) to re-pin those constants on each bump. Keeping the salt version-free makes every prefix stable, so a consumer derives it once. That derivation is public: **`getFnHash(fnKey, options?)`** (exported from `ts-runtypes`, backed by the Go-generated [`fnHashes.generated.ts`](../packages/ts-runtypes/src/go-generated/fnHashes.generated.ts) table so no hashing runs in JS) resolves the same `fnKey (+ compile-time options) → fnHash` the plugin injects, letting a framework rebuild `<fnHash>_<typeId>` from a type's injected `typeId` alone (`getFnHash('val') + '_' + typeId`) without the injected function tuple.

**Several families in one marker.** `InjectTypeFnArgs<T, F1, F2, …>` names any number of families after `T` — a framework wrapper such as mion's `route()` asks for `'verr'`, `'jsonDecoder'`, `'jsonEncoder'` in a single trailing marker. `marker.fnKeysFromAlias` reads every type argument after `T` (in declaration order), the scanner computes one `[typeId, fnHash]` per family (carried on `protocol.Site.FnIds`), and the transformer injects an **array** of those tuples at the trailing slot (`buildInsertion` renders `[__rt_a, __rt_b, …]` when `len(FnIds) > 1`, else the bare single binding); the wrapper destructures the array positionally (`fns?.[0]`, `fns?.[1]`, …) and forwards each handle to its factory. `createStandardSchema`'s `'val'`+`'verr'` is the built-in two-family case. The alias declares a generous fixed arity (`F1`…`F12`, comfortably above the ~11 public families) because a TS type alias cannot be variadic. Combined with the duplicate rule below, that makes the effective limit the number of distinct families. Naming the same family twice is rejected with **MKR006** (`resolver.dedupeFnKeys` in [`ts-go-runtypes/internal/compiler/resolver/scan.go`](../ts-go-runtypes/internal/compiler/resolver/scan.go)) — a repeat would inject an identical redundant handle nothing reads — and the scanner dedupes before emitting so the injection stays sound even if a host surfaces the diagnostic as non-fatal.

**Recovering a family without a factory.** A wrapper forwards each injected handle to that family's `createX` factory — but the JSON value-level primitives (`'pj'`/`'pjs'`/`'rj'`/`'sj'`/`'ukuw'`/`'cj'`/`'cjr'`, the per-strategy `prepareForJson`/`restoreFromJson` building blocks the encoder/decoder composites wrap) have no dedicated factory. The generic `getRTFunction<K>(injected)` resolver ([`packages/ts-runtypes/src/createRTFunctions.ts`](../packages/ts-runtypes/src/createRTFunctions.ts)) recovers ANY of them from the injected tuple — it registers the tuple's closure and returns `entry.fn` by key, exactly like a `createX` does internally, but family-agnostic (the tuple key already encodes which function). It is keyed by the SAME fnKey the marker names (`getRTFunction<'pjs'>(fns?.[0])`), and the `RTFunctionByKey` map turns that key into the return type — no need to hand-write the fn type. So a framework that owns its own JSON envelope (mion, decoding one request body and encoding one response per route) names the per-strategy primitive it wants in the marker and resolves the slot with `getRTFunction` — parse once, `restore` each value; `prepare` each value, stringify once — without a string round-trip and without a factory per primitive. Marker resolution never gated `Fn` on whether a factory exists, so this was already wired end-to-end; `getRTFunction` is the missing public resolver.

The `[typeId, fnHash]` literals are computed by `CompTimeFnArgs<T>` — the fn-selecting variant of `CompTimeArgs<T>` (both in [`packages/ts-runtypes/src/markers.ts`](../packages/ts-runtypes/src/markers.ts)): it validates literals identically (CTA0xx) but also marks the parameter whose literal value selects the `createX` variant. Plain `CompTimeArgs<T>` stays for other literal params (pure-fn keys, builder configs).

Both markers accept a **spread of a statically-resolvable container fragment** inside their object / array literal — `object({...base, name: string()})`, `createJsonDecoder<T>({...preset, strategy: 'mutate'})`. The operand must resolve (via the const-chain trace) to a literal container of the matching kind: an object literal for an object spread, an array literal for an array spread; a dynamic value or a shape mismatch stays a CTA0xx error. The spread-operand trace follows **import aliases** (`comptimeargs.ResolveSpreadContainer`, the same cross-module hop the regex-literal trace uses), so an imported shared fragment merges too. For builders the merge is free — TypeScript performs the type-level spread, so the scanner reflects the merged type with no Go-side value-merge. For the `CompTimeFnArgs` option bags the value IS read from the AST, so `eachOptionProperty` (in [`ts-go-runtypes/internal/compiler/resolver/scan.go`](../ts-go-runtypes/internal/compiler/resolver/scan.go)) descends into spreads **in source order, last-write-wins** — a spread-merged options preset selects exactly the same fnHash variant as the fully-inlined equivalent (the soundness contract: relaxing the validator and teaching the option reader to merge land together).

**Whole-const args resolve cross-module too**, for parity with the spread form. A WHOLE imported `const` used as a builder child or option bag — `object({inner: importedConst})`, `createValidate(undefined, importedPreset)` — resolves through its import alias, not just `{...importedConst}`. The literal-value walk (`traceIdentifier`) uses `resolveConstInitializerCrossModule`, and `eachOptionProperty` follows an identifier options bag through the same hop; the same-module-only `resolveConstInitializer` is kept for the pure-fn / string-literal traces, which deliberately do NOT cross modules. The guard that keeps this sound is the **`as const` rule (CTA004)**: when a comptime arg resolves to a `const` bound to an **object literal**, the const's TYPE must carry literal values — a widened member (`{strategy: 'mutate'}` inferred as `{strategy: string}` for want of `as const`) is rejected, because the build reads the value from the AST while TypeScript resolves the overload against the type, and a widened type lets the two pick different fn variants. The check is shallow and object-only: a property typed as an object (a value-first builder result like `number()`) is left alone, so only flat option bags / literal-valued objects trip it.

Because each function site now carries structured demand, the per-family cache modules are **demand-driven**: a family emits factories only for the types its own `createX` call sites request. A hash isn't reversible, so demand is no longer reverse-parsed from `fnId` — the scanner computes it and carries it on `protocol.Site.Demand []SiteDemand{FamilyTag, VariantSuffix, Options, FnHash}`, closed transitively by `collectFamilyDemand` in [`ts-go-runtypes/internal/cachegen/typefunctions/module.go`](../ts-go-runtypes/internal/cachegen/typefunctions/module.go). All function families are demand-driven (no "migrated families" gate — the gate is simply `len(dump.Sites) > 0`). A file whose only marker call is `getRunTypeId<T>()` emits **zero** function-cache entries — the reflection cache is untouched, every function family is empty for it.

The renderable RT "operations" — the 11 `createX`-backed ops + the 7 JSON value-level primitives (`pj`/`pjs`/`rj`/`sj`/`ukuw`/`cj`/`cjr`) — and the fnHash machinery (`FnHashFor`/`PlainHash`/`Canonical`/`DemandFor`, `FnHashLen = 3`, plus an `init()` collision guard that fails the build if the closed operation/option set ever collides at that length — a collision is an internal bug, never auto-grown) live in the single-source-of-truth package [`ts-go-runtypes/internal/cachegen/operations`](../ts-go-runtypes/internal/cachegen/operations/). Every operation is now `Public` (marker-recoverable): the `createX`-backed ops via their factory, the primitives via `getRTFunction`. `Public` still gates the `overrideX` path (only a Public op can be overridden), and it no longer means "has a factory".

JSON encoder/decoder composition is **Go-emitted**, not assembled at runtime: one COMPOSITE cache entry per (typeId, strategy), keyed by the composite fnHash (see [`ts-go-runtypes/internal/cachegen/typefunctions/json_composite.go`](../ts-go-runtypes/internal/cachegen/typefunctions/json_composite.go)), wraps the underlying primitives with native JSON. So `createJsonEncoder`/`createJsonDecoder` collapse to the same pure `resolveTupleEntry` lookup as binary — no runtime strategy branching. Per-strategy composite tags live in `constants.jsonCompositeTags` (deliberately NOT in `CacheModules`, so the generated TS mirror is untouched). The encoder strategy set is `clone` | `mutate` | `direct` | `compact` — `clone` (default) is shape-derived and strips undeclared keys by construction (wraps `prepareForJsonSafe`), so there is no separate strip variant; `mutate` transforms in place and preserves extras; `direct` is single-pass; `compact` emits each object's declared props as a positional array (no key names on the wire, wraps the new `compactForJson`/`cj` walking primitive) and strips extras like `clone`, paired with a `compact` decoder (`compactFromJson`/`cjr`) that rebuilds the keyed object from positions. Absent optionals ride a `null` placeholder. Any object carrying an index signature (a record, or a fixed object with dynamic keys) stays keyed — only fixed-shape objects with no index signature are tupled (a nested fixed object inside a record still goes positional). Unlike `clone`/`mutate`/`direct` (which reuse keyed-object primitives), `compact` adds two real type-walking primitive families (`cj`/`cjr`, in `CacheModules`); its composite tags are `jeCO`/`jdCO`. `cj`/`cjr` DELEGATE their diagnostic codes exactly as they delegate emit and noop predicates — `cj → prepareForJsonSafe` (PJS\*), `cjr → restoreFromJson` (RJ\*) — so an unserialisable leaf (function / symbol) at a propagating position renders an `alwaysThrow` entry with the sibling's exact per-family code instead of silently skipping (which would leave the composite binding a never-rendered primitive — the JCP001 internal breach `AssertCompositeSoftDeps` guards, now anchored at the demanding call site).

Disk cache format is **v15** (keys embed fnHash; the payload is the tuple `ArgsText` for per-entry modules — with default-valued slots rendered as JS array holes, interior ones included, see `typefunctions.holeifyArgs` — plus a persisted `IsNoop` bit and `PureFnRefs`). v15 adds `PureFnRefs` — the entry's demand-driven built-in pure-fn edges (`rt::newRunTypeErr`, `rt::findCycle`, …) as stable `<ns>::<fn>` keys — so a warm entry rebuilds its `SoftDeps` pure-fn imports off the cache instead of dropping them (a runtime `getPureFn(...) === undefined`); v14 payloads lack the field and must miss. v14 drops `constants.Version` from the fnHash salt: the fnHash prefix baked into every cached key slot changes WITHOUT the version-folded typeID directory moving, so a same-version rebuild must miss v13 payloads (whose key slots carry the old prefix) rather than register an entry the resolver keys under the new prefix — the same failure mode as the v2→v3 fnHash naming flip. v13 adds the class-name argument to emitted class-serializer lookups (`utl.getClassSerializer('<id>', '<className>')`, the name-fallback lane that lets one registration cover every generic instantiation of a class), so stale single-arg payloads must miss. v12 made the noop verdict predicate-decided (`IsNoopType` over the type graph; the compiled body's shape survives only as the renderer's protective tripwire), so v11 IsNoop bits must miss. v11 extends noop elision to the entries themselves: all-elided JSON composites emit the noop short-form tuple and stringifyJson flags its native `return JSON.stringify(v)` roots, so v10 payloads (stale full bodies / isNoop verdicts) must miss. v10 embeds the `alwaysThrow` runtime throw message directly in the entry's final tuple slot (`[CODE] Cannot … (at site)`, rendered by the Go emitter) so the shipped marker package throws it without carrying a diagnostic catalog; the Go↔plugin wire still carries only the diagnostic code, and the build-time diagnostic catalog (code → headline, for the build log / IDE) lives once in @ts-runtypes/devtools. The plugin's **`emitMode`** option (`code` | `functions` | `both`; mirrored as the binary's `--emit-mode` flag, validated against `constants.EmitMode`) selects what each fn entry ships in its code/factory slots: `code` (default) ships only the body string (runtime rebuilds the factory via `new Function`); `functions` ships only the live `createRTFn` closure (code derived lazily from `createRTFn.toString()` via `entryCode` if ever read); `both` ships both (the body twice — for CSP runtimes that also read `.code`). **Pure-fn entries honor the same gating** (`purefunctions.CollectEntries`, symmetric with the type-fn `codeArg`/`createRTFnArg`): `code` ships the pure fn's `code` string and drops the live `createPureFn` (the runtime rebuilds it via `new Function(...paramNames, code)` — `initPureFunction` in [`rtUtils.ts`](../packages/ts-runtypes/src/runtypes/rtUtils.ts)), `functions` drops the `code` string, `both` ships both. So a `code`-mode build no longer ships every pure-fn body twice, and the on-demand built-in bodies (`rt::findCycle`, the format validators, …) follow the same rule instead of being unconditionally live. The disk fingerprint ([`ts-go-runtypes/internal/cachegen/diskcache/fingerprint.go`](../ts-go-runtypes/internal/cachegen/diskcache/fingerprint.go), tag v8) folds `emitMode` in so the three modes never cross-read; pure fns are rendered per session, not per-entry disk-cached, so they carry no new cross-read surface. The [compile-time benchmark](../container/benchmarks/README.md#compile-time-cost-benchcompiletime) wipes this cache before its transform build, so it measures the honest from-scratch resolve rather than a cache hit.

`it` (validate) is the one **cross-family** dependency: the JSON and binary union decoders discriminate members at runtime via `val_<member>.fn(value)` and `validationErrors` delegates child checks to `val_` too. Those edges ride each entry's module dependencies (`SoftDeps` — imported but never cascade-dropped, because the emitted bodies guard the lookups with `?.fn(…) ?? true`), and the resolver renders the referenced foreign entries to fixpoint (`resolveCrossFamilyEdges` in [`ts-go-runtypes/internal/compiler/resolver/dispatch.go`](../ts-go-runtypes/internal/compiler/resolver/dispatch.go)). A file that only serializes a union therefore still gets the per-member `val_` entries its decoder needs at runtime — they arrive through the union entry's import closure.

**Custom per-type overrides — `overrideX<T>(pureFn)`** are the WRITE side of the same `(family, typeId)` routing `createX` reads. A user registers a custom **pure function** for one `T`; every `createX<T>()` then returns it instead of the Go-emitted body. The override fn rides the existing pure-fn pipeline (purity gate, body hashing, cache, virtual-module emission) as a **`cfn`** — keyed `cfn::<CodeHash(body)>`, content-addressed so identical bodies dedup. The cfn's hash is then **folded into the structural type id** of `T` as a `|cfn:<family>:<hash>` suffix (mirroring the format-annotation fold in [`ts-go-runtypes/internal/cachegen/runtype/typeid/`](../ts-go-runtypes/internal/cachegen/runtype/typeid/)). Folding rather than swapping the entry body is what keeps the cache **idempotent**: an overridden type gets a DISTINCT id from its un-overridden twin, so no `<fnHash>_<typeId>` key is ever reused with two different bodies across builds, and because type ids propagate structurally the override reaches **every containing type** (`overrideJsonEncoder<string>` shifts `{a, b: string}`'s id and the body emitted for its `b` field). A whole-program collection pass (`ensureOverrides` in [`ts-go-runtypes/internal/compiler/resolver/overrides.go`](../ts-go-runtypes/internal/compiler/resolver/overrides.go)) runs **before any id is minted** — the map is global, an override anywhere shifts ids everywhere — extracting cfns and installing the override map on the cache (`Cache.SetOverrides`). At emit time the type-fn entry for an overridden `(family, typeId)` is a thin **redirect** (`buildRedirectEntry`): `return utl.usePureFn('cfn::<hash>')`, with the cfn on `SoftDeps`; the walker forces a dependency call at an overridden child so containing types reference the redirect rather than inlining the structural body. Overriding a JSON composite redirects every strategy tag and **prunes** the now-dead internal primitives (pj/sj/rj/ukuw) for that type — which is also the escape valve for a wire format the structural emitter can't produce (the primitive would `alwaysThrow` otherwise). Diagnostics: **OVR001** (two overrides of one `(type, family)` with different bodies — order-dependent, an error; same body dedups silently), **OVR002** (a build tripwire: `AssertOverrideCfn` fails the build if a redirect's cfn module didn't render, rather than a runtime `usePureFn` throw), **OVR010** (a Warning that overriding `validate` — the cross-family-shared family above — also changes how decoders narrow unions containing `T`). The override fn's own purity reuses the `PureFunction` marker layer (PFN001 not-inline / PFN002 external-handle / PFE9006-9011 purity). The **literal-only rule** governs every `PureFunction<F>` literal — the build extracts and AOT-compiles the body, so the only accepted form is an **inline** arrow / function expression (modulo `as`/parens/`satisfies` wrappers). A named reference is rejected even when it's a module-private `const f = …` / `function f(){}` (PFN001 "inline it"), because any named binding is a handle something else could reach; an imported or exported binding is the same problem made worse and gets the more specific PFN002. Detection: inline literal → accept; imported symbol (`SymbolFlagsAlias`) → PFN002 "imported"; a local binding that's exported (`export` modifier, `Symbol.ExportSymbol`, or an `export {…}` specifier resolving back to it) → PFN002 "exported"; any other named/non-function node → PFN001. Composition between pure-fns stays available through the factory's `utl` (`utl.usePureFn('ns::id')`, tracked as a dependency), so the literal-only rule never blocks reuse — it only forbids a raw value handle. Runtime is unchanged: a redirect is an ordinary type-fn entry whose `createRTFn` returns a pure fn.

**The third marker — `InjectPureFnHash<F>` and anonymous pure-fn registration.** The pure-fn surface is **two-laned** (named vs anonymous identity) × **two-formed** (factory vs direct authoring shape), so four registrars sit over one shared extraction + cache path ([`packages/ts-runtypes/src/runtypes/pureFn.ts`](../packages/ts-runtypes/src/runtypes/pureFn.ts)). Both forms end up as the same runtime `CompiledPureFunction` — the cache always stores a factory `(utl) => fn` for lazy materialisation — and differ only in the AUTHORING shape a **form marker** declares on the pure-fn parameter: `PureFunctionFactory<F>` means the argument IS the factory (emitted as-is, so it can do one-time setup + compose siblings via `utl.usePureFn('ns::id')`), while `PureFunction<F>` means the argument IS the pure fn itself and the compiler WRAPS it into `() => fn` — the ergonomic single-callback shape a framework's `serverMapFrom(t => t.id)` wants. The wrap difference is BUILD-TIME only: for the direct form the Go extractor synthesises the factory by rendering the whole fn as `return <fn>;` (mirroring the override lane's `stripTypesFromExpr`, zero params, no `utl`), for the factory form it strips the factory's own body and extracts its params + `utl` deps; at runtime the plugin has already rewritten the argument to its entry-module tuple, so all four registrars share the same core. `PureFunctionFactory<F>` is a new form marker (`marker.KindPureFunctionFactory`, brand `__rtPureFunctionFactoryBrand`) alongside the existing `PureFunction<F>`; `pureFnFormMarker` reads which one slot carries and returns the `wrap` bit that steers extraction, so the intent propagates THROUGH a wrapper — a library declares `serverMapFrom<F>(mapper: PureFunction<F>, hash?)` or `registerXFactory<F>(factory: PureFunctionFactory<F>, hash?)` and its call sites resolve the same form the primitive would.

The **named** lane — `registerPureFn(pureFnId: CompTimeArgs<PureFnId>, fn: PureFunction<F>)` (direct) / `registerPureFnFactory(pureFnId, createPureFn: PureFunctionFactory<F>)` (factory) — takes the identity as an author-written `"<namespace>::<name>"` literal and stays build-TRACKED (`usePureFn("ns::fn")` references drive PFE9012 + pure-fn→pure-fn dep edges). It is meant for direct developer use, so it deliberately isn't wrappable: composing the id inside a library wrapper produces a computed key (**CTA003**). The **anonymous** lane — `registerAnonymousPureFn<F>(fn: PureFunction<F>, hash?: InjectPureFnHash<F>)` (direct) / `registerAnonymousPureFnFactory<F>(createPureFn: PureFunctionFactory<F>, hash?)` (factory) — moves the identity onto an INJECTED marker in the callee signature, mirroring `InjectRunTypeId<T>` / `InjectTypeFnArgs<T, Fn>`, so it propagates through wrappers exactly the way `createValidate<T>` does. `InjectPureFnHash<F>` is a pure injection marker (a new `marker.KindInjectPureFnHash` spec — [`ts-go-runtypes/internal/compiler/marker/marker.go`](../ts-go-runtypes/internal/compiler/marker/marker.go)); absent at author time, the plugin fills the trailing `hash?` slot with `"rt::<bodyHash>"`, the SAME content hash (`CodeHash` of the extracted code — `return <fn>;` for the direct form, the factory body for the factory form — not of `<F>`) the entry is keyed under, so a same-body registration made DIRECTLY and one made THROUGH A LIBRARY WRAPPER inject a byte-identical id. Because the two forms hash DIFFERENT code, a direct fn and a factory that sets up around it never alias; a direct fn and a trivial `() => fn` factory correctly DO (identical `return <fn>;`). The extractor recognizes the call by brand (a form marker on the pure-fn arg + `InjectPureFnHash<F>` on the trailing slot — `purefunctions.isAnonymousPureFnCall`), runs the same purity gate + body extraction keyed `rt::<CodeHash>`, rewrites the arg to its entry-module tuple, and splices the hash as a plain string literal via a point-insertion `Replacement` at the call's closing paren (`Entry.HashInjectText`; no `Site`, so the resolver's marker walk carries no case for the kind). Because the injected id is lost on any call site the rewrite skips, the per-file rewrite draws from `purefunctions.RawEntries` — EVERY registration call site, not the cross-file-deduped set — so a same-file duplicate body's second call is rewritten too (the emitted MODULE still dedups by key). And because a pure fn is a `Replacement`, never a `Site`, the transform gate that would otherwise miss these files folds pure-fn replacement files into `OpGenerate`'s `SiteFiles` (`sess.pureFnReplacementFiles`) — so a consumer calling a wrapper (its file imports neither the marker package by name nor the primitive textually) is transformed with zero configuration. The companion **runtime-key accessor** — `getRTUtils().getPureFnByKey(key: string)` / `hasPureFnByKey(key: string)` ([`rtUtils.ts`](../packages/ts-runtypes/src/runtypes/rtUtils.ts)) — takes a plain `string`, NOT `CompTimeArgs<string>`, so it is deliberately NOT build-tracked (no PFE9012, no dep edges): it is the door for a framework dispatching on a pure-fn id received over the WIRE (inherently non-literal), kept distinct from the tracked literal lookups so no consumer silently loses demand-checking. Both lanes, both forms, and the untracked accessor are additive: nothing about `usePureFn` / the literal lookups changed.

**Noop elision is semantic, never shape-based** ([`ts-go-runtypes/internal/cachegen/typefunctions/noop_types.go`](../ts-go-runtypes/internal/cachegen/typefunctions/noop_types.go)). EVERY family implements `IsNoopType` — a predicate over the TYPE GRAPH (cycle-safe greatest fixpoint, memoized in `FactsTable`) answering "is this family's entry for T the family identity?" — and the renderer takes the entry's noop VERDICT from it; emitted text never decides. Each predicate mirrors ITS OWN emitter arm-by-arm, delegating exactly where the emitter delegates (compactForJson reuses prepareForJsonSafe's predicate wholesale — their diverging object arms agree on never-noop; compactFromJson delegates restoreFromJson's shared arms but answers false at its own object arms, where the positional rebuild is real work; the five unknown-keys families share one parameterised mirror) and calling the emitter's OWN helpers where the emitter decides through one (`isStrippedUnionMember` for dropped property slots, `objectHasIndexSignatureChild`, `iterableInnerTypes`, `literalFlavour`, the flat-union `roundTripsRaw` bucketing). Consumers: (1) the walker's dispatch gate composes around external children the predicate proves identity — no dep call, no import (this is also what collapses circular identity bodies). The gate is fed only by families asserting `NoopComposeAround` — empty code is a correct composition only under value-transform semantics; stringifyJson (parents concatenate the child's JSON FRAGMENT) and fromBinary (parents advance positionally through the byte stream) are root-verdict-only. (2) JSON composites read their primitives' RENDERED `entrymod.Entry.IsNoop` flags (collected after the family merge) and elide dead bindings — `return JSON.parse(s)` instead of `rjFn(JSON.parse(s))` — and when EVERY binding elides (and the root needs no `[v]` envelope, `rootNeedsDataOnlyWrap`) the composite itself ships as the noop SHORT-FORM tuple: no body, no factory, no imports; the runtime registers the composite family noop from `entryTuple.ts` (native `JSON.stringify` for `je*` tags, `JSON.parse` for `jd*` — NOT the host primitive's value identity, which would corrupt the wire). So a mutate encoder / preserve decoder over a fully JSON-compatible type emits one ~70-byte tuple, and the orphaned pj/rj entries are pruned. SOUNDNESS CONTRACT (one-directional): predicate true ⇒ the emitted body is identity — a false negative only costs bytes, a false positive silently skips a transform. Two mechanical guards pin it: the corpus test ([`ts-go-runtypes/internal/compiler/resolver/noop_predicate_test.go`](../ts-go-runtypes/internal/compiler/resolver/noop_predicate_test.go)) compares every family's predicate against the gate-disabled fully-inlined compile across a real checker corpus, and the renderer's TRIPWIRE (`renderEntryWithDeps`) demotes any predicate-true / body-live disagreement to the LIVE body with a loud stderr log — the compiled body's shape can only veto in the protective direction, never produce a noop verdict. When adding a kind or changing an emit arm, keep the predicate arm in sync; both guards fail loudly on drift.

**Adding a new RT function family:** add an entry to the [`ts-go-runtypes/internal/cachegen/operations`](../ts-go-runtypes/internal/cachegen/operations/) registry (Name + FamilyTag + Axis + FnKey), add a row to `typefunctions.Families` ([`ts-go-runtypes/internal/cachegen/typefunctions/families.go`](../ts-go-runtypes/internal/cachegen/typefunctions/families.go)) plus a `familyAddedFlags` row in [`ts-go-runtypes/internal/compiler/resolver/dispatch.go`](../ts-go-runtypes/internal/compiler/resolver/dispatch.go), add its tag to the runtime `familyMeta` table in [`packages/ts-runtypes/src/runtypes/entryTuple.ts`](../packages/ts-runtypes/src/runtypes/entryTuple.ts) (fnID / args / defaultParamValues / noop identity), and give its `createX` an `InjectTypeFnArgs<T, '<fnKey>'>` trailing param (plus a `CompTimeFnArgs` option slot if it has comptime options) reading the injected entry tuple. `FamilySpec.Collect`/`FamilySpec.AnySupported` cover the per-family wrappers. Cross-family references work automatically — record them as walker cross-family deps and they ride the entry's SoftDeps into the resolver fixpoint (the `it` precedent).

### Built-in pure fns, delivered on demand

The package owns a set of runtime helpers in the `rt::` and `rtFormats::` pure-fn namespaces — the validation-error push (`rt::newRunTypeErr`), the unknown-keys scanners (`rt::hasUnknownKeysFromArray`, …), every format validator (`rtFormats::isUUID`, `rtFormats::isDateString_*`, …), and the circular-reference walker (`rt::findCycle`). Emitted bodies reach them by name (`utl.getPureFn('rt::newRunTypeErr')`), so historically the package **side-effect-imported** every registration file up front and shipped all those bodies in every bundle — ~1.6 KB (`rt::`) + ~9.7 KB (`rtFormats::`) whether an app used them or not.

They now ride the same on-demand machinery user pure fns already use:

- **One choke point.** Every emitter reference goes through `EmitContext.UsePureFn(ns, fn, path)` ([`ts-go-runtypes/internal/cachegen/typefunctions/emitter.go`](../ts-go-runtypes/internal/cachegen/typefunctions/emitter.go)) — it records the dependency, hoists the deduped `const <alias> = utl.getPureFn('<ns>::<fn>')` prologue line, and returns the alias, so a body reference and its recording are inseparable (a corpus tripwire pins it).
- **Bodies travel via a generated table.** A published consumer is dist + `.d.ts` — nothing to extract — so [`ts-go-runtypes/internal/cachegen/builtinpurefns/`](../ts-go-runtypes/internal/cachegen/builtinpurefns/) carries the bodies, extracted from `packages/ts-runtypes/src/` by the same pure-fn extractor the resolver runs on user pure fns (`pnpm rtx core codegen builtinpurefns [--check]`). The TS files stay the authored, type-checked source of truth.
- **The resolver serves + wires them.** A live fn entry's recorded pure-fn deps land on its `SoftDeps`; `serveBuiltinPureFns` delivers each demanded built-in from the table as an ordinary pure-fn virtual module, so the entry imports it and its deps thunk (tuple slot 1) registers it before the body's `getPureFn` runs. A demanded key the table lacks is a build error (PFE9012). Warm entries recover their pure-fn edges from the disk cache's `PureFnRefs` (format v15).
- **The dist ships hollow scaffolding.** The side-effect imports stay (upgraded apps keep working with no code change), but the dist build hollows the registration files ([`scripts/core/hollow-builtin-purefns.mjs`](../scripts/core/hollow-builtin-purefns.mjs)) — factory bodies replaced by `null`, line count and `.d.ts` preserved — so they cost scaffolding bytes only. `registerCore`'s null lane is an inert, uncached no-op for a built-in key, so it never masks the real body arriving via a deps thunk.
- **They honor `emitMode` like every other entry.** The table stores each built-in's `code` + `paramNames`, and delivery flows through `purefunctions.CollectEntries`, so the served module's `code` / `createPureFn` slots are gated by the build's `emitMode` for free. In `code` mode a built-in ships only its `code` string and the runtime rebuilds the factory via `new Function` (so it is NOT CSP-safe without `unsafe-eval` — the same as a `code`-mode type fn); in `functions`/`both` the live closure ships and a CSP runtime already on those modes keeps working. The circular walker `rt::findCycle` is no exception.

The circular-reference walker is one consumer of this: `rt::findCycle` is demanded by a BODY reference like every other built-in — only an **armed** (`{rejectCircularRefs: true}`) guarded factory over a cyclable type inlines the `utl.getPureFn('rt::findCycle')` call, so the walker loads only into bundles that actually arm the guard. It takes a baked path skeleton rather than the RunType graph, so a plain cyclable type ships neither the walker nor a data bundle (see the [Circular-reference guard](#circular-reference-guard--a-compile-time-option-value-level) section).

### ts-go-runtypes/internal/compiler/program

Wraps the [`oxc-project/tsgolint`](https://github.com/oxc-project/tsgolint) shim packages into a simple `program.New(Options)` that:

- layers an optional overlay VFS on top of `osvfs` + `cachedvfs` + `bundled`
- parses the supplied tsconfig via `tsoptions.GetParsedCommandLineOfConfigFile`
- **drops tsconfig project `references`** before building the program — they are a `tsc --build` orchestration concept the bundler never honors; keeping them redirected imports landing in a referenced project to its declaration OUTPUTS (usually unbuilt in a dev loop), silently blanking the marker scan (see [docs/done/project-references-unbuilt-outputs-silent-zero-sites.md](./done/project-references-unbuilt-outputs-silent-zero-sites.md))
- builds a `compiler.Program` and calls `BindSourceFiles()`

`NewInferred` is a second constructor that skips tsconfig for one-shot queries on loose files.

### ts-go-runtypes/internal/compiler/resolver

Owns both the AST call-walk and op dispatch. `walk.go` contains `NodeAt`, `CallExpressionAt`, and `ForEachCallExpression` (depth-first visitor over every `CallExpression` in a source file). `scan.go` handles the `scanFiles` op logic. Dispatches six operations (see `Op*` constants in [`ts-go-runtypes/internal/protocol/protocol.go`](../ts-go-runtypes/internal/protocol/protocol.go)):

| op           | semantics                                                                                                                                                                                                                                                                         |
| ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `scanFiles`  | Walks every CallExpression in file. For each whose resolved signature has a trailing marker param with bound T, returns a `Site{Pos, ID, ParamIndex, ArgsCount, Demand}`. Opt-in `includeEntryModules` projects the per-entry virtual modules scoped to just the requested files. |
| `dump`       | Returns the full cache (every RunType) + the running Sites slice + the rendered cache module sources.                                                                                                                                                                             |
| `setSources` | Replaces the resolver's in-memory source overlay and rebuilds the Program (drives test harnesses and the daemon's HMR path).                                                                                                                                                      |
| `reset`      | Wipes all resolver state: cache, sites, Program, checker.                                                                                                                                                                                                                         |
| `resolveId`  | Returns the canonical full RunType for a hash id; child slots stay `KindRef` stubs, the caller re-issues `resolveId` per id to drill in.                                                                                                                                          |
| `tsCompile`  | Runs the embedded tsgo through bind + typecheck + emit without the marker scan — a pure-TS latency baseline for benchmarks.                                                                                                                                                       |

`Pos` is the byte offset of the closing `)` of the call — the TS-side patcher inserts at that offset. `ParamIndex` is the 0-based slot the injected id goes into; `ArgsCount` is the number of arguments the user already wrote (so the patcher knows whether to pad with `undefined`).

**Reflect-form annotation honoring.** When the value argument is a const-bound identifier with a written type annotation (`const v: T = literal; createValidate(v);`), the resolver reads the annotation directly via `Checker_getTypeFromTypeNode` instead of trusting TypeScript's control-flow-analysis apparent type for `v`. Without this, CFA narrows the binding to its initializer's narrowest type (e.g. `Color.Red` for an enum or `'hello'` for a union), and the reflect-form hash would diverge from the static `createValidate<T>()` form. The walk only fires in the reflect form (no explicit type arguments) and only for `Identifier` arguments — property accesses, function calls, and element accesses don't go through const-binding CFA and don't exhibit the trap.

**Function-call argument warning.** The resolver flags `createValidate(getX())` (and any other reflect-form marker call with a `CallExpression` argument) as an anti-pattern: the function is invoked at runtime purely so the type checker can infer `T` from its return type, even though the value is discarded. The validator still works, but the recommended replacement is the static form using `ReturnType<typeof fn>`. The warning surfaces on the response's `markerDiagnostics` channel and the Vite plugin re-emits it through `this.warn` in canonical `tsc --pretty=false` format so VS Code's `$tsc` problem matcher picks it up.

### ts-go-runtypes/internal/compiler/marker

`Detect(t *checker.Type, opts Options) (typeArg, ok)` — given the type of a function's trailing parameter, returns whether it matches the configured marker (name + declaring-module check) and extracts the single type argument `T`. `IsFreeTypeParameter(t)` filters out calls inside generic bodies.

### ts-go-runtypes/internal/cachegen/runtype

The `runtype` serializer (`ts-go-runtypes/internal/cachegen/runtype/serialize.go`) interns `*checker.Type` → stable hash id (e.g. `abc123`). `AssignID(t)` is the public entry point used by the marker scanner; `Serialize(t)` returns a `KindRef` sentinel pointing at the cached entry. Recursion is broken by reserving the id before descending. Structural dedup means two distinct AST types that share the same shape end up with the same id. Per-node naming data is id-relevant: tuple element labels, signature parameter names, and the full format-annotation params (`mockSamples`/`message` included) fold into the structural id, so same-shape types that differ only there intern as distinct canonical nodes — `children[].name` / `parameters[].name` and per-format mock samples are reliable per node, never first-intern-wins. The `module.go` file renders both output formats (JSON dump and the self-wired TS module). The `typeid/` subdirectory holds the structural-id computation.

### ts-go-runtypes/internal/protocol

Pure struct definitions shared between the Go resolver and the TS plugin. Stdio protocol is newline-delimited JSON; one `Request` in, one `Response` out, EOF terminates. The daemon mode wraps a Unix-socket accept loop around the same handler, one client at a time.

### packages/ts-runtypes (`@ts-runtypes/core`)

The public runtime package, three entry points:

- `.` — the markers (`InjectRunTypeId<T>`, `InjectTypeFnArgs<T, Fn>`, `InjectPureFnHash<F>`, `CompTimeArgs<T>` / `CompTimeFnArgs<T>`, `PureFunction` / `PureFunctionFactory`), the reflection helpers `getRunTypeId<T>()` (static — explicit type argument, no value; throws if the plugin didn't inject an id) and `getRunTypeId(value)` (`T` inferred from the value), and the full `createX` factory surface: `createValidate` / `createGetValidationErrors`, the unknown-keys group (`createHasUnknownKeys` / `createCloneExactShape` / `createUnknownKeyErrors`), `createFormatTransform`, `createJsonEncoder` / `createJsonDecoder`, `createBinaryEncoder` / `createBinaryDecoder` / `createBinarySizer`, `createMockData`, the committed-enrichment surface (the `FriendlyText<T>` / `MockData<T>` DSL types and the `createFriendlyText` / `createFriendlyTextI18n` renderers — committed mirror artifacts consumed via ordinary imports, not cache modules; see [docs/AI_ENRICHMENT.md](./AI_ENRICHMENT.md)), plus the runtime registries (`registerPureFn` / `registerPureFnFactory`, `registerAnonymousPureFn` / `registerAnonymousPureFnFactory`, `registerClassSerializer`, `registerMockingFunction`, `registerFormatPattern`) and the DataView serializer helpers.
- `./schema` — the value-first NON-format builders (`boolean()`, `literal()`, `object({...})`, `array()`, `union()`, the utility builders, …) returning live `RunType<T>` nodes; `InferType<T>` recovers the TS type. Each builder converges on the same structural id as its type-first equivalent — except where labels/param names apply: those are id-relevant, and the `RT.tuple`/`RT.func` builders express unlabeled/unnamed shapes (TS function-type syntax requires param names), so value-first schemas for fn-bearing and labeled types carry distinct ids from their type-first twins with identical behavior. The value-first `func` expansion maps a labeled rest-tuple param's labels to parameter names, so a labeled tuple still converges with the equivalent written `(a: A, b: B)`; an unlabeled one converges only with other unlabeled forms. Label-capable builders would restore convergence (see ROADMAP). (The format builders — `string()` / `number()` / `email()` / … — and the `temporal.*` family moved to `./formats` + `./formats/temporal`.)
- `./formats` — the type-format aliases (`TF.String`, `TF.Email`, `TF.UUIDv4/v7`, `TF.Number`, `TF.Currency`, `TF.BigInt`, `TF.StringDate/Time/DateTime`, `TF.Date`, fixed-width int presets, …) AND their value-first builders (`TF.email()`, `TF.int32()`, …), namespaced as `import * as TF from '@ts-runtypes/core/formats'`; plus their mock / pure-fn / pattern registrations. The Temporal family (`TFT.Instant` / `TFT.PlainDate` / …) is on the `./formats/temporal` subpath (`TFT`). Format **validation** happens at build time on the Go side; the runtime only carries mocking + transform helpers.

Cache access goes through the runtime registry (`getRTUtils()` in `src/runtypes/rtUtils.ts`): each marker call site receives its entry-module tuple, `initFromTuple` (in `src/runtypes/entryTuple.ts`) registers the tuple's dependency closure (reflection nodes by typeId from the data bundle's rows, function factories by the `<fnHash>_<typeId>` key, pure fns by `<ns>::<fn>`), and the factories resolve via `resolveEntryTupleFn` / `getRunType(id)` with a noop fallback when a type has no supported entry (the Go side emits a resolvable `KindMissing` stub module for dropped entries).

### packages/ts-runtypes-devtools

- `ResolverClient` — spawns the Go binary, serialises outstanding queries, parses line-delimited responses.
- The byte-offset rewrite and its source map live in **Go** (the `transform` package — [transform.go](../ts-go-runtypes/internal/compiler/sourcerewrite/transform.go) + [editbuffer.go](../ts-go-runtypes/internal/compiler/sourcerewrite/editbuffer.go); see Rewrite mechanics below). `EditBuffer` is the from-scratch string-editor + source-map generator that replaced `magic-string`, so the published plugin has **no runtime dependencies**; its map matches magic-string's `hires: 'boundary'` segmentation, so the bundler's composite-map chain is byte-for-byte equivalent.
- `unplugin.ts` — the cross-bundler factory + glue. Gates each file on the resolver's own site-file set: `generate()` returns `siteFiles` (every program file the whole-program scan found marker sites in, PLUS every file carrying an extracted pure-fn registration — a pure fn is a `Replacement`, not a `Site`, so `uniqueSiteFiles` folds in `sess.pureFnReplacementFiles`) at `buildStart`, and `handleHotUpdate` keeps the set current per edited file — so wrapper-framework call sites (users' files import the framework, e.g. `@mionkit/router`'s `route()` or a library's own `registerXPureFn` over the anonymous pure-fn lane, never `@ts-runtypes/core`) rewrite with ZERO configuration, whatever package declared the wrapper (pinned by `packages/ts-runtypes-devtools/test/wrapper-zero-config.test.ts`). A textual check (quoted `'@ts-runtypes/core'` import prefix, or the substrings `registerPureFn` / `registerAnonymousPureFn` — covering all four named + anonymous registrars for the marker package's own relative-import sources) stays as the fallback for files created after the last scan. Its `transform` hook calls `OpTransform` per file and returns `{code, map}` — either forwarding the resolver's finished output (`transformMode: 'go'`) or applying the resolver's edit list itself (`transformMode: 'edits'`, the default; see Rewrite mechanics → Two transform wire modes below); at `buildStart` it runs `OpGenerate` to write the cache modules to disk under `<genDir>/types/`, and the bundler resolves the injected **relative** imports natively (no `resolveId`/`load` virtual-module hooks — the Go side is the single renderer). The `moduleMode` option (forwarded as `--module-mode`) selects the grouping: `default` (per-entry fn modules + the runtype data bundle), `allSingle` (one bundle module per fn family + `pf` bundle + facades folded into the runtypes bundle — fewest modules), or `allModules` (per-node runtype modules, the pre-bundle layout).

## Slot injection and padding

The id is injected at the trailing `InjectRunTypeId<T>` slot. The Go binary returns `ParamIndex` + `ArgsCount` per site; the TS-side `buildInsertion()` pads with `undefined` whenever the caller wrote fewer arguments than `paramIndex`:

```
getRunTypeId<T>()           →   getRunTypeId<T>(undefined, __rt_<hash>)
getRunTypeId(val)           →   getRunTypeId(val, __rt_<hash>)
validate<T>(v)              →   validate<T>(v, __rt_<fnHash>_<hash>)
```

(each `__rt_…` binding is the entry-module tuple imported at the top of the
file; `getRunTypeId` still RETURNS the id string, so the public contract is
unchanged)

`getRunTypeId<T>()` (static form, no value) is padded with one `undefined` so the injected id lands in the trailing slot 1 — the same slot the reflection form `getRunTypeId(val)` appends to after its value at slot 0. The padding mechanism also covers user-defined wrappers with additional intermediate parameters.

## Rewrite mechanics

> **Files-mode (landed).** The mechanism below was originally served as
> **virtual modules** (`virtual:rt/<key>.js`, the plugin's `resolveId`/`load`).
> It now writes every cache module to a **real file** under `<genDir>/types/`
> (default `<srcDir>/__runtypes`, inferred from tsconfig) at `buildStart`
> (`OpGenerate`), and the transform injects **relative** imports to those files
> — so every bundler resolves them natively with no per-bundler virtual-module
> plumbing. `virtual:rt/<key>.js` is still the INTERNAL render specifier; it is
> relativized at the resolver layer (post-render for inter-module imports,
> post-`Apply` for user code), which keeps the transform/entrymod golden corpus
> byte-stable. Read "virtual module" below as "real file under `<genDir>/types/`".
> Build-time enrichment (`<genDir>/enriched/`) and two-way dev sync are deferred
> (not yet implemented).

- Rewrites are positioned by **UTF-8 byte offsets, not string indices** — tsgo positions count bytes. The Go `transform` package ([transform.go](../ts-go-runtypes/internal/compiler/sourcerewrite/transform.go)) applies edits through the in-house `EditBuffer` ([editbuffer.go](../ts-go-runtypes/internal/compiler/sourcerewrite/editbuffer.go) — the from-scratch editor + source-map generator ported from `magic-string`, keeping the published plugin dependency-free; `OpTransform` returns finished code + a real source map and the user's original lines/columns survive the injected import block + bindings) and converts every resolver offset via `makeByteToChar` before indexing. Don't index the source string with a raw resolver offset; multibyte source characters will misalign the inserted hash.
- **Two transform wire modes** — `OpTransform` can either apply the rewrite in Go or hand the edit list to the plugin, selected by the plugin's `transformMode` option (per-request `emitEdits` on the wire, NEVER a disk-cache fingerprint input — the artifacts are identical either way):
  - **`'go'`** — the resolver applies the rewrite and returns the whole rewritten `code` + source map per file. The universal path (the only option for a non-JS / plugin-free host), and the fallback the plugin degrades to on any anomaly. Because the map embeds the original source in `sourcesContent` (the heaviest single wire item), the `sourcesContent: false` plugin option (`Request.OmitSourcesContent`) drops it — the bundler fills original content when composing its chained map.
  - **`'edits'` (default)** — the resolver ships `TransformResult{ImportBlock, Edits, SourceHash}` instead: `ComputeEdits` ([edits.go](../ts-go-runtypes/internal/compiler/sourcerewrite/edits.go)) reuses the SAME `buildInsertion` / `buildImportBlock` as `Apply`, converts every byte offset to a **UTF-16 code unit** (`makeByteToChar`), and the plugin applies the list with its own `EditBuffer` ([packages/ts-runtypes-devtools/src/edit-buffer.ts](../packages/ts-runtypes-devtools/src/edit-buffer.ts) — the JS twin the Go one was ported from) via `applyEdits` ([apply-edits.ts](../packages/ts-runtypes-devtools/src/apply-edits.ts)). It calls `prepend`/`appendLeft`/`update` in the identical sequence `Apply` does, so the two modes are **byte-identical by construction** (code AND map), pinned by the mode-parity corpus in [transform-modes.test.ts](../packages/ts-runtypes-devtools/test/transform-modes.test.ts). The wire drops from O(whole rewritten file + dense map) to **O(sites)** — the transform-wire benchmark measured `'edits'` cutting the inbound wire ~6-55× (growing with file size), which is why it is the default.
  - **Source-consistency guard.** `'edits'` applies the edits to the bundler-supplied `code`, but the offsets were computed against the resolver's bytes. `SourceHash` (FNV-1a/32 over the source, matched byte-for-byte by the FE `sourceHash`) lets the plugin detect an upstream `enforce:'pre'` plugin that edited the source first: on mismatch it re-uploads the source (`setSources`) and re-requests once, warning the user to order @ts-runtypes/devtools first; if it still diverges (or the applier throws), it falls back to `'go'`. `'go'` rebuilds from the resolver's view and so clobbers such upstream edits — it can't recover (its output ignores the incoming `code`), but it carries the same `SourceHash` so the plugin at least **detects and warns** on the mismatch instead of clobbering silently. Both modes assume this plugin sees pristine source.
- **Module grouping is configurable** via the plugin's `moduleMode` option (mirrored as the binary's `--module-mode` flag; values in [ts-go-runtypes/internal/constants/constants.go](../ts-go-runtypes/internal/constants/constants.go)): `default` is described below; `allSingle` bundles EVERYTHING (one module per fn family under `fns/<tag>` with one NAMED export per entry (the same `__rt_` binding name every module mode exports under) plus one `pf` pure-fn bundle, and the reflection facades fold into the runtypes bundle; `Site.Module` / `Replacement.ImportFrom` point the rewrite's clauses at the bundle specifier (the clause shape is the same named import everywhere), and the family bundles join the runtypes bundle as mutable modules invalidated in `handleHotUpdate`); `allModules` splits everything (per-node runtype modules, tuple kind 0 — the pre-bundle layout, kept as an escape hatch; measured slower on dense reflection graphs). The grouping layer is `entrymod.RenderGrouped` + `Resolver.moduleGrouping`; per-entry renderers are identical across modes.
- **Child inlining is configurable** via the plugin's `inlineMode` option (binary `--inline-mode`; values in [ts-go-runtypes/internal/constants/constants.go](../ts-go-runtypes/internal/constants/constants.go)): `default` applies the NAME RULE — unnamed compounds (arrays/tuples/object literals/unions/classes) inline into their parents (blocks hoist to per-factory context fns — `ctxFn<N>` — instead of per-call IIFEs) while named types (alias or interface) and circular types stay external as dedupe-worthy shared entries (Date/Temporal builtins always inline — atomic emits); `allInternal` is name-blind — everything except circular types inlines. The walker carries its own cycle breaker (`inlineWouldCycle`): a node already on the walk stack always goes external, because `IsCircular` flags only the serializer's re-entry node and union flattening can re-enter a cycle through an unflagged anonymous wrapper (e.g. the `U | undefined` of an optional `a?: U` prop). The two modes never share disk caches (fingerprint folds inlineMode in).
- **Every cache entry is its own virtual module** (default mode) — `virtual:rt/<key>.js` (`<fnHash>_<typeId>` for function entries and JSON composites, `pf/<ns>/<fn>` for pure fns) — EXCEPT runtype nodes, which ride as headless ROWS of THE single data bundle `virtual:rt/runtypes.js` (tuple kind 4: content-hash key, rows array, a parallel `rels` array wiring each row's ref slots by ROW INDEX, and a residual `ini` footer for expression-specials only — `classType`, bigint/symbol literals, `formatAnnotation`), with a tiny per-reflection-root facade `virtual:rt/<rootId>.js` (kind 5: imports the bundle, carries the root id) so the rewrite's binding-only injection is unchanged. Each node row exists exactly once app-wide, and runtype emission is demand-driven on REFLECTION sites — a createX-only file emits zero runtype modules. Every module exports ONE positional tuple under its BINDING NAME (`__rt_<key>`, identifier-escaped — the same name in the export, every import clause, and the call-site binding; imports never rename): `[kindOrFamilyTag, depsThunk|hole, ini|hole, …legacy positional args]`. Absent slots are JS array HOLES, not a `u`/`undefined` alias (the runtime reads tuples by index, so a hole is indistinguishable from undefined); there is no `const u=undefined;` line, and every default-valued fn-tuple slot renders as a hole even when interior (a later non-default slot — a live factory, the tb size estimate — no longer forces the earlier defaults to be spelled out). The assembler lives in [ts-go-runtypes/internal/compiler/virtualmodules/virtualmodules.go](../ts-go-runtypes/internal/compiler/virtualmodules/virtualmodules.go); the runtime consumer in [packages/ts-runtypes/src/runtypes/entryTuple.ts](../packages/ts-runtypes/src/runtypes/entryTuple.ts) — those two files are the tuple-layout sync boundary. Constants (virtual prefix, binding prefix) come from [ts-go-runtypes/internal/constants/constants.go](../ts-go-runtypes/internal/constants/constants.go) (mirrored to TS via `pnpm run gen:ts-constants`).
- **Imports and `deps()` carry DIRECT dependencies only** — never the flattened transitive closure (flattening was quadratic text on dense graphs: 6x wire payload / 4x render time on the real suites). Both lists are leaves-first, alphabetical within a dependency level (Tarjan SCC collapse for recursive types), never self (consumers already hold the tuple; dep-less entries leave the deps slot a JS array HOLE — `[tag,,ini,…]` — instead of a thunk), and `deps()` is a lazy THUNK inlined straight into the slot (`()=>[d1]`, no named `const deps=`) — so module-level import cycles never hit TDZ. The runtime's `initFromTuple` walks the thunks RECURSIVELY (post-order, processed-keys guard) in two phases: register every unseen tuple in the closure (children before parents), then wire each data bundle's `rels` array by ROW INDEX (`wireBundleRelations`) and run any residual runtype `ini` footer — refs always resolve. The residual `ini` bodies patch through the registry (`c(id)`), never through imported bindings.
- The rewrite injects one deduped import block at offset 0 per user file plus the entry-module BINDING at each call site (`createValidate<T>(__rt_<fnHash>_<id>)`) — no id strings ride along; ids derive from the tuple (slot 3). Entry modules are content-addressed (ids embed the binary version), so they are immutable and never need HMR invalidation — except the runtype data bundle, the ONE mutable module (its rows are the union of reflection demand): the plugin invalidates it in `handleHotUpdate` when a scan reports `addedRunTypes`. The Vite plugin serves bodies verbatim from the `entryModules` map on the resolver's `dump` response. Tests can short-circuit by setting `includeEntryModules: true` on `scanFiles` to receive the same map scoped to **just the files in that request** (per-request projection, not session-wide accumulation — callers wanting everything in memory use `dump`). Demanded entries that get dropped (unsupported kinds, dangling deps) still emit a resolvable `KindMissing` stub module; the runtime degrades to the family identity fn, preserving the old silent-degrade semantics.
- **Builtin classes project atomically** — `Date` / `Map` / `Set` / `RegExp` / Temporal / the non-serializable set stop at subKind + classRef (+ Map/Set element `Arguments`); their lib members are never walked or interned (`projectClass` in [ts-go-runtypes/internal/cachegen/runtype/serialize.go](../ts-go-runtypes/internal/cachegen/runtype/serialize.go)). Every consumer keys on subKind — don't reintroduce member expansion. Relatedly, an **OPTIONAL** member inherited from a default-lib global type (Error's `stack?` / `cause?`) is projected as **non-enumerable-guarded**: the serializer families that build output by name (prepareForJsonSafe / stringifyJson / compactForJson / toBinary) gate the write on a runtime own-enumerability check (`Object.prototype.propertyIsEnumerable`, i.e. `JSON.stringify` semantics), and the same flag folds into the structural id. The shared `typeid.IsNonEnumerable` (used by both `projectMembersInto` / `appendProperty` and the id's member walk, so id and projection can't drift) also fires for `@nonEnumerable`-tagged user props. The guard is gated on the member being OPTIONAL in the type on BOTH arms, so the invariant **guarded ⇒ optional-in-type** holds — which keeps `DataOnly<T>` sound by construction (a guarded member is always one the type already permits to be absent). Consequences: a vanilla `class X extends Error` still serializes the REQUIRED envelope `name` / `message` (always, so the error stays useful and `DataOnly<Error>` stays accurate) but never leaks `stack` unless a value makes it enumerable; a `@nonEnumerable` tag on a REQUIRED prop is a no-op (serialized unconditionally, flagged by the `NE` lint rule). Guarded members are also optional in the projected shape, so validators and the binary presence path accept their absence. See [docs/done/runtime-enumerability-checks-for-global-props.md](done/runtime-enumerability-checks-for-global-props.md).
- Types are **deduplicated twice** in [ts-go-runtypes/internal/cachegen/runtype/](../ts-go-runtypes/internal/cachegen/runtype/) — pointer identity (same `*checker.Type` reached via two paths) AND structural id (two distinct `Type` objects with the same shape). Both collapse to a single cache entry. "Same shape" includes per-node naming data: tuple element labels, signature parameter names, and format params (`mockSamples`/`message` included) are id-relevant, so nodes differing only there stay distinct entries carrying their own `children[].name` / `parameters[].name` — labels are node data, not parent-relative data (next bullet).
- **Never store parent-relative data on a canonical node**. Cache entries are shared singletons (one per structural id), so any field whose meaning depends on which parent referenced the node — `parent` back-link, "my slot index in MY parent", "I'm a discriminator for THIS parent union" — is silently wrong the moment that node appears under more than one parent. If a relationship is parent-scoped, store it on the **parent** (e.g. `RunType.UnionDiscriminators` lives on the union, not on the property), or have the consumer build the back-link at walk time from a known root. See [docs/ROADMAP.md](./ROADMAP.md) → "JSON shape — known limitations" for the `parent` row and the union discriminator wire shape rationale.

## Reflection shape

The protocol's `RunType` is the canonical reflection-shape discriminated union. Specifically:

- **Numeric `ReflectionKind`** is declared in a stable order (never=0, any=1, …, callSignature=35) so the integer values are wire-safe across releases. Sentinel `-1` is reserved for ref slots.
- **Container shape**: `KindObjectLiteral.children` is an array of `KindPropertySignature`/`KindMethodSignature`/`KindIndexSignature`/`KindCallSignature` nodes; `KindFunction.parameters` is an array of `KindParameter` nodes; tuple elements are wrapped as `KindTupleMember`.
- **Annotations carried**: `id`, `typeName`, `typeArguments`, `optional`, `readonly`, `nonEnumerable` (lib-global-inherited / `@nonEnumerable`-tagged — the write is enumerability-gated), `abstract`, `static`, `flags`, `default` (literal-only), `classRef` (builtin provenance for lazy-import), `formatAnnotation` (type-format name + canonicalised params), `unionDiscriminators` (parent-scoped, on the union node). Declared in the protocol but **not yet populated**: `inlined`, `isCircular`, `description` — tracked in [docs/ROADMAP.md](./ROADMAP.md) ("Reflection features that need AST-level scanning", status snapshot). The old number `brand` is subsumed by `typeMeta`.
- **Knotted output**: the runtime artifact pre-resolves cycles via a footer of direct child-slot assignments, so `getRunType(id)` hands back a fully-knotted `RunType` graph — no adapter layer needed. Canonical nodes never carry `parent` back-links (cache entries are shared singletons, one per structural id; parent-scoped data lives on the parent — see CLAUDE.md "Never store parent-relative data on a canonical node"). Consumers needing parent pointers build them at walk time from a known root.

Lossy mappings are recorded in [docs/ROADMAP.md](./ROADMAP.md). Highlights:

- Symbol-keyed property names → synthetic `@@<name>` strings + `flags: ["symbol"]`.
- Function/closure-valued `default` → omitted with `flags: ["nonLiteralDefault"]` marker.
- `bigint` literal values → string with `flags: ["bigint"]` (consumer parses with `BigInt(…)`).
- `parent` is never stored on a canonical node (in JSON or the `.ts` artifact); consumers build back-links at walk time. See ROADMAP "JSON shape — known limitations".

Implemented: `templateLiteral` (regex-compile at RT-build time); the full type-format families — string (`TF.String`/`Email`/`UUID`/`URL`/`Domain`/`IP`/…), number/bigint incl. fixed-width binary presets, string date/time/dateTime with min/max bounds, native `Date` bounds, and all 8 `Temporal.*` types (Go: `ts-go-runtypes/internal/cachegen/typefunctions/formats/`; JS: `packages/ts-runtypes/src/formats/`); regex literals via AST-trace from `const`-bound initializers. Generic type-metadata is surfaced via `typeMeta` (any `atomic & { obj }` intersection; renamed from `decorators`, subsumes the old number `brand`). Still deferred: `infer`, TS `@decorator`-syntax capture, validating constraint decorators (`MinLength<5>`-style beyond the format brand scanner), runtime-only fields (`function`, `classType`, `enum`).

### Member types and cycle resolution

Member types are the family of nodes that own exactly one child slot — `Array<T>`, `TupleMember`, `Property`, `PropertySignature`, `Parameter`, `IndexSignature` (which adds an `index` key slot alongside its `type`). `Promise<T>` is modeled as a native type with the same single-child shape. They show up everywhere a parent composite needs to point at "the type of this slot".

The wire format keeps these slots small via the `KindRef = -1` sentinel: every child RunType returned by `serialize.Cache.Serialize` is a `{kind: -1, id: "<hash>"}` stub. The canonical full RunType lives once in the cache, keyed by id. JSON consumers detect the sentinel and dereference manually; the runtime derefs during tuple registration (each runtype entry module's `ini` patches its ref slots through the registry) and hands consumers a fully-knotted graph.

Cycles close at two layers without special-case code:

- **Serializer**: the `runtype` serializer in [`ts-go-runtypes/internal/cachegen/runtype/serialize.go`](../ts-go-runtypes/internal/cachegen/runtype/serialize.go) reserves the id and inserts a placeholder cache entry **before** projecting the type's children. A recursive walk that re-enters the same `*checker.Type` hits the `byPtr` lookup and gets back the reserved id immediately — no infinite recursion, no second projection.
- **Emit**: the runtype data bundle ([`ts-go-runtypes/internal/cachegen/runtype/entries.go`](../ts-go-runtypes/internal/cachegen/runtype/entries.go) + [`ts-go-runtypes/internal/compiler/virtualmodules`](../ts-go-runtypes/internal/compiler/virtualmodules/)) carries one scalar-only row per reflection-demanded node plus a parallel `rels` array (bundle tuple slot 5) that wires each node's ref slots (child, children, parameters, …) by ROW INDEX, and a residual `ini(rtu)` for the rare expression-specials only (builtin `classType` via globalThis, bigint/symbol literals, `formatAnnotation`). Registration is two-phase — every row registers first, then `wireBundleRelations` patches the ref slots by index and any residual `ini` runs — so back-edges work without forward-reference errors and cycles need no special-casing (an index ref has no TDZ, unlike a direct binding); recursion needs no per-node modules at all because all rows share the bundle. The Vite plugin serves the rendered module strings from the resolver's `dump` response — there's no JS-side renderer to mirror.

Callers walking a member type's child ref can ask the resolver for the canonical RunType via the `resolveId` op (see `OpResolveID` in `ts-go-runtypes/internal/protocol/protocol.go`). The returned RunType's child slots remain `KindRef` sentinels — the caller drills in by re-issuing `resolveId` per id.

## Shims — reaching into tsgo's `internal/`

### The visibility problem

`microsoft/typescript-go` keeps almost everything we need under `internal/*`:

```
github.com/microsoft/typescript-go/internal/checker
github.com/microsoft/typescript-go/internal/ast
github.com/microsoft/typescript-go/internal/parser
github.com/microsoft/typescript-go/internal/compiler
...
```

Go enforces an [`internal` visibility rule](https://go.dev/doc/go1.4#internalpackages) at the compiler level: a package whose path contains `…/internal/…` may only be imported by code rooted in the same module. So **no external Go program is allowed to import `internal/checker`** — including this one. The TypeScript team did that deliberately: they want freedom to refactor those packages.

A _shim_ is a thin, **publicly-importable** wrapper package that lives outside `internal/` and re-exports the symbols inside. The shim is what makes external Go consumers (this project, the [oxc TypeScript linter](https://github.com/oxc-project/tsgolint), and others) possible.

### Two techniques, mixed

The shim files (under [`ts-go-runtypes/third_party/tsgolint/shim/*`](../ts-go-runtypes/third_party/tsgolint/shim/), auto-generated by tsgolint's `tools/gen_shims`) use **two complementary techniques**:

**(1) Type aliases** for types, methods, and exported functions:

```go
// shim/checker/shim.go (excerpt)
package checker

import "github.com/microsoft/typescript-go/internal/checker"

type Checker     = checker.Checker     // full method set preserved
type AccessFlags = checker.AccessFlags
const AccessFlagsNone = checker.AccessFlagsNone
```

A type alias preserves the _whole_ method set of the underlying type. Once `Checker` is aliased, every exported method on it — `Checker.GetTypeAtLocation`, `Checker.GetResolvedSignature`, etc. — is callable from outside with no further plumbing. This covers ~80% of the surface.

**(2) `//go:linkname` directives** for unexported helpers we still need:

```go
// shim/vfs/osvfs/shim.go (excerpt)
//go:linkname FS github.com/microsoft/typescript-go/internal/vfs/osvfs.FS
func FS() vfs.FS
```

`//go:linkname` is a Go-runtime escape hatch that says "treat the name on the left as an alias of the symbol on the right at link time," bypassing visibility entirely. It's officially discouraged (and required `-gcflags=-checklinkname=0` for a while) but it's the only way to call genuinely unexported helpers. The shim uses it sparingly — type aliases do the heavy lifting wherever possible.

### How we wire the shim into this module

[`ts-go-runtypes/go.mod`](../ts-go-runtypes/go.mod) maps the shim import paths into our vendored tsgolint submodule via `replace` directives:

```go
replace (
  github.com/microsoft/typescript-go/shim/ast      => ./third_party/tsgolint/shim/ast
  github.com/microsoft/typescript-go/shim/checker  => ./third_party/tsgolint/shim/checker
  github.com/microsoft/typescript-go/shim/compiler => ./third_party/tsgolint/shim/compiler
  ...
)
```

So when [`ts-go-runtypes/internal/compiler/program/program.go`](../ts-go-runtypes/internal/compiler/program/) writes `import "github.com/microsoft/typescript-go/shim/checker"`, it picks up the aliased `internal/checker` from the typescript-go that tsgolint vendors. We never write `internal/*` import paths ourselves — Go would refuse to compile if we did.

### Patches we layer on top

A handful of upstream `internal` symbols that tsgolint _wants_ to alias are themselves unexported (lowercase) in typescript-go. For those, [`ts-go-runtypes/third_party/tsgolint/patches/*.patch`](../ts-go-runtypes/third_party/tsgolint/patches/) apply small visibility lifts to the nested typescript-go submodule (e.g. uppercase a method name, or expose a constructor). The bootstrap step (`git apply --3way ../patches/*.patch`) applies them on a fresh checkout.

[SETUP.md → Patching tsgolint's typescript-go](../SETUP.md#patching-tsgolints-typescript-go) documents how to add a new patch.

### Maintenance model

- The oxc-project tsgolint repo tracks typescript-go via renovate. Whenever typescript-go bumps, tsgolint regenerates its shim and we pick it up via `git submodule update`. Drift is bounded and automatable.
- The runtime cost of going through a shim is **zero**: aliases and `//go:linkname` are link-time, not runtime constructs.
- When [microsoft/typescript-go#516](https://github.com/microsoft/typescript-go/issues/516) lands with an official transformer/linter API, we replace the shim imports with that API — call sites in [`ts-go-runtypes/internal/compiler/program`](../ts-go-runtypes/internal/compiler/program/), [`ts-go-runtypes/internal/compiler/marker`](../ts-go-runtypes/internal/compiler/marker/), and [`ts-go-runtypes/internal/compiler/resolver`](../ts-go-runtypes/internal/compiler/resolver/) change; everything else (sentinel detection, structural ids, hash dictionary, cache emission, Vite plugin) stays put.

## Workspace self-imports in tests

Two independent resolvers see `import { … } from '@ts-runtypes/core'` inside the marker package's own tests, and both must land on the same target:

1. **Vitest / Vite** — runtime resolution. Driven by `resolve.conditions: ['source']` in [`vitest.config.ts`](../packages/ts-runtypes/vitest.config.ts).
2. **tsgo** — the Go-side TypeScript checker invoked by [`@ts-runtypes/devtools`](../packages/ts-runtypes-devtools/) to compute runtype ids. Driven by `customConditions: ["source"]` in [`tsconfig.test.json`](../packages/ts-runtypes/tsconfig.test.json).

Both conditions select the `"source": "./src/index.ts"` entry on the marker package's `package.json` `exports` map. External consumers (without `source` in their conditions) fall through to the normal `"types"` / `"import"` / `"require"` entries, getting the built `dist/`. The `"source"` lane is opt-in and workspace-internal.

### Why two condition flags, not one resolver

Vite's resolver and tsgo's resolver are unrelated implementations. Vite reads `resolve.conditions` from `vitest.config.ts`. tsgo reads `customConditions` from the tsconfig it was launched with. Neither sees the other's setting. Wiring both is required for consistent behavior across the runtime path (vitest loads the test module) and the type-checking path (tsgo's marker scan walks call sites).

### How the marker gate plays into this

[`ts-go-runtypes/internal/compiler/marker/marker.go`](../ts-go-runtypes/internal/compiler/marker/marker.go) gates whether a `InjectRunTypeId<T>` reference counts as the marker by checking that the alias' declaration lives inside the configured marker package. The check accepts two forms:

1. The declaration is inside an ambient `declare module "<module>" { … }` block (used by [`ts-go-runtypes/internal/testfixtures/runtypes.d.ts`](../ts-go-runtypes/internal/testfixtures/runtypes.d.ts) and any other synthetic fixture without a real package.json on disk).
2. The declaration's file belongs to an on-disk package whose `package.json` `"name"` equals `<module>`. The check walks parent directories from the declaration file looking for the nearest `package.json` and reads its name. The on-disk directory name is irrelevant — only the `"name"` field matters.

Form 2 is what makes the source-condition path work. When tsgo resolves `@ts-runtypes/core` to `packages/ts-runtypes/src/index.ts`, the marker scanner walks up that path, finds `packages/ts-runtypes/package.json`, reads `"name": "@ts-runtypes/core"`, and accepts. The same check accepts files from `node_modules/@ts-runtypes/core/dist/index.d.ts` for external consumers (its enclosing `package.json` has the same name).

An earlier version of `DeclaredInModule` matched by **file-path string** instead of by `package.json` name — accepting any path containing `/@ts-runtypes/core/`. That worked for `node_modules` consumers but failed for workspace self-imports, where the on-disk directory (`packages/ts-runtypes/`) doesn't contain the published name. To work around it, the marker package shipped a `test/runtypes.d.ts` ambient overlay file mirroring the public API. The package.json walk removed that workaround — there's no overlay file for `@ts-runtypes/core` anymore.

The Go test suite still needs `ts-go-runtypes/internal/testfixtures/runtypes.d.ts` because the fixture files live under the monorepo's `ts-go-runtypes/internal/` tree, which doesn't have its own `package.json` and shouldn't pretend to be the marker package. The ambient-module form (gate form 1) is the right tool for synthetic fixtures.

## Validate contract — serializable data only

`createValidate<T>()` and `createGetValidationErrors<T>()` validate **serializable data**, not the full TypeScript type. Non-serialisable members (functions, methods, symbols, symbol-keyed properties, getters/setters with no backing data) are **silently dropped** from the validated shape with a build-time **Warning** diagnostic (VL010/VL011/VL012/VL013, VE010/…). This is by design — JSON drops them on the wire anyway, so validating against a JSON-shaped projection of `T` is the right semantic for the typical use case (RPC, persistence, network IO).

Consequence: `interface User { name: string; onClick: () => void }` produces a validator that only checks `name`. A user passing `{name: 'x', onClick: 'not-a-fn'}` will see `isUser(value)` return `true` — the schema **does not enforce** `onClick`. The VL010 warning at build time is the only build-time signal of this; do **not** treat it as an error.

This same rule applies to the JSON / binary serialisation families: a property that can't be serialised silently drops at the **property** position (with a per-family Warning code), while at the **root** or other propagating positions (array element, tuple slot, union member, function param/return) it generates an `alwaysThrow` factory (its throw message rendered at build time and embedded in the entry) with an **Error**-severity diagnostic — calling that factory throws at runtime, so the build halts.

The clean line: **Warning = expected drop, the user should know but it's fine**. **Error = will throw at runtime, build must fail**.

The plugin enforces the Error side in **every lane** via its `failOnError` option (default `true`; documented opt-out for programs that deliberately contain error-case types): `buildStart` surfaces every diagnostic family and rejects on Error severity — vitest and dev serve fail as loudly as `vite build` — the per-file transform paths halt the same way for files entering the program mid-session (HMR alone stays warnings-only mid-edit; the halt re-applies on the next run), and pure-fn extraction errors halt unconditionally (files-mode has no fallback for a failed generation). The whole-program scan's marker diagnostics (MKR/CTA/TMP) persist on the session and ride the `OpGenerate`/`OpDump` responses, so buildStart sees them in every bundler lane — including MKR007 (marker type resolved to `any` because the site's file has an unresolved import), which turns the silent accept-everything degrade into a halting diagnostic naming the file and specifier. **First-party scoping:** these diagnostics are consumer feedback, so `dispatchScanFiles` drops any anchored in a source file TypeScript resolved by *searching node_modules* (`program.IsSourceFileFromExternalLibrary`, folded once over the aggregated list — [`ts-go-runtypes/internal/compiler/resolver/scan.go`](../ts-go-runtypes/internal/compiler/resolver/scan.go)). A dependency normally resolves to its `.d.ts` (call-expression-free, already scan-inert), but when one resolves to its own `.ts` source — its package.json `source` export condition, or a consumer's `customConditions:["source"]` (e.g. `@ts-runtypes/core` ships `src/`) — the scan walks the library's own internal generics and would false-positive on them (`registerPureFnFactory`'s non-literal `CompTimeArgs<PureFnId>`, …). Using resolution *provenance* rather than a `node_modules`-in-path check keeps it correct through workspace symlinks, and leaves the marker package's own self-import (resolved via its `source` export, not a node_modules search) first-party so the RT suite still lints it. Marker collection is untouched — only the diagnostic list is scoped.

**Decoders return the data-only projection.** `createJsonDecoder<T>()` and `createBinaryDecoder<T>()` return `DataOnly<T>` (the `// #region dataonly-extract` type in [`src/runtypes/dataOnly.ts`](../packages/ts-runtypes/src/runtypes/dataOnly.ts)), NOT bare `T`. A decoded value is reconstructed from JSON/bytes, so it can only ever hold serialisable data — the old `=> T` over-promised methods/`Promise`s/symbols the value doesn't have (calling them type-checked but threw). On a clean DTO `DataOnly<T> ≡ T`, so nothing changes. The projection lives on the **factory overload return** (`JsonDecoderFn<DataOnly<T>>` / `BinaryDecoderFn<DataOnly<T>>`), not on the `JsonDecoderFn`/`BinaryDecoderFn` aliases (those stay `=> T`). **Encoders are unchanged** (they take `T` as input). This is purely a type-level annotation: no runtime or emitter change.

Future direction (out of scope for the current code): we may refine the return type to `ValidateFn<DataOnly<T>>` (where `DataOnly<T>` strips non-serialisable members from the type), rename `createValidate` → `createIsDataType`, or introduce a separate stricter `createIsFullType` that errors instead of dropping. Discuss in [docs/ROADMAP.md](./ROADMAP.md) before changing — current callers depend on the silent-drop semantics.

**Where these drops show up cross-library:** the [cross-library validation alignment report](./cross-library-validation-alignment-report.md) audits where the benchmark competitors (zod, TypeBox, ajv, typia) disagree with RunTypes about what counts as valid. The serializable-data drops above are the one place RunTypes is intentionally *different* (typia validates function-valued props where we drop them); every other divergence is a competitor being looser than RunTypes (ajv/typia accept `NaN`/`Infinity` as numbers, typia accepts Invalid Date), with RunTypes always backed by at least one other library.

## Circular-reference guard — a compile-time option, value-level

The validators / encoders are straight-line walkers with no cycle protection: a runtime **value** that contains a reference cycle (`a.next = a`) makes them recurse until the stack overflows. Arming the guard is a **compile-time option**, `{rejectCircularRefs: true}`, on the four live-object families — `createValidate`, `createGetValidationErrors`, `createJsonEncoder`, `createBinaryEncoder` (decoders take serialized input, which can't cycle). It folds into the `fnHash` exactly like `noLiterals`: the option is orthogonal to every axis, so it appends a `~C` suffix to a CircularGuarded operation's canonical key ([`operations/fnhash.go`](../ts-go-runtypes/internal/cachegen/operations/fnhash.go) `circularCanonicalSuffix`), and an armed factory and a plain one for the same `T` compile to **distinct entries** — pay-for-use, pinned by `TestRejectCircularRefsForksFnHash`. There is **no global toggle** (`setRejectCircularRefs` was removed); the fnHash mirror table (`fnHashes.generated.ts`) grows a `C`-suffixed variant per CircularGuarded family so `getFnHash` resolves the armed prefix too. The scanner reads `rejectCircularRefs` off the options literal for all four families ([`scan.go`](../ts-go-runtypes/internal/compiler/resolver/scan.go) `extractRejectCircularOption`) and forks the demand; it is **not** normalised away for an acyclic type (circularity is unknown at scan time), so arming an acyclic type mints a harmless duplicate entry.

The armed body **inlines** the check — no RunType graph at runtime:

- **Build** ([`ts-go-runtypes/internal/cachegen/typefunctions/circular_skeleton.go`](../ts-go-runtypes/internal/cachegen/typefunctions/circular_skeleton.go)): for an armed guarded entry over a cyclable type, `BuildCircularSkeleton` walks the RunType graph and computes a compact **path skeleton** — the pruned graph of cycle-capable positions: the property/element access paths that lead from one circular node (`RunType.IsCircular`) to the next, with acyclic intermediates collapsed into an edge's path. The walker (`Compile` → `emitCircularGuard`) bakes that skeleton into the factory closure as a const and prepends the family's guard reaction to the body — `if (fcp(v, cyP)) return false;` (validate), a push + `return er` (validationErrors), or `throw utl.circularError(cyR)` (the encoders; the JSON composites inline the same throw in `json_composite.go`). `fcp` is `utl.getPureFn('rt::findCycle')`, referenced through `ctx.UsePureFn`, so demand is by BODY reference — delivered like any built-in (no type-shape wiring). A plain (unarmed) cyclable type ships **nothing** — no guard, no walker, no RunType bundle.
- **Runtime** ([`circular-pure-fns.ts`](../packages/ts-runtypes/src/runtypes/circular-pure-fns.ts) `rt::findCycle`): `findCycle(value, skeleton)` does a restricted DFS from `value` following ONLY the baked circular edges, with a descent stack LOCAL to itself (add-on-descent / delete-on-ascent identity tracking) — shared DAGs pass, only a true back-edge flags, matching `JSON.stringify`. It needs no RunType graph and no ancestor stack threaded through the main traversal. Only the error class + path formatter stay static in `circular.ts`; `CircularReferenceError` is reachable from an emitted encoder body via `utl.circularError`.

## The lint surface — one pass, OXlint/ESLint transport

The compiler is also the lint engine. `packages/ts-runtypes-devtools/src/eslint/` ships an OXlint JS plugin (that doubles as an ESLint v9 flat-config plugin) whose rules are **pure transport** — no diagnostics logic lives in JS:

- **One resolver pass per file.** The rule pushes the buffer text (`setSources`, same pivot as HMR) and calls `scanFiles` with two opt-in flags: `checkEnrich` (the enrichment-health pass: tag hygiene FT020–FT022 / MD020–MD022 attributed to the mirror's family by [`hygiene.go`](../ts-go-runtypes/internal/enrichment/mirror/hygiene.go)'s `FamilyClassifier`, FriendlyText/MockData content FT/MD codes, breadcrumb drift GE002/GE003 — `FamilyEnrich`, produced by [`ts-go-runtypes/internal/compiler/resolver/enrichcheck.go`](../ts-go-runtypes/internal/compiler/resolver/enrichcheck.go) over [`ts-go-runtypes/internal/enrichment/mirror/hygiene.go`](../ts-go-runtypes/internal/enrichment/mirror/hygiene.go), [`drift.go`](../ts-go-runtypes/internal/enrichment/mirror/drift.go) and the shared [`ts-go-runtypes/internal/enrichment/astcheck`](../ts-go-runtypes/internal/enrichment/astcheck/) walk; hygiene's comment spans come from one per-file `mirror.Scan` built off the file's EXISTING parse — [`scanTags.go`](../ts-go-runtypes/internal/enrichment/mirror/scanTags.go)'s literal-token oracle, shared verbatim by lint, `gen --prune` and the restore-on-reappear index, so all three agree on what a carcass is) and `includeRtDiagnostics` (the RunType render diagnostics — VL010, PJ001, … — rendered exactly as a build would, with the module payload dropped from the wire). The same findings ship from the CLI: `ts-runtypes check <file> --json`, `ts-runtypes gen --check --json`.
- **Routing, not rules.** [`diagnosticRouting.ts`](../packages/ts-runtypes-devtools/src/eslint/diagnosticRouting.ts) maps each wire diagnostic to a rule grouped by **diagnostic family** (the Go catalog's code-prefix families) and NAMED for what it catches, not for severity. `PREFIX_TO_FAMILY` folds each prefix into a product family (JSON = PJ/PJS/RJ/SJ/JCP, binary = TB/FB, marker = MKR/CTA/PFN/TMP; `validate` absorbs `validationErrors` = VL/VE) and gives each family a `{primary, warn?}` rule pair. Within a family the two tiers are two DIFFERENT findings, so each has its own descriptive rule: a root the feature can't represent → `runtypes/validate-non-serializable` (error), a member it silently skips → `runtypes/validate-skipped-member` (warn); likewise `json-`/`binary-non-serializable` + `-skipped-member`, `clone-unsupported-type` + `clone-shared-reference`, `invalid-marker` + `redundant-marker`, `invalid-override` + `override-side-effect`. Enrich codes route per concern (`no-enrichment-todo`, `no-orphan-carcass`, `enrichment-field`/`enrichment-message`, `enrichment-broken-source`/`enrichment-misplaced-file`). Severity, not the name, picks the tier. **Severity is the host's job** — the plugin ships each rule at its Go-catalog default (in the `RULE_SPECS` single-source table that also builds `configs.recommended`) and the configured level is what applies. The `catalog coverage` test in [`routing.test.ts`](../packages/ts-runtypes-devtools/test/eslint/routing.test.ts) asserts every catalog code routes to a rule whose default matches its severity, so a new Go prefix or a severity move fails at PR time. Messages render from the same catalog the bundler plugin uses: the GENERATED dictionary [`diagnosticCatalog.generated.ts`](../packages/ts-runtypes-devtools/src/go-generated/diagnosticCatalog.generated.ts) (now carrying each code's `severity`), emitted by `pnpm run gen:diag-catalog` from the single Go source of wording ([`ts-go-runtypes/internal/diagnostics/messages.go`](../ts-go-runtypes/internal/diagnostics/messages.go)), with an unknown-code fallback (by wire family) so a diagnostic is never silently dropped. The mapping is transport-agnostic — an LSP sink can reuse it unchanged.
- **Sync bridge.** Lint rule visitors are synchronous; the resolver clients are Promise-based. [`session.ts`](../packages/ts-runtypes-devtools/src/eslint/session.ts) blocks the rule thread on `Atomics.wait` while a worker thread ([`lint-worker.ts`](../packages/ts-runtypes-devtools/src/eslint/lint-worker.ts)) owns one long-lived `--inline-server` connection for the whole run, memoized per (file, text-hash).
- **Pre-spawned launcher.** oxlint runs the Rust linter inside its Node process and reserves tens of GB of address space once linting starts, after which `fork()` fails on Linux — so the plugin's load is held (top-level await) until the worker has spawned the tiny [`spawn-shim.ts`](../packages/ts-runtypes-devtools/src/eslint/spawn-shim.ts) launcher; the first linted file hands it the real binary + argv over its own pipes. `RT_LINT_PRESPAWN=0` opts out for small hosts.
- **Tag single-source.** The dirty-tag literals (`@todo`, `@rtOrphan`, `@rtOrphanChild`) and the guard signals (`/** @rtType ` marker prefix, `: FriendlyText<` annotation form) live in [`ts-go-runtypes/internal/enrichment/mirror/tags.go`](../ts-go-runtypes/internal/enrichment/mirror/tags.go); the emitters, the Go detector, and (via `gen:ts-constants`) the JS pre-filters all derive from them, so emitter and detector cannot drift.

## Limitations

- The shim locks us into tsgo's internal API surface. A renovate-driven sync on the tsgolint submodule keeps it current.
- Lint-time analysis builds an inferred single-root Program per file (the `setSources` path): imports resolve from disk, but tsconfig compilerOptions are not applied, so tsconfig-sensitive diagnostics (e.g. the Temporal lib check) can differ from a build. A tsconfig-fidelity lint mode is a possible follow-up (see ROADMAP).
- Concurrency: `Cache` is not safe for concurrent use; the resolver holds one checker per process and serialises requests.
- A signature may carry SEVERAL injection markers (multi-slot injection) — `InjectRunTypeId<T>` for reflection sites and/or `InjectTypeFnArgs<T, Fn…>` for factory sites, in any parameter positions. Each marker parameter injects at its own index: the scanner emits one Site per injecting slot (shared closing-paren `Pos`), and the transform groups a call's slots (`groupSitesByPos` in [sourcerewrite/transform.go](../ts-go-runtypes/internal/compiler/sourcerewrite/transform.go)) into a single positional insertion, padding non-marker optional gaps with `undefined`. A marker slot the caller already filled is a pass-through (a forwarded handle), left untouched. The common single-marker call is a group of one and stays byte-identical. This is what lets a framework wrapper (mion's `route(handler, opts?, paramsFns?, responseFns?)`) carry a separate marker per side, each resolving its own `T`.
