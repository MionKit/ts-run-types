# Consolidate the Go→TS generated mirrors into one shared package

Status: **TODO — exploration + recommendation** (owner request 2026-07-18; not
committed to implementing). This doc inventories everything the Go side currently
autogenerates for the JS workspace, analyses the problems the current split does
and does **not** have, evaluates whether a single shared package fixes them, and
lands on a recommendation. It exists so the trade-off can be judged before any code
moves.

## The request

> Move all autogenerated files from Go to TS into a single new package that every
> other package consumes (devtools and core included). Anything the Go side currently
> autogenerates goes into this package, **except** the website-facing autogen, which
> keeps its current behaviour. The package is committed to the repo, and PR linting
> keeps it in sync with the Go source.

## Inventory — what the Go side generates for the JS workspace today

Generators are registered in **one place**: the `CODEGEN` map in
[scripts/rt.mjs](../../scripts/rt.mjs) (`pnpm rtx core codegen <target> [--check]`).
Every `run` is invoked directly; adding a mirror is one registry entry, no companion
`gen:*` script and no CI edit.

| Target (`rtx core codegen …`) | Generator | Committed output | Lives in | Consumed by | Direction |
| --- | --- | --- | --- | --- | --- |
| `constants` | `cmd/gen-ts-constants` | `runtypes-constants.generated.ts` (147 L) | **devtools** | devtools only (`protocol.ts`, `unplugin.ts`, `eslint/prefilter.ts`, tests) | Go→TS |
| `kind` | `cmd/gen-run-type-kind` | `runTypeKind.ts` (88 L) | **core** | core — **public API** (`index.ts` re-exports `RunTypeKind`/`RunTypeSubKind`) + `mocking/*` | Go→TS |
| `kind` | `cmd/gen-run-type-kind` (same run) | `reflectionKind.generated.ts` (86 L) | **devtools** | devtools only (`protocol.ts`) | Go→TS |
| `fnhashes` | `cmd/gen-fn-hashes` | `fnHashes.generated.ts` (61 L) | **core** | core only (`fnHash.ts`) | Go→TS |
| `diag` | `cmd/gen-diag-catalog` → `scripts/core/gen-diagnostics-catalog.mjs` | `diagnosticCatalog.generated.ts` (798 L) | **devtools** | devtools only (`diagnosticCatalog.ts`) | Go→TS |
| `diag` | (same run) | `container/website/app/components/content/diagnostics-catalog.json` | **website** | website diagnostics page | Go→JSON — **stays put per the request** |
| `builtinpurefns` | `cmd/gen-builtin-purefns` | `internal/cachegen/builtinpurefns/table.generated.go` | **Go internal** | Go resolver | **TS→Go (reverse)** — not a candidate |

Candidate set (Go→TS, committed, JS-consumed): **`constants`, `runTypeKind`,
`reflectionKind`, `fnhashes`, `diagnosticCatalog`** — five files, ~1180 lines.

Two items are **not** movable as framed:

- **`builtinpurefns`** flows the other way: it extracts `rt::`/`rtFormats::` bodies
  from `packages/ts-runtypes/src` **into a Go table**. It is a `.go` file consumed by
  the resolver; it does not belong in a TS package.
- **`diag`'s website JSON** shares one generator run with the devtools TS dict (both
  fan out of a single `go run ./cmd/gen-diag-catalog` dump). "Website autogen stays"
  means that one generator would have to write *into the new package* (the TS dict)
  **and** into `container/website/` (the JSON) — its outputs straddle the boundary.

### Consumer split (the important part)

- **core-only:** `runTypeKind` (public API), `fnHashes`
- **devtools-only:** `constants`, `reflectionKind`, `diagnosticCatalog`
- **conceptually shared:** only the numeric **kind discriminators**, and even those
  ship today as **two deliberately different shapes** — core's `RunTypeKind`
  const-object (tree-shakeable, plus `RunTypeKindValue` etc.) vs devtools' `enum
  ReflectionKind` + `KIND_REF = -1` + `REFLECTION_SUB_KIND`.

So **exactly one** contract is genuinely duplicated (the kind discriminators). The
other four files are single-consumer and already sit in their natural home.

## Does the current architecture actually have the problem this solves?

The request's motivation is "scattered generated files → drift risk → consolidate +
lint". Measured against the codebase:

1. **Drift is already gated on every PR.** `pnpm rtx core codegen all --check`
   (regenerate + format + `git diff --exit-code` over the registry's `outputs`) runs
   in [ci.yml](../../.github/workflows/ci.yml) `js-lint` **and**
   [release-gate.yml](../../.github/workflows/release-gate.yml). The `rtx` registry
   owns the generator list, so a new mirror is picked up for free. Go-side unit tests
   (`gen_test.go`, `TestRunTypeKindFileInSync`) pin the same contract. **The
   "add linting to keep in sync" ask is essentially already delivered** — and it is
   agnostic to how many packages the outputs live in (it diffs paths, not packages).
   Consolidation buys **nothing** here.

2. **The double-mirror is real but drift-safe.** `runTypeKind.ts` and
   `reflectionKind.generated.ts` are written by **one generator from one protocol
   parse**, so they cannot silently diverge. The cost today is cosmetic: two files +
   two shapes to hold in your head. This is the one genuine wart.

3. **Discoverability is a doc problem, not a package problem.** "Where does Go
   generate JS?" is answered by the `CODEGEN` map. A single package would make it
   more physically obvious, but that is a small, non-functional gain.

## The crux tension: dedup vs. a zero-dependency `@ts-runtypes/core`

This is the deciding constraint.

- **`@ts-runtypes/core` markets zero runtime dependencies.** [README.md](../../README.md)
  line ~30: *"Zero runtime dependencies. The only thing in your bundle is the small
  [marker package]."* It is a headline property of the flagship package.
- **Both FE packages build with plain `tsc --build`** — multi-file dist, **no
  bundler**, by explicit design (CLAUDE.md: *"the plugin carries no bundling
  deps"*). A `tsc` emit does **not** inline or tree-shake cross-package imports.

Therefore, if core imported a shared `@ts-runtypes/generated`:

- The emitted dist keeps a literal `import { RunTypeKind } from '@ts-runtypes/generated'`
  → the shared package becomes a **real runtime dependency** → **core's zero-dep
  claim breaks** (a `ts-runtypes` install pulls a second package).
- Avoiding that means **introducing a bundler into core just to inline generated
  constants** — which directly contradicts the repo's bundler-free stance and adds
  build surface.
- And because `tsc` does not tree-shake a barrel, a naive shared-package index means
  core would load the devtools-only 798-line diagnostic catalog + cache-module
  constants at module-eval time. Avoiding that needs careful subpath exports
  (`@ts-runtypes/generated/kind`, `/fnhashes`, …) — extra config.

There is **no clean way to both dedup the mirror and keep core zero-dep + bundler-free.**
That tension is the whole story.

## Costs a new published package adds

1. **Regresses a marketed property** (zero-dep core) or forces a bundler — pick your
   poison.
2. **A 4th first-party published package** in the lockstep release: dual ESM/CJS
   exports like core, the `source` export condition wired for **both** core and
   devtools dev/test resolution, 3 tsconfigs + vitest config, and additions to the
   publish topo-order / `pack.mjs` / `stage-approve.mjs` / `publish.mjs` filter list /
   `bump-version.mjs`. The release machinery is intricate; +1 package is ongoing cost.
3. **Transitive weight** for every `ts-runtypes` consumer (extra node_modules entry
   unless bundled).
4. **Couples core to devtools-only data** — core would depend on a package that also
   carries the diagnostic catalog + bundler-plugin constants it never uses.
5. **The `diag` generator straddles the boundary** (writes both the new package and
   the website JSON), so "move the target" is not a clean cut.

Against all that, the benefit is: **dedup one 86-line file** and **make the generated
home more obvious**. The sync guarantee — the stated motivation — is unchanged.

## Options

- **Option A — full published `@ts-runtypes/generated`** (the literal request). Highest
  effort, hits the zero-dep property, modest benefit. Only pays off if the project
  expects *many* more cross-consumed wire constants than today's one.
- **Option B — private, source-only internal package** (`"private": true`, never
  published), imported by core/devtools **only via the `source` condition** in
  dev/test, while each publishable package still emits its **own** copy of the
  constants it uses into its dist (generator writes into both the shared source and
  each consumer's dist input). Preserves zero-dep + bundler-free at publish, gives one
  editable home in dev. But with `tsc` this "one source, per-package emit" needs a
  copy/codegen step that largely reproduces today's multi-target generation — i.e. it
  is close to a rename of the status quo.
- **Option C — keep the split; pay down only the real wart.** Leave each mirror in its
  consumer, keep the `--check` gate, and, if the double-mirror bothers us, reconcile
  `RunTypeKind`/`ReflectionKind` to one shape generated into both files (still two
  files, one shape) or document the generated-file map in
  [docs/ARCHITECTURE.md](../../docs/ARCHITECTURE.md).

## Recommendation

**Do not implement Option A.** The change's headline motivation (keep the mirrors in
sync) is already satisfied by the existing `codegen --check` gate + Go tests, so
consolidation adds no sync safety. Its real, non-trivial cost is regressing
`@ts-runtypes/core`'s zero-runtime-dependency guarantee (or importing a bundler into a
deliberately bundler-free package) plus standing up and maintaining a fourth
lockstep-published package — to dedup a single 86-line file and improve
discoverability. The benefit does not justify the cost at today's scale.

**Revisit if** the count of constants that *both* core and devtools must share grows
well beyond the one it is today (e.g. a shared binary-format tag table, more dual-
consumed protocol enums), **or** if core adopts a bundler for another reason — either
event flips the amortisation and makes Option A cheap.

**If we want to act now, prefer Option C**: keep the generated files where their
consumers are, lean on the existing drift gate, and (optionally) add a short
"Generated Go→TS mirrors" section to `docs/ARCHITECTURE.md` pointing at the `CODEGEN`
registry as the single index. Low risk, keeps every current property, removes the
"where does this live" confusion that motivated the request.

## If Option A is chosen anyway — concrete plan

1. **Scaffold `packages/ts-runtypes-generated`** (`@ts-runtypes/generated`), zero-dep,
   dual ESM/CJS exports mirroring core's `package.json`, with a `source` condition and
   **subpath exports** (`./kind`, `./fnhashes`, `./constants`, `./diagnostics`) so
   `tsc` consumers only pull what they import. Add `tsconfig.json` / `tsconfig.cjs.json`
   / `tsconfig.test.json` / `vitest.config.ts`.
2. **Reconcile the kind discriminators to one shape** consumed by both core and
   devtools (decide const-object vs enum; keep `KIND_REF`/`REFLECTION_SUB_KIND`). This
   is a **public-API change** for `@ts-runtypes/core` (it re-exports `RunTypeKind`) —
   preserve the existing names via re-export or accept a breaking rename in ROADMAP.
3. **Retarget the generators** in `scripts/rt.mjs` `CODEGEN` `outputs`/`stdoutTo`/`fmt`
   to the new package; split `gen-run-type-kind` to emit one file. The `diag` generator
   keeps writing the website JSON to `container/website/` **and** the TS dict to the new
   package (its outputs straddle both).
4. **Update importers**: core `index.ts` / `fnHash.ts` / `mocking/*`; devtools
   `protocol.ts` / `unplugin.ts` / `eslint/prefilter.ts` / `diagnosticCatalog.ts`; all
   tests. Add `@ts-runtypes/generated` as a `workspace:*` dep of core and devtools —
   **accept that core is no longer zero-dep** (update README's claim) or add a bundle
   step to inline it.
5. **Wire the release machinery**: publish topo-order (before core + devtools),
   `pack.mjs`, `stage-approve.mjs`, `publish.mjs` filter, `bump-version.mjs`,
   lockstep version pin.
6. **Sync lint**: no new mechanism needed — `rtx core codegen all --check` already
   diffs the registry's `outputs`; it follows the files to their new paths for free.
7. **Leave `builtinpurefns` (TS→Go) untouched.**
