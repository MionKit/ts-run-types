# 03 — getTypeErrors: mion vs ts-go-run-types port audit

> Date: 2026-05-29 · Method: static source comparison (Go binary + Vitest suites NOT executed here; test registrations counted, marked "counted, not executed"). Central run reported: `getTypeErrors.test.ts` passed with 0 failures; `formatGetTypeErrors.test.ts` had ~29 skipped/`it.todo`.

## 1. Verdict

⚠️ **Ported with gaps.** Every mion `emitTypeErrors` override that maps to a serializable reflection kind has an equivalent arm in the single Go switch `internal/compiled/typefns/typeerrors.go`, the three-arg factory shape (`v, pth=[], er=[]) => er`) and the `cpf_newRunTypeErr(pth, er, expected, accessPath?)` push contract are faithful ports of mion's `JitErrorsFnCompiler` + `cpf_newRunTypeErr`, and the union "one error, not N" semantics, path accumulation, and never/any/non-serializable finalize all match. **However two wire-contract divergences exist in the iterable path segments** — Map *value* errors use `failed:'mapValue'` where mion emits `failed:'mapVal'` (§5 #1, **High**), and Set item errors use a bare numeric index where mion emits a `{key,index}` object (§5 #2, **Med**) — and **both are blessed by the test fixtures rather than caught**. Plus the headline test gap: the format-error surface has **29 `it.todo`** in `formatGetTypeErrors.test.ts` (§5 #3). Three by-design deviations carry over verbatim from isType (Promise validated as thenable instead of throwing; function arity/param validation omitted; non-serializable array element emits an unconditional error instead of an alwaysThrow factory). The `emitIsTypeErrors` "combined emit" is a FORMAT-runtype concept (`BaseRunTypeFormat`), not a node override — covered via the format hook (§3, §5 #3), not a missing kind.

## 4. Intentional deviations (by design)

1. **Promise validated as thenable, not thrown.** mion `promise.ts:20-22` throws in `emitTypeErrors`; the port emits a thenable check (typeerrors.go:350-356) and `Supports` returns true (typeerrors.go:83-86). Identical, deliberate divergence to isType (02-istype §5 #1/#3); reconciled in `docs/UNSUPPORTED-KINDS.md`. typeErrors and isType stay consistent with each other.
2. **Union = one error, not per-arm.** Both mion (union.ts:101-106) and the port (typeerrors.go:990-999) delegate to the isType boolean and push a single `'union'` error. mion explicitly does NOT report a "best matching arm" breakdown; the port matches. (Answers task item #4.)
3. **`callJitErr` accumulates ALL leaf errors (no short-circuit).** Both sides record every failing leaf into `er` and keep walking — object children are joined with `;` (not `&&`), array/map/set loops run to completion, optional guards only gate presence. No early `return`. The port mirrors mion exactly. The *only* short-circuit is mion's index-signature template-key `continue` (skip value check when key already failed) — the port reproduces it via `; else` (typeerrors.go:760-776). (Answers task item #4.)
4. **`emitIsTypeErrors` is a FORMAT concept, not a node.** `grep emitIsTypeErrors nodes/` → 0 hits; the symbol lives only on `BaseRunTypeFormat` (baseRunTypeFormat.ts:226) and is dispatched from `compileFormat` for `JitFunctions.typeErrors.id` (baseRunTypeFormat.ts:171-173). It is the FORMAT runtype's combined "validate-as-base-type + push format error" emit (string/number/bigint format base classes). In the port this is the format hook in `Emit` (typeerrors.go:165-196) calling the per-format `EmitTypeErrorsCheck`. Behaviour is *structurally* covered; the test surface is stubbed (§5 #3). (Answers task item #5.)
5. **Single switch vs per-node classes / resolve-time options.** Architectural (overview §"Why the architectures differ"); not behavioural. Same as isType.
6. **CodeS statement-blocks for collections.** The port composes `if(!…){err} else {…}` blocks where mion sometimes uses the same; iterables get an explicit `;` separator after dep-call comma-expressions to avoid `(expr)const`/`(expr)i++` syntax errors (typeerrors.go:921-923, 946-948, 1031-1033) — a JS-codegen detail, no behaviour change.

## 5. Gaps, mismatches & missed optimisations

1. **Map *value* error path segment uses `failed:'mapValue'` — mion emits `failed:'mapVal'` (wire-contract divergence).** Severity: **High** (consumer-visible wire shape; the type name in mion's public `MapValuePathSegment` is `'mapVal'`). Evidence: mion `nodes/native/map.ts:79` `return \`{key:${safeKeyFn}(${custom.vλl}[0]),index:${parent.getIndexVarName(comp)},failed:'mapVal'}\`;` and the public type `packages/core/src/types/general.types.ts:110` `export type MapValuePathSegment = {key: unknown; index: number; failed: 'mapVal'};`. Port: `internal/compiled/typefns/typeerrors.go:934` `ctx.SetChildPathLiteral("{key:" + safeKey + "(" + entryVar + "[0]),index:" + idxVar + ",failed:'mapValue'}")` (key side at :905 correctly uses `'mapKey'`). The JS type comment also propagates it: `createRTFunctions.ts:47` `\`{key, index, failed: 'mapKey' | 'mapValue'}\``. Impact: any consumer that branches on `seg.failed === 'mapVal'` (mion's documented value) gets `'mapValue'` from the port — silent mis-routing of Map-value validation errors; breaks drop-in compatibility with mion-aware error renderers (e.g. `@mionjs/core` friendlyErrors). **The test suite does NOT catch this — it asserts the port's own value:** `test/suites/validation-suite.ts:7382,7385` `[{path: [{key: 'a', index: 0, failed: 'mapValue'}], expected: 'number'}]`. Fix: change `'mapValue'` → `'mapVal'` at typeerrors.go:934, the comment at createRTFunctions.ts:47, and the two suite fixtures.

2. **Set item error path segment is a bare numeric index — mion emits a `{key,index}` object.** Severity: **Med** (wire-contract divergence; structurally different segment type). Evidence: mion `nodes/native/set.ts:44-48` `getStaticPathLiteral` returns `\`{key:${safeKeyFn}(${custom.vλl}),index:${parent.getIndexVarName(comp)}}\``, and the public type `general.types.ts:113` `export type SetItemPathSegment = {key: unknown; index: number};`. Port: `internal/compiled/typefns/typeerrors.go:1018` `ctx.SetChildPathLiteral(idxVar)` — a bare loop counter (number), not an object. Impact: for `Set<string>` with an invalid member the port produces `path:[0]` where mion produces `path:[{key:<safeItem>,index:0}]` — consumers lose the offending member value and the segment shape differs. **Test suite blesses the port's bare-index shape:** `test/suites/validation-suite.ts` set_string `getExpectedErrors` → `[{path: [0], expected: 'string'}]`. Fix: emit `{key:safeKey(item),index:i}` in `emitSetTypeErrors` (register `mapSafeKeyContextItem`), then update the suite fixtures; or, if the simpler index is intentional, document the divergence in `docs/ROADMAP.md` and note mion-incompatibility.

3. **Format-error getTypeErrors test surface is heavily stubbed — `29` `it.todo` (tests left behind).** Severity: **Med** (no shipped-code defect — the Go format hook is wired — but the format-error wire contract is largely unverified). Evidence: `test/adapters/formatGetTypeErrors.test.ts` has **29 `it.todo(...)`** registrations vs **37 live `it()`**; all 29 todos are in the STRING_FORMAT block (NUMBER_FORMAT: 11 live / 0 todo; BIGINT_FORMAT: 9 live / 0 todo). Stubbed (typeerrors-relevant) cases include: FormatString minLength / length / minLength+maxLength / allowedChars(ignoreCase, regex-special) / disallowedChars / allowedValues(ignoreCase, regex-special) / disallowedValues (formatGetTypeErrors.test.ts:77-87); FormatAlphaNumeric / Numeric / Alpha+maxLength / Lowercase (:91-94); FormatUUIDv7 (:96); FormatStringDate DD-MM-YYYY / YYYY-MM / MM-DD (:98-100); FormatStringTime ISO / HH:mm:ss[.mmm] (:101-103); FormatStringDateTime custom layouts (:105); FormatIPv6 / IP(both) / IPv4WithPort / IPv6WithPort (:107-110); FormatEmailPunycode (:114); FormatUrlHttp / UrlFile (:117-118); registerFormatPattern slug / {source,flags} (:119-120). Impact: the `cpf_formatErr` push shape (`{name, formatPath, val}` inside `RunTypeError.format`) is only exercised for a thin slice of string formats plus number/bigint — most string-format `format.val`/`formatPath`/`name` outputs are unverified against the port. Fix: fill the 29 `it.todo`s with `getExpectedErrors` fixtures as items 07-09 land the format emitters; track in items 07-09 follow-ups.

4. **`expected:'objectLiteral'` for the TS `object` kind — mion emits `'object'`.** Severity: **Med** (wire-contract: the error `expected` string differs). Evidence: mion `nodes/atomic/object.ts:18` `… ${comp.callJitErr(this)}` with kind 30 → `constants.kind.ts:42` `30: 'object'` → `expected:'object'`. Port `internal/compiled/typefns/typeerrors.go:285` `callRTErr(ctx, "objectLiteral", "")`. (For isType this is invisible because isType returns a boolean — 02-istype §3 maps `object`→`objectLiteral` but never surfaces the name; for typeErrors the string is the contract.) Impact: `createGetTypeErrors<object>()` against a primitive reports `expected:'objectLiteral'` where mion reports `expected:'object'`. **Verify whether the suite exercises a bare-`object`-typed case** (no `object`-kind invalid fixture was located — likely UNVERIFIED in tests, so undetected). Fix: emit `"object"` at typeerrors.go:285 to match mion's distinct `object`-vs-`objectLiteral` naming.

5. **Non-serializable array element emits an unconditional `'array'` error instead of an alwaysThrow factory.** Severity: **Low** (mirrors isType Gap #3). Evidence: mion `nodes/member/array.ts:148` `checkNonSkipTypes` **throws** "Arrays can not have non serializable types" at RT-compile; port `internal/compiled/typefns/typeerrors.go:411-417` emits `callRTErr(ctx, "array", "")+';'` (every input → one `'array'` error) for symbol/function element kinds. Impact: `createGetTypeErrors<symbol[]>()` returns a validator that always reports one `'array'` error rather than throwing at factory creation; inconsistent with the positional-member CodeNS→alwaysThrow rule (UNSUPPORTED-KINDS.md). Functionally a hard rejection (no false-negatives). Fix: return `CodeNS` to render an alwaysThrow factory (parity with mion's throw), or document the carve-out alongside the isType one.

6. **`strictTypes` option not plumbed.** Severity: **Low** (reserved; same as isType Gap #4). Evidence: mion `interface.ts:104-113` adds an unknown-key `for…of` that pushes `'never'` errors per excess key when `comp.opts.strictTypes`; port `emitObjectTypeErrors` (typeerrors.go:572-659) has no such clause and `RunTypeOptions.strictTypes` is "Reserved — not yet plumbed" (createRTFunctions.ts). Impact: callers can't get excess-property errors from `getTypeErrors`; the dedicated `unknownKeyErrors` family (tag `uke`) covers it. Fix: wire `strictTypes` (fork hash + add the unknown-key clause) or leave to the unknown-keys family by design.

7. **Index-signature template-literal-key error uses `expected:'never'` — mion's main `emitTypeErrors` uses `expected:'indexSignature'`.** Severity: **Low** (narrow surface — only template-literal index keys, e.g. `{[k: \`api/${string}\`]: T}`; and `'never'` is arguably the better semantic for "this key shouldn't exist"). Evidence: mion `nodes/member/indexProperty.ts:73` (the `emitTypeErrors` override) `… ${comp.callJitErrWithPath(this, prop)} …` → `expected = this.getKindName() = 'indexSignature'`; the `'never'` form is only in the *separate* `emitUnknownKeyErrors` (indexProperty.ts:129). Port `internal/compiled/typefns/typeerrors.go:769` `callRTErr(ctx, "never", keyVar)`. Impact: a key failing the template pattern reports `expected:'never'` (port) vs `expected:'indexSignature'` (mion). UNVERIFIED in tests (no template-literal-key invalid fixture located). Fix: emit `'indexSignature'` (pass the kind name) to match mion's `emitTypeErrors`, or accept `'never'` and note the cross-method choice.

8. **Function arity guard / param-arity omitted.** Severity: **Low** (mirrors isType Gap #2). Evidence: mion `function.ts:77-78` reuses `emitIsType` (which includes `v.length >= minLength [&& v.length <= max]`, function.ts:69-76) then negates; port `typeerrors.go:367-375` emits only `if (typeof v!=='function') err(<kind>)`. Impact: a function of the wrong arity passes the port's `getTypeErrors` but fails mion's. Functions are non-serializable; tiny surface. Fix: optionally re-add `v.length >= N`.

9. **`functionParams` (`Parameters<F>`) length/param validation not emitted.** Severity: **Low** (consistent with the opaque-function model). Evidence: mion `nodes/collection/functionParams.ts:29-40` `emitTypeErrors` checks `v.length` and validates each param's typeErrors with `expected:'params'`; the port has no `KindFunctionParams` / params arm — function-flavoured kinds are opaque `typeof === 'function'` (typeerrors.go:367-375), and `KindFunction`'s params/return children are not walked for typeErrors. Impact: `createGetTypeErrors<Parameters<F>>()`-style param validation is unavailable; matches the isType treatment. Fix: add a params arm if function-parameter validation becomes in-scope (cross-cutting with the isType arity decision).

## 6. Test-coverage comparison

**mion specs** (counted, not executed; the same `nodes/**/*.spec.ts` files test isType+typeErrors+json+binary — typeErrors-relevant assertions attributed). Per the audit progress reference, error assertions live across `atomic/*.spec.ts`, `collection/*.spec.ts` (union/tuple/interface/templateLiteral/class/circularRefs), `member/*.spec.ts` (array/indexProperty/tupleMember), `native/*.spec.ts` (map/set/promise/nonSerializable).
