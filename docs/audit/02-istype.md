# 02 — isType: mion vs ts-go-run-types port audit

> Date: 2026-05-29 · Method: static source comparison (Go binary + Vitest suites NOT executed here; test registrations counted, marked "counted, not executed"). The full JS suite was reported as executed centrally → `isType.test.ts` passed with 0 failures; only the FORMAT adapters carry `it.todo`, not isType.

## 1. Verdict

✅ **Fully ported** (with documented by-design divergences). Every mion `emitIsType` override has an equivalent arm in the single Go switch `internal/compiled/typefns/istype.go`, and the emitted JS is byte-comparable for the atomic/collection/member kinds. Three intentional behavioural deviations exist (Promise validated as thenable rather than throwing; function arity guard omitted; non-serializable array elements emit `return false` instead of throwing at RT-compile). Two minor missed optimisations (redundant union object guards; broad compound de-inlining). No correctness gaps found. The two "stale TODO" comments in `validation-suite.ts` flagged for investigation describe a past state — the corresponding isType cases are fully wired and NOT stubbed.

## 4. Intentional deviations (by design)

1. **Non-serializable unification → CodeNS / alwaysThrow.** mion scatters the decision: symbol/function/Promise/never/NonSerializable each either `skipJit` (property drop) or `throw` inside their emit / parent (`array.checkNonSkipTypes`, `union.checkAllowedChildren`, `nonSerializable.emitIsType`). The port unifies on the two-rule model (docs/UNSUPPORTED-KINDS.md): property position → silent drop + Warning IT01x (istype.go:1295-1302); every other position → `CodeNS` propagates to an alwaysThrow factory. Observable result matches mion (property dropped, root/element throws) but the throw moves from RT-compile-time to `createIsType()`-call-time.
2. **`noLiterals` / `noIsArrayCheck` handled at resolve time, not emit time.** Both options fork the runtype id/flags in `scan.go` so the existing kind arms need no per-option branching. `RunTypeOptions` is declared on the JS side (`createRTFunctions.ts:31-41`) and read by the Go marker scanner; baked into the validator hash so variants resolve to distinct cache entries.
3. **CodeRB for tuple/array/map/set/index-sig.** The port composes statement-blocks (`if(!…)return false; … return true`) where mion sometimes uses expression chains, sidestepping mion's documented bug of mixing for-loops with `&&`-chains (istype.go:473-481).
4. **Single switch vs per-node classes.** Architectural (overview §"Why the architectures differ"); not behavioural.

## 5. Gaps, mismatches & missed optimisations

1. **Promise validated as thenable instead of throwing — behavioural divergence.** Severity: **Low** (by-design, but diverges from mion AND from this repo's own doc). Evidence: mion `promise.ts:17-19` `emitIsType(){ throw new Error('Jit compilation disabled for Non Serializable types.') }`, asserted by `promise.spec.ts:13-20`; port `istype.go:324-333` emits `typeof v === 'object' && v !== null && typeof v.then === 'function'`, and `Supports` returns true (istype.go:82-87). Impact: `createIsType<Promise<string>>()` returns a working validator in the port (any thenable passes) where mion throws at compile. Also **was inconsistent with `docs/UNSUPPORTED-KINDS.md:44`** which lists `KindPromise → XX002` (unsupported/throws). Fix: this is deliberate (validation-suite.ts:7445-7482 documents it as "TS DIVERGENCE"). **RESOLVED in this audit pass** — UNSUPPORTED-KINDS.md now carries a "`KindPromise` is validation-supported" note clarifying that `IT`/`TE` validate thenable-ness and only the serialization families throw. No code change required for correctness.

2. **Function arity guard omitted.** Severity: **Low** (by-design, documented). Evidence: mion `function.ts:69-76` emits `(typeof v === 'function' && v.length >= minLength [&& v.length <= maxLength])`; port `istype.go:462-471` emits only `typeof v === 'function'` and explicitly documents the omission. Impact: a function with the wrong arity passes the port's validator but fails mion's. Functions are non-serializable anyway (validated only at top level / via `Parameters<F>`), so the practical surface is tiny. Fix: optionally re-add the `v.length >= N` clause using the param run-types' required-count; otherwise leave as documented divergence.

3. **Non-serializable array element emits `return false` instead of an alwaysThrow factory.** Severity: **Low**. Evidence: mion `array.ts:146-148` `checkNonSkipTypes` **throws** "Arrays can not have non serializable types" at RT-compile; port `istype.go:387-389` returns `RTCode{Code:"return false", Type:CodeRB}` — an always-false validator — for symbol/function element kinds (`isNonSerializableElementKind`, istype.go:1429). This is a *third* path distinct from the CodeNS→alwaysThrow rule used everywhere else for positional members (UNSUPPORTED-KINDS.md:35 says array element should propagate to alwaysThrow). Impact: `createIsType<symbol[]>()` returns `()=>false` (every input rejected) rather than throwing; mildly inconsistent with the tuple-slot/union-member positional rule which DOES propagate CodeNS. Functionally a hard rejection, so no false-positives. Fix: for parity with the positional-member rule, return `CodeNS` here too so the root renders an alwaysThrow factory (matching mion's throw); or keep `return false` and document the intentional carve-out. Suite covers it: `ARRAY.symbol_array` ("non-serializable — always rejected").

4. **`strictTypes` option not plumbed.** Severity: **Low** (reserved). Evidence: mion `interface.ts:84-87` adds an unknown-property rejection to the isType body when `comp.opts.strictTypes`; port `RunTypeOptions.strictTypes` is "Reserved — not yet plumbed" (createRTFunctions.ts:39) and `emitObjectIsType` notes it as the "one remaining mion knob not yet wired" (istype.go:1139-1141). Impact: callers cannot get strict (excess-property) isType validation; the dedicated `hasUnknownKeys`/`stripUnknownKeys` families (tags `huk`/`suk`) cover the use case instead. Fix: wire `strictTypes` through `scan.go` (fork hash) + add the unknown-prop clause in `emitObjectIsType`, or leave to the unknown-keys families by design.

5. **Redundant per-object `typeof === 'object' && !== null` guards inside unions (missed optimisation).** Severity: **Low**. Evidence: mion suppresses the inner object guard for union members via `isPartOfUnion()` returning `''` (interface.ts:88), keeping exactly one shared guard; port `emitUnionIsType` explicitly keeps the inner guards ("pre-mature optimization to strip them is fragile", istype.go:783-789) AND wraps them in the shared outer guard (istype.go:777-789). Impact: each object union arm re-runs `typeof===object && !==null` — extra constant-factor work, no correctness effect. Fix: strip the leading guard from each object arm (textual strip is safe — object emit always starts with that literal prefix) or set an "in-union" flag so `emitObjectIsType` omits the guard, mirroring `isPartOfUnion`.

6. **All compound kinds de-inlined, not just named/circular (missed optimisation).** Severity: **Low**. Evidence: mion `baseRunTypes.ts:52-61` inlines anonymous non-circular collections; port `DefaultIsRTInlined` returns false for *every* ObjectLiteral/Class/Tuple/Union (inlining.go:106-129) because the serializer doesn't populate `TypeName` on anonymous interfaces and `IsCircular` isn't auto-set. Impact: anonymous non-circular composites get their own dependency-call factory where mion would inline — a small per-type constant cost, no correctness impact (documented inlining.go:75-86). Fix: when serializer-side circular detection + TypeName population lands, flip these arms back to "inline unless circular or named".

7. **Intersection loose-check coverage narrower than mion (latent, currently moot).** Severity: **Low**. Evidence: mion's `getChildIsTypeWithLooseCheck` applies the weak-type gate to `isIntersectionRunType` too (union.ts:59); port `looseCheckGate` only matches `KindObjectLiteral`/`KindClass` (json_prepare.go:612). Impact: none today — both mion (intersection.ts:18) and tsgo pre-resolve intersections to ObjectLiteral before the union arm runs, so a raw `KindIntersection` never reaches `looseCheckGate`. Fix: none needed unless a future tsgo version surfaces an unresolved intersection union-member; add `KindIntersection` to the `looseCheckGate` kind check defensively if so.
