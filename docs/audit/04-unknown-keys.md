# 04 — unknown-keys family: mion vs ts-go-run-types port audit

> Date: 2026-05-29 · Method: static source comparison (Go binary + Vitest suites NOT executed here; test registrations counted, marked "counted, not executed"). Per the port-status snapshot, `@mionjs/ts-go-run-types` reported 867 pass / 0 fail, incl. the 12 Map/Set unknown-keys tests; `test/adapters/unknownKeys.test.ts` has 45 `it()`, 0 todo/skip.

## 1. Verdict

⚠️ **Ported with gaps.** All five functions — `hasUnknownKeys` (`huk`), `stripUnknownKeys` (`suk`), `unknownKeyErrors` (`uke`), `unknownKeysToUndefined` (`uku`), plus the decoder-internal wire sibling `unknownKeysToUndefinedWire` (`ukuw`) — are present as five Go emitters, five JS factories (four public + the internal `ukuw` via `createJsonDecoder`), and the two pure-fns `cpf_getUnknownKeysFromArray`/`cpf_hasUnknownKeysFromArray` (faithful ports of mion's, incl. the `MAX_UNKNOWN_KEYS = 10` cap). The object/property/array/tuple/tupleMember/indexSignature/Map/Set/atomic per-kind matrix matches mion node-for-node; the four `Finalize` identities (huk→`return false`, suk→`return v`, uku→`return v`, uke→`return er`) are exact; the **symbol-keyed index-signature skip** (mion `skipRT`) is reproduced in all five emitters; and the **known-keys array `sort.Strings`** divergence is a documented determinism trade-off (§4 #1). **Two material divergences exist:** (1) the **union arm is re-architected** from mion's per-member OR/`;`-join onto a port-only "merged-allowlist" over the flat-union layout — a deliberate *correctness fix* (mion's per-member emit mis-reports/over-deletes), and the **`ukuw` wire sibling is entirely port-specific** infrastructure with no mion equivalent (§4 #2, #3); (2) two **wire-contract mismatches carried over from getTypeErrors** surface in `uke`'s Map/Set path segments — Map *value* uses `failed:'mapValue'` where mion emits `'mapVal'` (§5 #1, **High**), and Set items use a bare numeric index where mion emits `{key,index}` (§5 #2, **Med**). Plus a JS-API option rename `checkNonJitProps`→`checkNonRTProps` (§5 #3, **Med**, drop-in incompatibility). The mion `nonSerializable` compile-throw becomes a silent noop (§5 #4, **Low**) and the array-element non-serializable throw is dropped (§5 #5, **Low**) — both consistent with the port's JSON-family philosophy. `KindIntersection` has no Emit arm but is **unreachable** (intersections are collapsed to objects pre-emit) — not a gap (§4 #4).

## 4. Intentional deviations (by design)

1. **Known-keys arrays sorted (`sort.Strings`) vs mion insertion order.** Evidence: mion `addObjectPropsToContext` (interface.ts:248-249) `Array.from(new Set(jitArrNames))` preserves source declaration order; port `dedupSortStrings` (unknownkeys_shared.go:115-130, `sort.Strings(out)` at :128) sorts. The arrays are consumed only as a membership list by `cpf_hasUnknownKeysFromArray`/`cpf_getUnknownKeysFromArray` (linear `===` scan), so **semantics are identical** — only the emitted literal's byte order changes. Rationale (shared.go:111-114 + port-status deviation #2): Go `map` iteration is randomized, so an unsorted literal would yield a different cache-module hash every binary run, breaking the Vite plugin's deterministic-output invariant. **Confirmed; correctly assessed as a non-bug.** The same `sort.Strings` also orders the sibling-named-prop Set (shared.go:328) and the merged-allowlist disjunction follows first-appearance order via `buildMergedProps` (union_flat_layout.go:148-156).

2. **Union arm re-architected onto a merged allowlist (port-only correctness fix).** mion's `UnionRunType` has **no** unknown-keys override (union.ts:24) → it uses the `CollectionRunType` default: OR-join (huk) / `;`-join (suk/uku/uke) of each member's compiled emit, where every object member's `InterfaceRunType` emit independently runs `callCheckUnknownProperties` against the *whole* value. The port replaces this with one for-in over `v` gated by the UNION of all object-member property names (`emitUnionUnknownKeysMerged`, union.go:56-109; `buildAllowlistGuard` :115-124). The port's own comment (unknownkeys_has.go:424-429) documents *why*: the per-member form "silently mis-reported hits because each member's own emit ran against the entire value regardless of which union arm matched" — and for strip it would emit "delete keys∉{a}; delete keys∉{b}", deleting BOTH members' declared keys (test comment unknownKeys.test.ts:204-208). The merged-allowlist is the deliberate, *more-correct* behaviour with a documented "loose" semantic (a key on ≥1 arm survives; unknownKeys.test.ts:265-273). Index-sig-bearing unions opt out entirely (union.go:62-69). This is the most significant deviation but is a fix, not a regression.

3. **`unknownKeysToUndefinedWire` (`ukuw`) is port-only decoder infrastructure.** No mion equivalent exists — it is tied to the port's flat-union wire format (the `[-1, mergedObject]` envelope is a ts-go encoding fiction). `ukuw` (to_undefined_wire.go) delegates to the base `uku` for every kind EXCEPT (a) **union**, where it emits the wire-aware strip — gate `if (Array.isArray(v) && v.length===2 && v[0]===-1)` then run the merged-allowlist over `v[1]` (union.go:81-108, `target = ctx.Vλl + "[1]"` at :82) — and (b) **Map/Set**, kept as an explicit noop (to_undefined_wire.go:88-93). **Existence + correctness:** the public `uku` operates on the raw user object and its union arm would corrupt a wire payload by treating the envelope array's index `0` (`-1`) / index `1` (the object) as object keys; `ukuw` instead peels the wrapper and strips inside `v[1]`, leaving indices `0`/`1` intact, so the downstream `restore` sees a clean merged object. The Map/Set noop is correct because at the `ukuw` stage `v` is the `JSON.parse`-output array (not yet a `Map`/`Set`), so an `instanceof` guard would fail-through and crash the downstream `restore` on `v.length` (to_undefined_wire.go:64-77); the safe encoder already stripped inner extras pre-wire. **`createJsonDecoder` confirmed to use `ukuw`, not `uku`, for strip:** `createRTFunctions.ts:341` `lookupRTFn<…>('createJsonDecoder', 'ukuw', id, …)` composed as `restore(ukuFn(JSON.parse(s)))` at :342. (Encoder mutate+strip still uses `uku` at :314, which is correct — there `v` is the raw object.) Exercised by `test/adapters/decoderSafeMode.test.ts` (4 `it()`, hand-crafted `[-1,{a:'hi',evil:'sneaky'}]` wire at :33).

4. **`KindIntersection` has no Emit arm but is unreachable.** All five `Supports` return true for `KindIntersection` (defensive), but no `Emit` switch has an intersection arm. This is harmless: the resolver **collapses intersections before emit** — `serialize.go:605-608` routes `TypeFlagsIntersection` to `collapseIntersection` (intersection_collapse.go:11-24), projecting object×object intersections into a single `KindObjectLiteral` (merged property list) and primitive×brand into the primitive. So a TS `A & B` reaches the unknown-keys emitter as a `KindObjectLiteral` and is handled by `emitObject*`. The dead `KindIntersection` Supports-true is not a behavioural gap — listed here so it is not mistaken for one.

5. **`uku` at a bare tuple node is an explicit noop.** to_undefined.go:254-267: the per-position concat would blindly recurse into every slot, breaking on circular optional self-ref tuple slots (unguarded `v[i].x` against `undefined`) and is semantically ill-defined (a tuple's "unknown key" would be a slot past the declared length, undetectable by per-position emit). huk/suk/uke keep the tuple recurse; only uku opts out. Safe because the safe encoder (`prepareForJsonSafe`) strips at encode time so the decode pipeline never relies on uku-at-tuple to converge. Documented inline.

6. **Single Go switch vs per-node classes; AOT runtime-throw vs emit-throw.** Architectural (overview §"Why the architectures differ"). The port also always emits a real factory for noop entries (never elides) so parent dep-calls don't hit `undefined.fn` — same trade-off as the JSON families (port-status deviation #1).

## 5. Gaps, mismatches & missed optimisations

1. **Map *value* unknown-key error path segment uses `failed:'mapValue'` — mion emits `failed:'mapVal'` (wire-contract divergence).** Severity: **High** (consumer-visible; mion's public `MapValuePathSegment` type pins `'mapVal'`). Evidence: mion `nodes/native/map.ts:79` `` return `{key:${safeKeyFn}(${custom.vλl}[0]),index:${parent.getIndexVarName(comp)},failed:'mapVal'}`; ``. Port: `internal/compiled/typefns/unknownkeys_errors.go:443` `ctx.SetChildPathLiteral("{key:" + safeKey + "(" + entryVar + "[0]),index:" + idxVar + ",failed:'mapValue'}")` (key side at :426 correctly uses `'mapKey'`). The JS type comment propagates the wrong value: `createRTFunctions.ts:47` documents `'mapKey' | 'mapValue'`. Impact: a consumer of `createUnknownKeyErrors<Map<…, ObjWithExtras>>()` branching on `seg.failed === 'mapVal'` (mion's documented value) never matches — silent mis-routing of Map-value unknown-key errors; breaks drop-in compatibility with mion-aware error renderers. **This is the SAME root divergence as getTypeErrors §5 #1** (same `'mapValue'` literal, both families). The unknownKeys test does NOT pin the segment shape — it asserts only `out[0].path` *contains* `'extra'` (unknownKeys.test.ts:355), so the green suite does not protect the contract. Fix: change `'mapValue'`→`'mapVal'` at unknownkeys_errors.go:443 (and jointly with typeerrors.go:934 + the createRTFunctions.ts:47 comment).

2. **Set item unknown-key error path segment is a bare numeric index — mion emits a `{key,index}` object; AND the explanatory Go comment is factually wrong.** Severity: **Med** (wire-contract divergence + misleading comment). Evidence: mion `nodes/native/set.ts:44-48` `getStaticPathLiteral` returns `` `{key:${safeKeyFn}(${custom.vλl}),index:${parent.getIndexVarName(comp)}}` `` (verified — set.ts DOES override it). Port: `internal/compiled/typefns/unknownkeys_errors.go:480` `ctx.SetChildPathLiteral(idxVar)` — a bare loop counter. The Go comment at errors.go:467-471 asserts *"mion's set.ts doesn't override getStaticPathLiteral like Map does, so the path falls back to the bare index"* — this is **incorrect**; set.ts:44-48 is exactly such an override. Impact: for a `Set<ObjWithExtras>` with an inner extra key, the port produces `path:[0, 'extra']` where mion produces `path:[{key:<safeItem>,index:0}, 'extra']` — consumers lose the offending element value and the segment shape differs. **SAME divergence as getTypeErrors §5 #2.** Test does not catch it (unknownKeys.test.ts:405 asserts only `path` contains `'extra'`). Fix: emit `{key:mapSafeKey(item),index:i}` in `emitSetUnknownKeyErrors` (register `mapSafeKeyContextItem`, mirror `emitMapUnknownKeyErrors`), correct the comment, and update jointly with typeerrors.go:1018.

3. **`hasUnknownKeys` runtime option renamed `checkNonJitProps` → `checkNonRTProps` (drop-in incompatibility).** Severity: **Med** (public-API wire shape for the options bag). Evidence: mion `constants.functions.ts:152` `runTimeOptions: {checkNonJitProps: {keyName: 'checkNonJitProps', …}}` and the emitted guard reads `opts.checkNonJitProps` (interface.ts:285 via `JitFunctions.hasUnknownKeys.runTimeOptions.checkNonJitProps.keyName`). Port emits `opts.checkNonRTProps` (unknownkeys_shared.go:202) and the JS type is `HasUnknownKeysOptions.checkNonRTProps` (createRTFunctions.ts:73-75). Impact: a caller migrating from mion who passes `hasUnknownKeys(v, {checkNonJitProps: true})` gets the *default* (false) behaviour from the port — the non-RT-children key set is never folded in — with no error. This is a deliberate "JIT → RT" terminology rename (consistent with `getJitChildren`→`getRTChildren`, the `huk` etc. naming), but is a silent behavioural incompatibility for the option. Fix: document the rename in `docs/ROADMAP.md` / a migration note, or accept both keys (`opts.checkNonRTProps ?? opts.checkNonJitProps`). (Lower-impact than #1/#2 because `checkNonRTProps` only matters for objects mixing serializable + function/static children, and defaults to off.)

4. **`nonSerializable` (WeakMap / typed arrays / …) is a silent noop — mion throws at RT-compile in all four emits.** Severity: **Low** (consistent with the port's JSON-family philosophy; non-serializable types carry no JSON-relevant unknown keys). Evidence: mion `nodes/native/nonSerializable.ts` `emitHasUnknownKeys`/`emitUnknownKeyErrors`/`emitStripUnknownKeys`/`emitUnknownKeysToUndefined` each `throw new Error('Jit compilation disabled for Non Serializable types.')`. Port: `Supports` includes `SubKindNonSerializable` (unknownkeys_has.go:60 etc.), but the `KindClass` inner Emit switch handles only `SubKindNone`/`SubKindMap`/`SubKindSet` (e.g. unknownkeys_has.go:131-137), so NonSerializable falls through to the default empty noop → Finalize. Impact: `createStripUnknownKeys<WeakMap<…>>()` returns an identity instead of a throwing factory (mion's `factoryThrows`); a value that should be rejected as non-serializable is silently passed through unmodified. Functionally harmless for the unknown-keys use case (no enumerable own data keys to strip), and aligns with the unified throw architecture's choice to noop these for non-JSON families — but it does diverge from mion's hard reject. Fix: either route NonSerializable through an alwaysThrow factory (parity with mion + the JSON families), or document the carve-out in `docs/UNSUPPORTED-KINDS.md` alongside the existing entries.

5. **Non-serializable array element (`symbol[]` / `Function[]`) does not throw — mion throws via `checkNonSkipTypes`.** Severity: **Low** (mirrors getTypeErrors §5 #5 / isType Gap #3; result is identical false/noop because such elements carry no keys). Evidence: mion `nodes/member/array.ts` calls `this.checkNonSkipTypes(comp)` first in all four emits, which `throw`s "Arrays can not have non serializable types" for symbol/function element kinds (array.ts:148-150). Port `emitArrayHasUnknownKeys` (unknownkeys_has.go:257-268) only short-circuits on `FamilyOf(child)==FamilyAtomic` (which covers `symbol`) → returns noop `return false`; function-element arrays recurse into the function child whose own emit is noop → also noop. Impact: `createHasUnknownKeys<symbol[]>()` returns an always-`false` validator rather than throwing at factory creation. No false results (symbol/function arrays genuinely have no unknown keys), so impact is purely the missing hard-reject signal. Fix: decide jointly with the array-element policy in isType/getTypeErrors (shared CodeNS→alwaysThrow vs noop decision), or document.
