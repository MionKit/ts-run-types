# 05 ‚Äî JSON serialization: mion vs ts-go-run-types port audit

> Date: 2026-05-29 ¬∑ Method: static source comparison (Go binary + Vitest suites NOT executed here; test registrations counted, marked "counted, not executed"). Per `docs/port-status.md` the central run reported: `@mionjs/ts-go-run-types` 867 pass / 0 fail.

## 1. Verdict

‚öÝÔ∏è **Ported with gaps ‚Äî one HIGH wire-shape divergence at unions.** The per-kind encode (`pj`), decode (`rj`) and single-pass stringify (`sj`) atomic/collection/member arms are faithful ports of mion's `emitPrepareForJson` / `emitRestoreFromJson` (29 / 32 nodes) and `createStringifyCompiler` ‚Äî bigint‚Üîstring, regexp split/rebuild, date `new Date`, Map/Set per-entry recursion, tuple null-padding, object optional-first stringify sort, symbol-keyed-index-sig skip, and the non-serializable throw-factory model all match. The two NEW clone families `pjs`/`pjsp` (no mion equivalent) deep-clone correctly without mutating the input and strip (`pjs`) vs `...v`-preserve (`pjsp`) extras, reusing the per-kind logic. The public `createJsonEncoder`/`createJsonDecoder` option‚Üítag matrix is correct and matches the contract exactly. **However the union encoding was rewritten into a "flat-union" layout (`union_flat*.go`) that is NOT mion-compatible on the wire**: object union members are *merged* into a single `[-1, mergedObject]` envelope (mion keeps each object as its own `[memberIndex,value]` arm), and the per-member `needsTupleEncoding` decision was replaced by an *all-or-nothing* `AtomicNeedsTuple` flag driven by a type-recursive `isJsonCompatible` predicate ‚Äî so `string|number|boolean|Date` wraps **all four** arms where mion wraps only the `Date` arm (¬ß5 #1, **High**). port-status.md ¬ß5 / "Queued optimisations" still describe a `unionMemberNeedsTuple` single-source-of-truth that **no longer exists in the code** (¬ß5 #2). Three carry-over by-design deviations: symbol is unsupported (mion encodes `'Symbol:'+desc`), non-serializable throws at first-call not at compile, and the noop-factory is always emitted. Round-trip parity holds for every shipped sample (the suite drives 130 cases √ó 3 encode paths); the divergences are wire-bytes/`isNoop`-flag, not round-trip correctness, and the mion code-introspection assertions that would catch them have no ts-go counterpart (¬ß6).

## 4. Intentional deviations (by design)

1. **Symbol is unsupported in `pj`/`rj`/`sj` (bare-symbol kind).** mion encodes `'Symbol:'+desc` and decodes `Symbol(v.substring(7))` (symbol.ts:25,28); the port returns `CodeNS` (json_prepare.go:195-200, json_restore.go:161-164, json_stringify.go:188-190). Rationale (`docs/UNSUPPORTED-KINDS.md` FAQ): `Symbol("x") !== Symbol("x")`, so mion's encoding is lossy by construction (round-trip never deep-equals). The port treats a bare `symbol` value position as a non-serializable leaf ‚Üí property-drop or alwaysThrow. Note the *literal-symbol* arms are NOT aligned with this decision ‚Äî ¬ß5 #6.
2. **Non-serializable kinds throw at first-call, not at RT-compile.** mion throws synchronously from the emit method (never/promise/nonSerializable/function); the port emits a runtime-throwing factory (`CodeNS` ‚Üí alwaysThrow). Same observable contract (`factoryThrows` adapter asserts the throw). Carried over from isType/typeErrors; documented in `docs/port-status.md` Deviation #4 and `docs/UNSUPPORTED-KINDS.md`. Also: a non-serializable *array element* / *tuple slot* / *union member* surfaces alwaysThrow rather than mion's compile-throw or `v[i]=undefined` restore.
3. **`isNoop` factory always emitted (mion elides the inner factory when noop).** `Finalize` returns `isNoop: true` for `return v` bodies but the renderer still emits an identity factory, so parent `<childHash>.fn(v[i])` dep-calls hit a real fn (`docs/port-status.md` Deviation #1; `serializationNoop.test.ts` header). Cost ~30 bytes/noop. The `isNoop` flag is still correctly set ‚Äî but see ¬ß5 #4 for a union-specific `isNoop` divergence.
4. **`pjs`/`pjsp` clone families are a NEW capability with no mion equivalent.** mion ships only the mutate path (`prepareForJson`) and the single-pass `stringifyJson`. The port adds a *non-mutating clone* strategy: `pjs` builds a fresh value containing only declared keys (strips extras), `pjsp` builds `{...v, declared:<transformed>}` (preserves extras). Assessment in ¬ß5 (correct; reuse confirmed).
5. **Single Go switch vs mion's dual model** (per-node methods for pj/rj, switch-compiler for sj). Architectural; see overview "Why the architectures differ".

## 5. Gaps, mismatches & missed optimisations

1. **Union wire format is NOT mion-compatible: object members merged into `[-1, mergedObject]` + all-or-nothing atomic wrap.** Severity: **High** (consumer-visible wire bytes; breaks drop-in interop with a mion peer and with mion's own stringify code-introspection contract). Evidence:
   - mion encodes **each object member as its own `[memberIndex, value]` arm** (union.ts:114-152, index via `getUnionItemIndex` :41-46) and wraps a member **iff that member's own pj OR rj is non-empty** (`needsTupleEncoding = !!toJit.code || !!fromJit.code`, union.ts:127; mirrored in stringifyJson.ts:299). Decode runtime-sniffs the wrapper: `if (v?.length === 2 && Array.isArray(v) && typeof v[0] === 'number') {const dec = v[0]; v = v[1]; ‚Ä¶}` (union.ts:181-187).
   - The port MERGES every object/class(SubKindNone) member into a single `[-1, mergedObject]` envelope with per-property dispatch (`emitUnionPrepareForJsonFlat`, union_flat.go:108-138 `body += v + " = [-1, " + v + "]"`; stringify `envelope := "'[-1,' + " + objExpr + " + ']'"` union_flat.go:478). Atomic members wrap under an **all-or-nothing** `AtomicNeedsTuple` flag (union_flat_layout.go:113-124) driven by the type-recursive `isJsonCompatible` (json_compat.go:40), NOT per-member emit-emptiness. Decode unconditionally unwraps (`const dec0 = v[0]; v = v[1];`, union_flat.go:281) with **no runtime shape-sniff**.
   - **Concrete byte divergence (the mion test that would fail):** for `string | number | boolean | Date`, mion `stringifySpec/09StringifyUnions.spec.ts:54-57` asserts `stringifyCode.not.toContain('[0,')`, `not.toContain('[1,')`, `not.toContain('[2,')`, **`toContain('[3,')`** ‚Äî only the Date arm wraps. The port: Date is non-`isJsonCompatible` ‚Üí `AtomicNeedsTuple=true` ‚Üí it emits `'[0,'`, `'[1,'`, `'[2,'`, `'[3,'` for **all four** arms (union_flat.go:366-367 `emitted = "'[" + strconv.Itoa(m.OriginalIndex) + ",' + " + childRT.Code + " + ']'"`). The port would FAIL `not.toContain('[0,')`. Same for `with_discriminator` (:73-76: only `[1,`/`[3,` should wrap).
   - Impact: a mion encoder feeding a port decoder (or vice-versa) for any object-bearing or mixed union sees a different envelope (`[-1,{merged}]` vs `[i,{obj}]`) and a different set of wrapped atomic indices ‚Üí cross-impl round-trips break; payload bytes differ; the merged envelope also conflates which member matched (decoder restores by per-property dispatch, not by member index). Self-consistent round-trips (port‚Üîport) pass ‚Äî every shipped sample is green.
   - Fix: either (a) restore mion's per-member `[memberIndex,value]` emit with the per-member `needsTupleEncoding` gate and the decode shape-sniff, or (b) explicitly document the flat layout as an intentional wire-incompatible optimisation in `docs/ROADMAP.md` + `docs/UNSUPPORTED-KINDS.md` and add a ts-go code-introspection test asserting the `[-1,` / per-arm indices so the contract is pinned.

2. **port-status.md still claims a `unionMemberNeedsTuple` single-source-of-truth that no longer exists.** Severity: **Med** (stale/misleading docs ‚Äî directly contradicts the shipped code and this audit's task framing). Evidence: `docs/port-status.md` Deviation #5 ("RESOLVED") and "Queued mion optimisations ‚Üí Union per-member skip-encode ‚Ä¶ ‚Äî DONE" describe a shared `unionMemberNeedsTuple(member, ctx)` helper "the single source of truth used by all three union emitters" and a memoised `peekMemberIsNoop` on `Walker.peekedNoops`. Both symbols are **absent**: `grep -rn "unionMemberNeedsTuple\|peekMemberIsNoop" internal/` ‚Üí 0 hits. The actual basis is `isJsonCompatible` (json_compat.go) + `FlatLayout.AtomicNeedsTuple` / `FlatMergedProp.NeedsSubWrap` (union_flat_layout.go). The two predicates differ semantically: mion's `needsTupleEncoding` is "this member's pj OR rj emits non-empty code"; the port's `isJsonCompatible` is "this type round-trips natively with no transform". They disagree for `undefined` / `void` members (see ¬ß5 #3) and for the all-or-nothing vs per-member granularity (¬ß5 #1). Impact: anyone reading port-status believes union parity is achieved per-member; the real behaviour is flat-merge + all-or-nothing. Fix: rewrite port-status Deviation #5 + the "Queued optimisations" entry to describe the flat layout and `isJsonCompatible`, and cross-link this audit.

3. **`isJsonCompatible` over-wraps `undefined`/`void` union members vs mion's `needsTupleEncoding`.** Severity: **Low** (extra wrap on a niche union; round-trip still correct). Evidence: json_compat.go:83-94 returns `false` for `KindUndefined` and `KindVoid`, so a union containing `undefined`/`void` forces `AtomicNeedsTuple=true` and wraps every atomic arm. mion only wraps a member when its pj/rj is non-empty: `undefined.ts` has no `emitPrepareForJson` (encode noop) and `emitRestoreFromJson` returns `undefined` (non-empty) ‚Üí mion `needsTupleEncoding` is true for `undefined` too ‚Äî so this particular case *coincidentally* agrees, but the *granularity* still differs (mion wraps only the `undefined` arm; the port wraps all arms because of the all-or-nothing flag). Impact: subset of ¬ß5 #1; extra `[idx,` markers on JSON-natural co-members. Fix: subsumed by ¬ß5 #1.

4. **Union `restoreFromJson` `isNoop` diverges from mion's `00JsonOnly.spec.ts` for all-noop unions.** Severity: **Low** (flag-only; no wire/round-trip impact). Evidence: mion `00JsonOnly.spec.ts:45-55` asserts a `number | string` union has `isNoop === false` on **both** prepareForJson AND restoreFromJson ("encoding/decoding is always required"). The port's `rj` for `number|string`: `AtomicNeedsTuple=false` ‚Üí `emitUnionRestoreFromJsonFlat` returns empty (union_flat.go:219-223) ‚Üí `Finalize` collapses to `return v` + `isNoop=true`. The port's own `serializationNoop.test.ts:94-95` asserts `pjEntry(noopId)?.isNoop === false` but **`rjEntry(noopId)?.isNoop === true`** ‚Äî encoding the divergence as the expected result (the test comment at :77-90 acknowledges it). Impact: a consumer keying off `rj.isNoop` to short-circuit dispatch behaves differently than under mion for noop unions; harmless because the decode genuinely is identity. Fix: accept (document) or emit a dead shape-gate to keep `isNoop:false` parity (not worth the bytes).

5. **port-status stringifyJson notes contain two inaccuracies (object order + symbol).** Severity: **Low** (doc/cross-reference correctness). Evidence: (a) `json_stringify.go:22-24` and `docs/port-status.md` ("mion sorts optional members first; we keep declaration order") claim the port keeps declaration order ‚Äî but `emitObjectStringifyJson` performs the SAME optional-first stable sort as mion (json_stringify.go:412-419 vs interface.ts:205-213). The "deviation" doesn't exist for `sj`; the note is wrong. (b) The port-status `stringifyJson per-kind coverage` table row `symbol | JSON.stringify('Symbol:' + (v.description \|\| ''))` describes mion, not the port ‚Äî the port emits `CodeNS` (json_stringify.go:188-190). Impact: misleading lead for future maintainers. Fix: correct both rows in port-status.

6. **Literal-symbol arms still emit the `'Symbol:'` encoding while bare-symbol is unsupported ‚Äî internal inconsistency.** Severity: **Low** (narrow surface: a `symbol` *literal* type, e.g. `const s: unique symbol`; rare). Evidence: bare `KindSymbol` ‚Üí CodeNS in all three families (¬ß4 #1), but the literal dispatch still emits the encoding: `emitLiteralPrepareForJson` `v = 'Symbol:' + (v.description || '')` (json_prepare.go:335), `emitLiteralRestoreFromJson` `v = Symbol(v.substring(7))` (json_restore.go:273), `emitLiteralStringifyJson` `JSON.stringify('Symbol:'+(v.description||''))` (json_stringify.go:308); `isJsonCompatible` also flags symbol-literals non-compatible (json_compat.go:71-74). Impact: a `symbol`-literal value gets the lossy round-trip the bare-symbol decision rejected (decode produces a *fresh* symbol, never deep-equal to the original) ‚Äî inconsistent semantics within the same family. Fix: either return CodeNS for symbol-literals too (consistent with ¬ß4 #1) or re-enable bare-symbol (consistent the other way); pick one rule.

7. **`createStringifyJson.test.ts` does not exercise the `sj` (direct) family despite its name.** Severity: **Low** (test-clarity / coverage-attribution gap, not a code defect). Evidence: all 42 `createJsonEncoder<‚Ä¶>()` calls in `test/createStringifyJson.test.ts` use the **default `clone` strategy** (34 bare) or `mutate` (5) ‚Äî **0** use `{strategy:'direct'}` (counted, not executed). So the "no-mutation invariant" (:270-306) and "extras stripped in the EMIT" (:308-343) describes are actually validating `pjs`/`pj`, not `sj`. The `sj` family IS covered ‚Äî via `serialization-suite.ts` `safeDirectEncoder` (130 thunks ‚Üí `assertSafeDirectRoundTrip`) + `encoderModes.test.ts:62-71` (1 direct case) ‚Äî but the dedicated file's name implies a direct coverage it doesn't provide. Impact: maintainers may over-trust `sj` coverage from this file. Fix: add `{strategy:'direct'}` variants to the per-kind smoke tests (or rename the file).

8. **`createJsonEncoder` mislabeled extras test passes for the wrong reason.** Severity: **Low** (subset of #7). Evidence: `createStringifyJson.test.ts:308-343` asserts a bigint/symbol/function extra is dropped with "no throw" and credits "stringifyJson walks declared members only" ‚Äî but the encoder is the default `clone` (`pjs`), which strips extras by *building a fresh object with only declared keys* (json_prepare_safe.go:346-365), a different mechanism. Both strip, so the assertions hold, but the cited rationale is wrong. Fix: as #7.
