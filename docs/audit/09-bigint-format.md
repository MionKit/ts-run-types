# 09 — bigint type-format: mion vs ts-go-run-types port audit

> Date: 2026-05-29 · Method: static source comparison (Go binary + Vitest suites NOT executed here; test registrations counted, marked "counted, not executed"). All counts below are grep/read-derived, not from a test run.

## 1. Verdict

✅ **Fully ported** (one Med + one Low test-coverage gap, no shipped-code defect). The bigint type-format carries BOTH validation AND serialization, and **both are ported**. Validation (`isType` AND-chain + `typeErrors` per-predicate pushes for max/min/lt/gt/multipleOf) is a faithful port of mion's `emitIsType`/`emitIsTypeErrors`, emitting full-precision `…n` bigint literals via `*big.Int` (never float64) — `internal/compiled/typefns/formats/numeric/bigintformat.go:44-104`. **The binary serialization emitters ARE ported and wired**: the bigint format's `EmitToBinary`/`EmitFromBinary` (the 8-byte `setBigInt64`/`setBigUint64` packing) are spliced into the host's `KindBigInt` binary arm via `binaryToOverride`/`binaryFromOverride` (`binary_to.go:200-210`, `binary_from.go:126-134`) — ✅ ported, NOT a fallback. The "fall back to the plain bigint encoder" branch (string/`BigInt`) is **mion's own design** for out-of-64-bit-range / partial-bound brands (mion `bigIntFormat.runtype.ts:135-136,151-152` `{code: undefined}`), faithfully reproduced as the empty-override path. **JSON serialization carries NO bigint-format override on EITHER side** — mion's `BigIntRunTypeFormat` defines no `emitPrepareForJson`/`emitRestoreFromJson`/`stringifyJson` (method surface verified: only isType/typeErrors/format-noop/toBinary/fromBinary/_mock/validateParams), so branded bigints inherit the plain bigint JSON path (`v.toString()` → `BigInt(v)`); the port's JSON emitters have no bigint format hook either — **this is parity, not a gap**. `validateParams` (FMT002) and mocking are both ported. Gaps are test-only: 5 `it.todo` in `formatMockType.test.ts` for single-bound bigint mocks (§5 #1, Med), and the bigint-format `format.val` JSON-roundtrip is asserted via binary not directly (§5 #2, Low).

## 4. Intentional deviations (by design)

1. **JSON serialization of branded bigints uses the plain bigint encoder — and this is CORRECT parity, not a fallback gap.** mion's `BigIntRunTypeFormat` overrides only `emitToBinary`/`emitFromBinary`, never any JSON emit method (verified: `grep emitPrepareForJson|emitRestoreFromJson|emitStringifyJson type-formats/src/` → 0 hits; method surface is isType/typeErrors/format-noop/toBinary/fromBinary/_mock/validateParams only). So in mion a `FormatBigInt64` already serializes to JSON as a decimal string via the base bigint node (`nodes/atomic/bigInt.ts`). The port mirrors this exactly: `json_prepare.go`/`json_restore.go`/`json_stringify.go` have no `FormatAnnotation`/`formats.Lookup` hook in their bigint arms, and `registry.go` defines no JSON-emitter capability interface. Branded bigints fall through to `v.toString()` / `v = BigInt(v)`. **Answers the critical task question: bigint-format JSON serialization emitters do not exist in mion, so "missing" is the correct port.**
2. **Binary string fallback for out-of-64-bit / partial-bound brands.** mion returns `{code: undefined}` from `emitToBinary`/`emitFromBinary` when the params don't fit 64-bit or lack both bounds (`bigIntFormat.runtype.ts:135-136,151-152`), deferring to the base string serializer. The port returns `""` from `EmitToBinary`/`EmitFromBinary` (bigintformat.go:120,136), which the host treats as "keep base arm" (binary_to.go:191-193, binary_from.go:118-120). This is the SAME behaviour, deliberately — `FormatBigPositive` (only `min`) and a `{min:0n;max:10^23}` brand both serialize as a decimal string. The Go test `TestBigIntBinary_RangeSelection` pins all four fallback rows (numeric_test.go:71-74).
3. **Param validation is an AOT build-time diagnostic (FMT002), not a runtime throw.** mion throws from `validateParams` during JIT compile. The port runs `ValidateParams` at scan/emit time from the isType walk and emits a `CodeFMTInvalidParams` (FMT002) diagnostic per message (istype.go:182-184). Identical to items 07-08; the validation runs once (isType is rendered for every format-bearing type).
4. **`val` rendered inline, not via the `cpf_formatErr` pure fn.** mion pushes format errors through `cpf_formatErr`; the port emits the `er.push({…format:{…}})` literal inline (shared.go:43-50) because the consumer's program does not import the type-formats pure-fn graph. Same wire shape (`{name, formatPath, val}`), documented at shared.go:34-36. Architectural, matches the string/number format ports.
5. **Brand names on the JS aliases.** The port's `FormatBigInt64` etc. carry an explicit `BrandName` (`'bigInt64'`, bigintFormats.ts:43) where mion's `defaultBigNumberFormats.ts` leaves them unbranded. Cosmetic — brand name does not affect validation/serialization params and is stripped from the structural id alongside `mockSamples`/`message` (formats.go:400-403). No behavioural impact.

## 5. Gaps, mismatches & missed optimisations

1. **`formatMockType.test.ts` BIGINT block has 5 `it.todo` for single-bound mocks (tests left behind).** Severity: **Med** (no shipped-code defect — `mockBigIntFormat.ts` handles all five — but the single-bound mock paths are unverified). Evidence: `packages/ts-go-run-types/test/adapters/formatMockType.test.ts:110-114` — `it.todo('FormatBigInt<{max: 100n}> …')`, `{min: 0n}`, `{lt: 10n}`, `{gt: 0n}`, `{multipleOf: 5n}` are all `it.todo`. Only `bigint_combined`, `bigint_int64`, `bigint_uint64` are live (`:115-117`). The corresponding suite cases `bigint_max`/`bigint_min`/`bigint_lt`/`bigint_gt`/`bigint_multipleOf` in `format-validation-suite.ts:525-568` **carry no `mockType:` thunk** (only `bigint_combined`/`int64`/`uint64` have one at :573,585,599), so the todos can't be filled without adding the thunks. Impact: the `_mock` exclusive-bound adjustment (`gt+1n`/`lt-1n`, mockBigIntFormat.ts:33-34) and the single-`multipleOf` snap are unexercised; mion's `bigIntFormat.runtype.spec.ts:35-178` DOES mock all five (`mock bigint max/min/lt/gt/multipleOf`). Fix: add `mockType:` thunks to the five single-bound suite cases and replace the 5 `it.todo` with `assertMockType(BIGINT.bigint_<x>)`.

2. **Branded-bigint JSON `format.val` is not directly round-trip-asserted — only the binary half is.** Severity: **Low** (JSON path is plain-bigint, well-covered by core bigint tests; the format brand changes nothing on the JSON wire). Evidence: `format-serialization-suite.ts:309-312` comment on `bigint_int64`: "the round-trip is exercised via the binary half above, but JSON must still round-trip" — the JSON encoders/decoders are wired (`unsafeEncoder`/`safeDecoder` etc., :298-304) and the round-trip assertion runs through `formatSerializationRoundTrip.test.ts`, but there is no bigint-format-specific JSON assertion distinct from plain bigint. Since §4 #1 establishes the JSON path is intentionally identical to plain bigint, this is informational. Impact: minimal — a regression that broke ONLY branded-bigint JSON (impossible given no format hook exists) would not be caught by a dedicated test, but the generic round-trip covers it. Fix: none required; optionally document that branded-bigint JSON === plain-bigint JSON in `format-serialization-suite.ts`.

3. **No test case for `FormatBigPositiveInt` / `FormatBigNegativeInt`.** Severity: **Low** (these are `{min:0n;multipleOf:1n}` / `{max:0n;multipleOf:1n}` — structurally a combination already covered by `bigint_min`+`bigint_multipleOf` / `bigint_combined`). Evidence: `format-validation-suite.ts:524-606` defines `bigint_max/min/lt/gt/multipleOf/combined/int64/uint64` but no `bigint_positive_int`/`bigint_negative_int`; mion `defaultBigNumberFormats.spec.ts:73-95,99-121` tests `FormatBigPositiveInteger`/`FormatBigNegativeInteger` (isType+errors+mock, 6 cases). Impact: the two integer-default aliases ship in `bigintFormats.ts:41-42` with no direct coverage; their behaviour is the AND of two already-tested predicates. Fix: optionally add `bigint_positive_int`/`bigint_negative_int` suite cases for completeness parity with mion's 41.
