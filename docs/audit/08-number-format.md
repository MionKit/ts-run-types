## 1. Verdict

✅ **Fully ported** (validation + JSON serialization + binary serialization + param-validation + mock), with one test-coverage gap. The single Go emitter `internal/compiled/typefns/formats/numeric/numberformat.go` is a faithful port of mion's `NumberRunTypeFormat` (`packages/type-formats/src/number/numberFormat.runtype.ts`): `EmitIsTypeCheck` (:49-81) reproduces `emitIsType`'s predicate set and order (integer/float, max, min, lt, gt, multipleOf) verbatim; `EmitTypeErrorsCheck` (:88-122) reproduces `emitIsTypeErrors` one-`if`-per-predicate with the same `val` tags (`true` for integer/float, the bound for ranges). **The critical question — are number-format SERIALIZATION emitters ported, or do branded numbers fall back to the plain `number` encoder? — resolves to: BINARY is ✅ ported (custom width ladder), JSON is ✅ ported as identity (matching mion, which has NO custom JSON encoder for numbers).** `EmitToBinary`/`EmitFromBinary` (:128-188) port `emitToBinary`/`emitFromBinary` exactly — `FormatAnnotation` min/max drives the narrowest of `setUint8/16/32`/`setInt8/16/32` (1/2/4 bytes), with float / unbounded-integer / >int32 falling back to the base float64 arm; these are spliced into the `KindNumber` arm of `binary_to.go` (:185-194) and `binary_from.go` (:109-121) only when a `FormatAnnotation` is present — so `FormatInt8` packs 1 byte while a plain `number` keeps 8. `ValidateParams` (:242-284) ports `validateParams` AOT (FMT002), including mion's `filter(Boolean)` 0-bound quirk. `mockNumberFormat.ts` ports `_mock`. The one gap is **test-coverage, not behaviour: 6 `it.todo` in the NUMBER block of `formatMockType.test.ts`** (max/min/lt/gt/float/multipleOf mocks unverified — §5 #1) plus a thin number serialization round-trip set (§5 #2). isType / getTypeErrors / binary round-trip (with exact byte-width assertions) NUMBER blocks have **0 todo**.

## 4. Intentional deviations (by design)

1. **`validateParams` throw → AOT FMT002 diagnostic.** mion throws inside `validateParams` at JIT-compile time (numberFormat.runtype.ts:234-283, reached via `createIsTypeFn` — the spec asserts `rejects.toThrow()`, numberFormat.runtype.spec.ts:347-384). The port runs the same checks AOT in Go (`numberFormatEmitter.ValidateParams`, numberformat.go:242-284) and emits a build-time `CodeFMTInvalidParams`/`FMT002` (SeverityError) diagnostic per violation from the isType walk (istype.go:182-184). Same invariants enforced, surfaced earlier (build time vs first-call). This is the documented format-param model (registry.go:89-97); shared with the string/bigint formats (items 07/09). The mion-faithful `filter(Boolean)` quirk — a `0` bound is falsy and so escapes the mutual-exclusivity/range checks — is deliberately replicated via `numberTruthy` (numberformat.go:255-260,288-293) and pinned by `numeric_test.go:165-167` (`{min:0, gt:0}` → no error).
2. **Class-method emitter → single Go switch + capability interfaces.** mion's `NumberRunTypeFormat` owns per-fn methods; the port folds these into one registered `numberFormatEmitter` whose optional `BinaryEncoder`/`BinaryDecoder`/`ParamValidator` interfaces are type-asserted by each host fn (binary_to.go:149, binary_from.go:77, istype.go:182). Architectural (overview §"Why the architectures differ"), not behavioural.
3. **JSON serialization is identity for numbers in BOTH systems — not a port gap.** mion's `BaseRunTypeFormat` explicitly short-circuits `prepareForJson`/`restoreFromJson`/`stringifyJson` to `{code:'', type:'E'}` ("formats don't need special handling for JSON", baseRunTypeFormat.ts:183-190) — so a branded number serializes through the base number JSON arm (passthrough; `stringifyJson` root → `String(v)`, stringifyJson.ts:96-99). The port mirrors this by NOT implementing a JSON override interface for numbers (json_prepare.go / json_restore.go / json_stringify.go treat `KindNumber` as a plain leaf — no `binaryToOverride`-style hook). So "branded numbers fall back to the plain number JSON encoder" is the *correct, intended* behaviour for JSON, identical to mion — distinct from binary, where a real per-format override exists.

## 5. Gaps, mismatches & missed optimisations

1. **6 `it.todo` in the NUMBER block of `formatMockType.test.ts` (mock coverage gap).** Severity: **Med** (no shipped-code defect — `mockNumberFormat.ts` implements every branch — but the in-range sampling for the bound-only / float / multipleOf formats is unverified). Evidence: `test/adapters/formatMockType.test.ts:87-93` — `it.todo('FormatNumber<{max: 100}> …')`, `min`, `lt`, `gt`, `FormatFloat`, `FormatNumber<{multipleOf: 5}>` (6 todos); only 4 NUMBER mock cases are live (`number_integer` :91, combined :94, `number_int8` :96, `number_uint8` :97) + the coverage guard :99. Root cause: the corresponding suite cases lack a `mockType` thunk — only `number_integer`, `number_combined`, `number_int8`, `number_uint8` define `mockType:` in `test/suites/format-validation-suite.ts` (:459,491,504,516); `number_max/min/lt/gt/float/multipleOf` (:421-486) omit it. Impact: mion verifies all of these (`mock max/min/lt/gt/float/multipleOf`, numberFormat.runtype.spec.ts:41,78,115,152,218,279, each asserting 20 in-range samples), so the port's `_mock` port for exclusive-bound epsilon handling (mockNumberFormat.ts:26-33) and `multipleOf` snap-down (:45-48) is currently untested. Fix: add `mockType:` thunks to those 6 suite cases and replace the 6 `it.todo` with `assertMockType(...)`.

2. **Number-format SERIALIZATION round-trip suite is thinner than mion's binary spec.** Severity: **Low** (the headline widths ARE covered — int8/16/32, uint8/16/32, integer→8B, float→8B, ranged→uint16 — 9 live cases each in `formatSerializationRoundTrip.test.ts` and `formatBinaryRoundTrip.test.ts`, with exact `getBinaryByteSizes` byte-length assertions enforced at `util/serializationAsserts.ts:215-216`). Evidence: `test/suites/format-serialization-suite.ts:159-289` (NUMBER_FORMAT) covers the 6 width-named defaults + 3 derived, but does NOT add explicit cases for `FormatPositive`/`FormatNegative`/`FormatPositiveInt`/`FormatNegativeInt` (mion's `defaultNumberBinary.spec.ts:64-98` has all four, each pinning 8 bytes). Impact: the "min≥0 but unbounded max → float64" / "max≤0 but unbounded min → float64" fallback corners are covered in Go (`numeric_test.go:31` `{integer:true}`→"" and the `integerType` defaults numberformat.go:209-217) but not in the JS round-trip suite. mion's 12 binary `it`s vs the port's 9 JS round-trip cases + 10 Go subtests (`TestNumberBinary_IntegerWidthLadder`, numeric_test.go:18-56). Fix: optionally add `number_positive_8bytes` / `number_negativeint_8bytes` round-trip cases for full parity with `defaultNumberBinary.spec.ts`.

3. **No standalone mock-distribution / multipleOf-`Math.abs` assertion.** Severity: **Low**. mion's `mock multipleOf` asserts `Math.abs(item % 5) === 0` (numberFormat.runtype.spec.ts:284) — the port's `assertMockType` for the live combined case asserts `item % 5 === 0` via `getSamples`/isType round-trip, but the dedicated multipleOf mock case is a todo (folds into §5 #1). Impact: negligible — the combined case (:94) exercises `multipleOf:5` mock snap-down indirectly. Fix: covered by resolving §5 #1.
