# 06 — binary serialization: mion vs ts-go-run-types port audit

> Date: 2026-05-29 · Method: static source comparison (Go binary + Vitest suites NOT executed here; test registrations counted, marked "counted, not executed"). Central run reported: binary round-trip + format-binary suites passed, 0 failures.

## 1. Verdict

⚠️ **Ported with gaps (all by-design or low-severity).** Every mion `emitToBinary` / `emitFromBinary` switch arm that maps to a serializable reflection kind has an equivalent arm in the two Go switches `internal/compiled/typefns/{binary_to.go,binary_from.go}` (+ `union_flat_binary.go` for `KindUnion`), the args contract (`{vλl, sεr}` / `{vλl, dεs}`, return `Ser`/`ret`) is a faithful port, and the DataView wire machinery (`src/runtypes/dataView.ts`) is a near-verbatim re-implementation of mion's `packages/core/src/binary/dataView.ts` — same byte layout (LE float64, `[uint32 len, utf8]` strings, `[uint32 type, value]` enums, 1-bit-per-optional bitmaps), same string-bytes cache + adaptive sizing, plus a hardening (`RangeError` on `encodeInto` truncation where mion silently truncates). The optional-member bitmap, required-before-optional ordering, rest-length prefix, Map/Set `[uint32 size, entries]`, index-signature `[uint32 count, (key,value)*]`, circular self-recursion, and the throw-on-non-serializable (`never`/`Promise`/`WeakMap`/symbol/function-at-root → `CodeNS`→alwaysThrow) all match. **The deliberate divergences are five, all reconciled:** (a) `allParamsOptional` + `paramsSlice` router conveniences are NOT ported — `isFnParams` is hardcoded `false` in `binary_to.go:618` and absent entirely from `emitTupleFromBinary` (§4 #1, ROADMAP-blessed); (b) the **object-union wire shape differs** — mion encodes each object arm as a standalone object keyed by its discriminator index, the port MERGES all object arms into one `0xFF`-sentinel flat envelope with a merged-prop bitmap (§4 #2, the same flat-union strategy as the JSON family); (c) `symbol` at root → port emits `CodeNS`/alwaysThrow, mion serializes `v.description` (§4 #3, family-wide per `docs/UNSUPPORTED-KINDS.md`); (d) user-class instances decode to **plain objects** — the port's class arm never emits mion's `getDeserializeFn`/`getSerializeClass` reconstruction (§5 #1, Low, matches the `TypeClass.classType` won't-capture decision); (e) `noLiterals` not plumbed, so a literal always writes zero bytes (§5 #2, Low). Number/bigint binary **format packing** (int8/16/32, BigInt64/Uint64) is wired through the `binaryToOverride`/`binaryFromOverride` hook (§3) and audited as items 08/09. No round-trip-breaking defect was found; the binary round-trip suite inherits the full 129-case JSON serialization suite, so coverage parity is high.

### Bitmap / optional handling (Focus #2)

mion sets the optional bitmap **inside the property/tupleMember arm** (`setBitMask(bitMIndexVar, bitIndex)` after the value, toBinary.ts:189-192,209-212) using `optionalIndex`/`_bitmapVar`/`_isFnParam` fields stashed on the child by the parent (to:243-256,329-342). The port **moves the bitmap orchestration entirely to the parent** (`emitObjectToBinary` :528-563, `emitTupleToBinary` :646-679): the parent allocates the bitmap (`emitOptionalBitmapInit`, zeroes `ceil(N/8)` bytes), then for each optional child emits `if (accessor !== undefined) { <value>; setBitMask(bmI, i&7) }` and bumps `bmI` every 8 (`modIndex%8===0 && modIndex<len`). Decode reads the bitmap then gates each optional with `getUint8(bmI + i/8) & (1<<(i&7))` (binary_from.go:455-462, 526-529). **Same wire bytes, same bit positions, same byte-bump cadence** as mion — only the codegen locus differs (parent-driven vs child-stashed-fields). The mion encode bumps the bitmap index `every 8 props` (to:248-252) using `modIndex % 8 === 0` for objects (no `< length` guard) but `modIndex%8===0 && modIndex<optional.length` for tuples (to:337); the port uses the `&& modIndex<len` guard for BOTH object (binary_to.go:556) and tuple (:674) — harmless (the trailing bump is dead since no 9th prop follows), and decode reads `bmI + i/8` absolutely so it is robust to the bump either way.

`isFnParams` flag: **present and hardcoded `false`** in `emitTupleToBinary` (binary_to.go:618, with a multi-line rationale comment :608-617). `emitTupleFromBinary` (binary_from.go:469-544) has **no `isFnParams` variable at all** — the decode side simply never models all-params-optional. Confirms Focus #3.

## 4. Intentional deviations (by design)

1. **`allParamsOptional` + `paramsSlice` router conveniences NOT ported.** `docs/ROADMAP.md` → "Binary serialization — function-params router conveniences" documents this explicitly. mion's `13BinaryAllParamsOptional.spec.ts` (11 `it()`) + the `slice function params` test (`06BinaryFunctions.spec.ts:170`) exercise: (a) every top-level function-param tuple slot treated as optional at the wire level (partial-payload RPC, e.g. `['hello', undefined, true]` for `(a,b,c)`), driven by `rt.createRTParamsFunction(toBinary)` / `createSerializationParamsFn(rt, sliceStart)` in `binaryHelpers.ts`; (b) `paramsSlice` skipping the leading N params (router context-arg stripping). The port: `isFnParams := false` hardcoded (binary_to.go:618); `emitTupleFromBinary` has no flag; no `SubKindParams` on the protocol; no `createRTParamsFunction`/`sliceStart` on the public API. Rationale (ROADMAP): every other RT family consumes `Parameters<typeof fn>` as a plain tuple at the type-checker layer; adding `SubKindParams` for binary alone would force asymmetric dispatch across the whole family. Migration path documented (caller-driven `{allOptional?, sliceStart?}` options). **Verified ⚠️ by-design.** The 11 + 1 = **12 mion cases have no ts-go counterpart by design** (§6).

2. **Object-union members are MERGED into one flat envelope (`-1`/`0xFF` sentinel), not encoded per-arm.** mion (toBinary.ts:373-397) compiles each object union member as a **standalone full object** behind its own discriminator index (`getEncodeCode(childRt)` = `comp.compile(childRt)`), and decodes by `if (dec === index) { v = <full object decode> }` (fromBinary.ts:435-447). The port (`union_flat_binary.go`) instead writes ONE sentinel discriminator (`0xFF`/`0xFFFF`) for the whole object branch, then a **merged-property** layout: required merged props in order, then a shared optional-prop bitmap, with per-prop sub-discriminators when a prop has multiple candidate types across members (`buildFlatLayout`/`FlatMergedProp`, union_flat_layout.go). This is the **same flat-union strategy the JSON serialization family uses** (`union_flat.go`), shared via `buildFlatLayout`. It is a genuine wire-format divergence from mion (a mion-encoded object-union buffer will NOT decode with the port's decoder and vice-versa), but it is internally consistent (encode/decode are byte-symmetric) and round-trips correctly (suite cases `union_object_with_discriminator`, `union_mixed_with_discriminator`, `circular_union_with_discriminator`, `large_class_union`). **⚠️ by-design** (binary intentionally tracks the JSON family's flat layout; comment union_flat_binary.go:11-24). Not mion-wire-compatible — flagged so a future "mion interop" goal knows.

3. **`symbol` at root → unsupported (alwaysThrow), not serialized-by-description.** mion serializes `v.description` and reconstructs `Symbol(desc)` (to:68; from:71). The port returns `CodeNS` for `KindSymbol` (binary_to.go:216-219; binary_from.go:141-143) → alwaysThrow factory. This is the **family-wide** symbol decision in `docs/UNSUPPORTED-KINDS.md` ("symbol identity does not survive serialisation; not comparable across realms") applied uniformly to isType / typeErrors-leaf / JSON / binary. Symbol *literals* (well-known symbols, `KindLiteral` + `Flags=['symbol']`) remain supported (binary_from.go:259-266). Suite `symbol` case asserts the alwaysThrow. **⚠️ by-design**, consistent with items 02/03/05.

4. **Rest tuple member carries an explicit `uint32` length prefix.** The port writes `setUint32(index, v.length - pos)` before the rest items (binary_to.go:728-731) and reads it back (binary_from.go:569-570). mion's rest path reuses the array emit which already prefixes a `uint32` length (to:110-126 via the `rest`/`array` shared arm), so this is the **same wire bytes** reached by a different code path — documented in the Go comment (binary_to.go:724-727) as required to keep the decoder aligned. No behaviour difference; listed for completeness.

5. **DataView option renames + truncation hardening.** The port's `SerializationOptions` renames mion's fields for clarity: `routeId`→`cacheKey`, `bufferSize`→`defaultBufferSize`, `responseAverageSizes`→`sizeHistory`, `averageResponseSizeMultiplier`→`sizeMultiplier`, `workflowRouteIds`→`relatedKeys` (a `CreateSerializerOptions.relatedKeys` object/number overload), and DROPS mion's unused `maxPoolItems`. The port also **throws `RangeError` on `encodeInto` truncation** (dataView.ts:218-221,243-246) where mion silently writes a corrupt length prefix (mion dataView.ts:108,123 — `result.written` with no read-check). Both faithful intent-preserving; the rename means `setSerializationOptions({bufferSize})` (mion) ≠ `setSerializationOptions({defaultBufferSize})` (port) — a source-incompatible API surface, deliberate. **⚠️ by-design.**

6. **Single switch vs mion's two-file switch.** Architectural (overview §"Why the architectures differ"). mion already used a switch compiler for binary (not per-node methods), so this family is the closest structural match of any — the port is `binary_to.go`+`binary_from.go`+`union_flat_binary.go` vs mion's `toBinary.ts`+`fromBinary.ts`.

## 5. Gaps, mismatches & missed optimisations

1. **User-class instances decode to plain objects — class reconstruction not emitted.** Severity: **Low** (by-design carve-out; consistent with the documented `TypeClass.classType` won't-capture decision). Evidence: mion `fromBinary.ts:341-356` (the `class` default/`SubKindNone` arm) builds the plain object, then emits `let desFn = utl.getDeserializeFn(<className>); if (desFn) {v = desFn(v)} else if (desFn = utl.getSerializeClass(<className>)) {v = new desFn(v)}` — reconstructing the registered class instance. The port `binary_from.go:193-194` (`case protocol.SubKindNone: return emitObjectFromBinary(...)`) returns only the plain object; there is no `getDeserializeFn`/`getSerializeClass` call anywhere in `internal/compiled/typefns/` (grep → only `rtUtils.ts:141,156` define the registry, never called from emit). Impact: `createBinaryDecoder<MyClass>()` returns `{...data}`, not `new MyClass(...)` — methods/prototype lost. Matches JSON `restoreFromJson` behaviour and the ROADMAP "constructor reference is a JS value, not planned" row; the suite's `class` case uses `deserializedValues` to expect a plain object, so this is blessed, not caught. Fix: none required unless class-reconstruction becomes in-scope (would need the registry wired into `emitObjectFromBinary` for `SubKindNone`, + a className on the protocol RunType).

2. **`noLiterals` option not plumbed → a literal always writes zero bytes.** Severity: **Low** (the default `noLiterals=false` path is correct and is what every caller hits; only the rare `noLiterals` request diverges). Evidence: mion `toBinary.ts:86-106` / `fromBinary.ts:89-109`, under `comp.opts.noLiterals`, dispatch a literal to its underlying primitive's encode/decode (so the literal's runtime value is actually written/read). The port `emitLiteralToBinary` (binary_to.go:335-340) is an unconditional noop with a comment "v1: we don't carry noLiterals on the protocol RunType yet, so always fall through to the 'skip' branch", and `emitLiteralFromBinary` always restores from the RunType (binary_from.go:238-279). Impact: under `noLiterals`, a `42` literal would round-trip via the wrong path; but `noLiterals` is not surfaced on `BinaryEncoderOptions` so it is unreachable from the public API today. Same deferral pattern as isType/typeErrors `noLiterals`. Fix: surface `noLiterals` via `RunType.Flags` (forked hash) and dispatch in `emitLiteral{To,From}Binary`, when/if a caller needs it.

3. **No `MAX_UNION_ITEMS` ceiling.** Severity: **Low** (the discriminator auto-widens uint8→uint16 at >255 members, so correctness holds up to 65535; mion's ceiling is a guardrail, not a correctness requirement). Evidence: mion `toBinary.ts:359-364` / `fromBinary.ts:423-428` throw when `totalLength > MAX_UNION_ITEMS` (`constants.ts`). The port `union_flat_binary.go` has `discriminatorWidth` (uint8 ≤255, else uint16, :31-36) but no upper-bound check. Impact: a union with >65535 members would overflow the uint16 discriminator (practically impossible; tsgo would choke first). Fix: optional — add a `len > 65535` guard returning `CodeNS`/diagnostic for parity.

4. **`serFloat64` / `desFloat64` are dead methods on the DataView classes.** Severity: **Low** (cosmetic; mion carries them too). Evidence: `dataView.ts:253-256` (`serFloat64`) and `:323-327` (`desFloat64`) are defined but never referenced by the emitters — `grep -rn 'serFloat64|desFloat64' internal/ src/caches/` → 0 hits; both `Emit` sides write/read float64 inline (`view.setFloat64`/`view.getFloat64`, binary_to.go:190 / binary_from.go:117). mion's `dataView.ts:132-135,206-210` are identically dead. Impact: ~10 lines of unused public surface. Fix: keep (mion parity) or drop from the public `DataViewSerializer`/`Deserializer` interfaces.

5. **Stale `rtCompilers/binary/...` paths in Go doc-comments.** Severity: **Low** (documentation only). Evidence: binary_to.go:21-22 ("Mirrors mion's mega-switch at mion/packages/run-types/src/rtCompilers/binary/toBinary.ts"), binary_from.go:17, and ~30 inline `mion:binary/...` cites; the real mion path is `jitCompilers/binary/...` (`ls` confirms no `rtCompilers/` dir). Impact: a contributor following the comment hits a missing path. Fix: s/rtCompilers/jitCompilers/ in the binary Go comments.
