# RunTypeSubKind is not exported from the package index — DONE

## Original finding (mion drizze migration, 2026-07-12)

Builtin classes project atomically with a `subKind` (Date → `{kind: 20, subKind: 2001,
typeName: 'Date'}`), and `RunTypeSubKind` (with `date: 2001` etc.) existed in
`src/runTypeKind.ts` — but was NOT re-exported from the package index, so graph
consumers couldn't reach it and had to detect Date by `typeName === 'Date'`, which
false-positives on a user class literally named `Date`. The consumer keying rule in
CLAUDE.md ("every consumer keys on subKind") was not followable from outside the package.

## What shipped

- `packages/ts-runtypes/src/index.ts` now re-exports the full generated constant surface:
  `RunTypeKind`, `RunTypeSubKind`, and the `RunTypeKindName` / `RunTypeKindValue` /
  `RunTypeSubKindName` / `RunTypeSubKindValue` helper types.
- The requested "const map generated from Go" design was ALREADY in place and needed no
  change: `src/runTypeKind.ts` is auto-generated by `cmd/gen-run-type-kind` from
  `internal/protocol/protocol.go` + `internal/protocol/subkind.go` (`pnpm run
  gen:run-type-kind`), emits both const objects as `{name: index}` matching the Go wire
  values byte-for-byte, and the `TestRunTypeKindFileInSync` Go test fails CI on any drift.
  The gap was purely the missing index re-export.
- FE test `packages/ts-runtypes/test/features/kindConstants.test.ts`: asserts both maps
  are importable from the index with the Go wire values, and pins the consumer pattern
  (Date detection via `subKind === RunTypeSubKind.date` on a reflected node) in both
  `getRunType` call shapes.
