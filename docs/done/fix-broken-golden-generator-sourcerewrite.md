# `sourcerewrite/testdata/gen_golden.mjs` is broken â€” golden fixtures can't be regenerated

## Intent

The `TestApply_Golden` fixtures under [ts-go-runtypes/internal/compiler/sourcerewrite/testdata/](../../ts-go-runtypes/internal/compiler/sourcerewrite/testdata/) (`*.json`) can no longer be regenerated by their generator. `gen_golden.mjs` is **doubly stale** â€” both breakages predate the change that surfaced them (the `createX` â†’ `createXFn` rename PR, which had to work around it):

1. **Stale import path** after the Go-tree relocation (`99e7b177`): the generator imports via `../../../../packages/...`, which resolves to `ts-go-runtypes/packages/` (nonexistent). It needs `../../../../../packages/...` (5 levels up to the repo root). The "run from repo root" header comment is also stale (`internal/â€¦` â†’ `ts-go-runtypes/internal/â€¦`).
2. **Dead dependency (the real blocker):** it imports `{rewrite}` from `packages/ts-runtypes-devtools/dist/rewrite.js`, but that JS rewrite oracle was **deleted** in `7031676c` ("thin Vite wrapper â€” Go owns the transform via OpTransform"). The function `gen_golden.mjs` drives no longer exists in the dist.

Impact: the golden fixtures are effectively frozen. Any future change to the rewrite / source-map logic can't refresh them, and a change that touches the synthetic fixture source (identifier text) breaks the baked UTF-8 byte offsets with no regeneration path. The `createXFn` PR handled this by **leaving the `testdata/` fixtures at their committed synthetic names** (the callee identifier is immaterial to the byte-offset/sourcemap mechanics under test) and filing this.

## Direction

Rebuild the generator to drive the **current** transform oracle instead of the deleted JS `rewrite()`. Since Go now owns the transform, the natural oracle is the Go binary's `OpTransform` in `'go'` wire mode (returns whole `{code, map}`): spawn `bin/ts-runtypes` (or add a small `ts-go-runtypes/cmd/gen-golden` helper) and feed each case's `{file, code, sites, replacements}`, capturing `expectedCode` / `expectedMap`. Fix the import path + header comment staleness at the same time. The implementer decides whether spawning the prebuilt binary or a dedicated `cmd/` helper is cleaner, and works out the exact `OpTransform` request shape (see the `transformMode: 'go'` path and `protocol` package).

Verify by regenerating all fixtures, confirming `go -C ts-go-runtypes test ./internal/compiler/sourcerewrite/...` stays green, then perturbing one case to confirm the generator actually recomputes offsets (not just rewrites bytes).

## Resolution â€” shipped 2026-07-24

Replaced both broken `.mjs` generators with a single Go helper,
[`ts-go-runtypes/cmd/gen-sourcerewrite-golden/main.go`](../../ts-go-runtypes/cmd/gen-sourcerewrite-golden/main.go),
matching the repo's existing `cmd/gen-*` convention.

**Oracle: `sourcerewrite.Apply`, NOT the binary's `OpTransform`.** The Direction above suggested
driving `OpTransform` in `'go'` mode. Investigation showed that cannot work for these fixtures: an
`OpTransform` request carries only file paths, and the resolver computes `sites`/`replacements`
itself from a real type-check scan, so it has no seam to inject the fixtures' **synthetic** sites
(fabricated typeIds like `Abc1234`, multi-fn binding arrays, hand-authored pure-fn replacements).
Those synthetic inputs feed the lower-level `Apply(file, source, sites, replacements)` directly â€”
exactly what `TestApply_Golden` / `TestApply_ExtraDiff` / `TestComputeEdits_MatchesApply` call â€” so
the generator calls `Apply` directly. No binary spawn, no JS, no type-checking. Go byte offsets are
native (`strings.Index`), so the JS `Buffer.byteLength` conversion dance is gone.

**Both generators were broken, both replaced.** `testdata/extra/diff_extra.mjs` (which produces
`testdata/extra/cases.json`) had the identical dead `dist/rewrite.js` import; the one Go helper now
regenerates BOTH the per-case `testdata/*.json` and the `testdata/extra/cases.json` array.

**What the suite is now.** With the independent JS oracle deleted, `TestApply_Golden` is a
snapshot/regression test: the committed JSON is the reviewed baseline; any change to
`Apply`/`EditBuffer`/the VLQ source-map math fails it until the fixtures are regenerated and the
diff re-reviewed. The surviving differential is internal â€” `TestComputeEdits_MatchesApply` pins the
`'edits'` wire path byte-identical to `Apply` over the same corpus.
