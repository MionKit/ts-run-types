## 1. Goal

Migrate the JS half of the toolchain to the oxc suite end-to-end:

1. **oxlint** replaces ESLint + typescript-eslint as the repo's general linter (oxlint is already
   installed, but only as the enrichment-plugin host — see §2.1).
2. **oxfmt** (the oxc formatter) replaces Prettier for **TypeScript** dev-time formatting, where it
   reproduces our style. Prettier stays for markdown AND for the playground's in-browser use (§3.3).
3. **Vite 8+** (Rolldown-powered, oxc-based by default) replaces Vite 5 where the workspace builds or
   tests with Vite — `ts-runtypes-devtools` (devDep + peer range + its bundler test matrix), Vitest, the
   playground `demo`/`build:site`, and eventually the two container apps.
4. Every command that fronts these tools — `pnpm run lint`, `pnpm run format`, `pnpm run check-format`,
   the lint-staged / husky pre-commit entries, CI — is rewired and stays green with **stable script names**.

Strategic fit: oxlint's type-aware backend is **tsgolint — the same `oxc-project/tsgolint` shim this repo
already vendors at [`third_party/tsgolint/`](../../third_party/tsgolint) and builds the resolver on**.
Converging the toolchain onto oxc aligns the repo with the stack the product itself is built on, and the
[OXlint diagnostics plugin](oxlint-diagnostics-plugin.md) gets an in-repo dogfooding host for free — in
fact it **already has one** (§2.1), which this migration merges into the general lint config.

  binary with no browser standalone bundle, so it **cannot** replace this. Leave the dependency and the
  in-browser usage as-is. (Future: if oxfmt or oxc ever ships a WASM/browser build, revisit — but that's
  a separate investigation, not part of this migration.)
- **Prettier therefore remains a repo dependency** (md formatting + playground runtime). This migration
  only demotes Prettier from "the TS formatter" — it does NOT remove the package.
- `check-format` mirrors: `oxfmt --check <ts> && prettier --check 'packages/**/*.md' && gofmt -l …`.
  `check-format-pre-commit` swaps `prettier --check` → `oxfmt --check` for staged `.ts` (md staged files
  keep prettier).
- **One-time reformat commit:** run the new pipeline over `packages/` and land any byte diffs as an
  isolated `style:` commit (no logic changes mixed in). **GATE:** if the diff is unreasonably large or
  mangles constructs, that's a no-go — ship the lint + Vite halves alone and defer oxfmt (document the
  deferral here). Update CLAUDE.md's "Format means…" wording (it names Prettier) + SETUP.md in the same PR.

### 3.4 Sequencing / independence

The three tracks are independently landable, in rising risk order: **oxlint (lowest risk, land first) →
Vite 8 + Vitest 4 (medium; blast radius = the whole test suite) → oxfmt (youngest tool; can trail or be
dropped without blocking the others)**. The original spec says "do not couple them in one PR"; per the
current request they may ride **one branch as three sequential, individually-green commits** (each COULD
be split into its own PR) so the whole migration lands together while staying reviewable step-by-step.

## 8. `scripts/setup-claude-web.sh` — does it need changes? (answer: cache-bust only)

Reviewed 2026-07-04. The migration is **pure npm-dependency churn** (add oxfmt, bump vite/vitest, add root
vite, remove eslint packages) plus config-file edits. The web setup script:

- runs `pnpm install --frozen-lockfile` (step 6) → picks up the new lockfile automatically **on a fresh
  container**;
- builds the Go binary (unchanged) and the devtools dist (unchanged);
- installs no linter/formatter/bundler at the system level (they're all workspace npm deps).

So **its logic needs NO change**. The ONE required action is a **cache-bust**: `install_workspace_deps`
skips `pnpm install` when `node_modules` already exists, so an existing/reused web container would keep the
OLD deps (eslint present, Vite 5, no oxfmt) even though the lockfile changed. Bumping `SETUP_DATE`
(line 3, currently `2026-07-02`) to the migration's merge date forces a clean container that re-clones +
reinstalls against the new lockfile. **The user must re-paste the updated script into the Claude-web
environment's setup-script field after the bump** (the web env holds a verbatim copy, per the file's
self-contained-copy banner). Caveat to check at implementation time: if `@rolldown/binding-*` needs a
build-script allowlist (§4), that's a `pnpm-workspace.yaml` change — the setup script still just runs
`pnpm install`, but a blocked build script would surface as an install failure there.
