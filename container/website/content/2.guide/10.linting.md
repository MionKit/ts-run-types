RunTypes ships a lint plugin inside the `ts-runtypes-devtools` package. It works with OXlint (the primary target, with live editor feedback through the oxc language server) and with ESLint v9 flat config. The same compiler that powers your build produces the findings, so what you see in the editor is exactly what the build would say.

The plugin covers two areas:

1. **Compiler problems in your source files.** Unsupported types at a call site, marker misuse, and members that validators or encoders silently skip. These appear at the exact call site while you type.
2. **Enrichment file health.** Generated `FriendlyText` and `MockData` files must be finished before they land: no unfilled `@todo` placeholders, no stale `@rtOrphan` carcasses, no entries pointing at fields the type no longer has, and no mirrors whose source file moved or disappeared.

## Setup with OXlint

Install `oxlint` and point its config at the plugin entry inside `ts-runtypes-devtools`. There is no separate package to install.

```json
{
  "$schema": "./node_modules/oxlint/configuration_schema.json",
  "jsPlugins": ["./node_modules/ts-runtypes-devtools/dist/eslint/index.js"],
  "rules": {
    "runtypes/error": "error",
    "runtypes/warn": "warn",
    "runtypes/info": "off",
    "runtypes/no-enrichment-todo": "error",
    "runtypes/no-orphan-carcass": "error",
    "runtypes/enrichment-field": "error",
    "runtypes/enrichment-drift": "error"
  }
}
```

With the oxc editor extension installed (VS Code, Zed, JetBrains, Neovim and others through the same language server), findings show up in the Problems panel and as squiggles while you edit. No extra editor wiring is needed.

## Setup with ESLint

The same entry is a regular flat-config plugin:

```js
import runtypes from 'ts-runtypes-devtools/eslint';

export default [
  {
    files: ['**/*.ts'],
    plugins: {runtypes},
    rules: {
      'runtypes/error': 'error',
      'runtypes/warn': 'warn',
      'runtypes/no-enrichment-todo': 'error',
      'runtypes/no-orphan-carcass': 'error',
      'runtypes/enrichment-field': 'error',
      'runtypes/enrichment-drift': 'error',
    },
  },
];
```

## The rules

Compiler findings carry their own severity, so they route to a rule named after it. The specific code (for example `[VL010]`) is always part of the message.

| Rule | What it reports |
| --- | --- |
| `runtypes/error` | Problems that will throw at runtime or stop the build, such as a type that cannot be encoded. |
| `runtypes/warn` | Intentional behaviour worth knowing about, such as a function property a validator silently skips. |
| `runtypes/info` | Advisory notes. Off by default. |

Enrichment findings get one rule per concern, so a team can tune each gate:

| Rule | What it reports |
| --- | --- |
| `runtypes/no-enrichment-todo` | An unfilled `@todo` scaffold line. Fill in the value, then delete the line. |
| `runtypes/no-orphan-carcass` | A stale `@rtOrphan` or `@rtOrphanChild` block. Run `ts-runtypes gen --prune`, or restore the type. |
| `runtypes/enrichment-field` | A map entry the type no longer declares, an unknown error key, or a bad `$[placeholder]`. |
| `runtypes/enrichment-drift` | A mirror whose source file no longer exists or no longer declares an imported type. |

The `@rtType` and `@rtIds` markers are part of every healthy generated file and are never reported.

## Blocking dirty files at commit time

`oxlint` exits non zero when an error level rule fires, so the same config works as a commit gate. With husky and lint-staged:

```json
{
  "lint-staged": {
    "**/*.ts": ["oxlint"]
  }
}
```

A staged enrichment file with an unfilled `@todo` (or any other error level finding) now blocks the commit. Run the same command in CI so pushes that skip the hook are still covered.

The enrichment rules only ever fire on files that look like generated enrichment files (a reconcile marker or a `FriendlyText` / `MockData` annotation). A hand written file with a `@todo` comment is never flagged, even if your lint glob is broad. If you want an extra fence, scope the rules to your mirror directory with an override:

```json
{
  "overrides": [
    {
      "files": ["runtypes/generated/**/*.ts"],
      "rules": {
        "runtypes/no-enrichment-todo": "error",
        "runtypes/no-orphan-carcass": "error"
      }
    }
  ]
}
```

## Checking without a linter

The same findings are available straight from the CLI, which is handy for CI jobs or scripts that do not run node linting:

```bash
ts-runtypes check src/runtypes/generated/user.ts --json
ts-runtypes gen --check --json
```

`check` reports the tag hygiene, content validity, and drift findings for one file (exit 1 on errors). `gen --check` walks the whole mirror directory for drift.

## Configuration

The plugin resolves the RunTypes binary the same way the build plugins do, through the `ts-runtypes-bin` launcher that ships with `ts-runtypes-devtools`. Most projects need no configuration at all. When you do need it, use the shared lint settings:

```json
{
  "settings": {
    "runtypes": {
      "binary": "./bin/ts-runtypes",
      "cwd": ".",
      "timeoutMs": 60000
    }
  }
}
```

`binary` overrides the resolver executable, `cwd` sets the directory file paths are resolved against, and `timeoutMs` bounds how long one file may take (the first file pays the engine start).

Two notes on how the plugin runs:

1. The first linted file starts one small resolver process that then serves the whole run, so a full lint pass stays fast and repeated passes on unchanged files answer from a cache.
2. The plugin analyses each file with an inferred TypeScript project rooted at that file. Imports resolve from disk as usual, but options from your `tsconfig.json` are not applied during linting. If a finding depends on a tsconfig option (for example the Temporal lib), confirm it with a build.
