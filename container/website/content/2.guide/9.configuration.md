RunTypes reads its settings from your `tsconfig.json`, in the same `plugins` list TypeScript already understands. You configure a project in one place, and every build tool (Vite, Rollup, webpack, Rspack, esbuild) picks it up for free.

## The tsconfig plugin entry

Add an entry named `ts-runtypes` under `compilerOptions.plugins`:

```json
{
  "compilerOptions": {
    "plugins": [
      {
        "name": "ts-runtypes",
        "emitMode": "code",
        "moduleMode": "default",
        "inlineMode": "default",
        "hashLength": 7,
        "cacheDir": "node_modules/.cache/ts-runtypes",
        "parallelScan": true,
        "parallelRender": true
      }
    ]
  }
}
```

Every key is optional. Leave one out and the compiler uses its default.

## Options

| Option | Default | What it does |
| --- | --- | --- |
| `emitMode` | `code` | What each generated function ships. `code` rebuilds the function from a body string on first use (smallest output). `functions` ships a live function. `both` ships both, for runtimes that forbid dynamic code (Cloudflare Workers, a strict CSP). |
| `moduleMode` | `default` | How cache entries group into modules. `default` gives the best code splitting. `allSingle` bundles each family into one module (fewest files). `allModules` splits everything (an escape hatch). |
| `inlineMode` | `default` | Whether unnamed nested types inline into their parent (`default`) or everything except recursive types inlines (`allInternal`). |
| `hashLength` | `7` | Length of the short type ids in generated names. Raise it only if a build ever reports an id collision. |
| `cacheDir` | `node_modules/.cache/ts-runtypes` | Where the incremental build cache lives. A relative path resolves against the project root. |
| `singleThreaded` | `false` | Run the compiler on one thread. Mostly for debugging. |
| `parallelScan`, `parallelRender` | `true` | Run the scan and the render in parallel. Turn either off for a clean benchmark baseline. |
| `sizeBias` | `0.8` | How generous the `dynamic` binary encoder's first buffer is, from 0 (tightest, may grow more often) to 1 (most generous). |
| `sizeItems` | `100` | Assumed element count for an unbounded array, Map or Set when sizing that first buffer. |
| `sizeStringBytes` | `32` | Assumed length of an unbounded string when sizing that first buffer. |
| `sizeMaxBytes` | `65536` | Upper bound on the estimated first buffer, so a very large declared limit never sizes a huge buffer. |
| `enrichDir` | `runtypes/generated` | Where the [enrichment CLI](/guide/validation) writes its committed mirror files, one subfolder per family: `friendly/` for labels and messages, `mock/` for sample data (plus `i18n/` for translations). |
| `friendlyErrors` | `perConstraint` | Which `$errors` shape `gen` scaffolds for a **new** node in a [FriendlyType](/ai-integration/friendly-type) map: `perConstraint` (one key per rule the field declares) or `default` (a single `$default` catch-all). Only affects first scaffolds; once a node exists, its authored mode is kept on every sync. |
| `i18n` | unset | Translation settings for the friendly maps: source locale, target locales and the translation folder. An object, detailed [below](#translations-i18n). |

## Translations (i18n)

The `i18n` object configures [translations](/ai-integration/i18n) for your [FriendlyType](/ai-integration/friendly-type) maps. Leave the key out entirely and translations are off, with nothing else changed:

```json
{
  "name": "ts-runtypes",
  "i18n": {
    "sourceLocale": "en",
    "dir": "runtypes/generated/i18n",
    "locales": ["es", "pl", "pt-BR"],
    "strict": false
  }
}
```

| Key | Default | What it does |
| --- | --- | --- |
| `sourceLocale` | `en` | The language your friendly maps are written in. It decides which plural forms `gen` scaffolds in them. |
| `dir` | `<enrichDir>/i18n` | Where translation files live, one folder per locale. A relative path resolves against the project root. |
| `locales` | none | The locales you translate into; `gen --translate all` and `check --translate all` walk this list. Don't list the source locale. |
| `strict` | `false` | When `true`, an incomplete or out-of-date translation fails `check --translate` instead of warning. Rendering at runtime stays lenient either way. |

## Precedence

Settings layer the way `tsc` does. A command-line flag or a build-tool option wins over the tsconfig entry, which wins over the built-in default:

```
build-tool option / CLI flag   (highest)
tsconfig plugin entry
built-in default               (lowest)
```

So tsconfig sets your project baseline, and a single build can override one knob without touching tsconfig.

::tip
Set your options once in tsconfig. Reach for a build-tool option or a CLI flag only when a single build needs to differ.
::

## Build-tool options

A few settings belong to the build tool rather than the project, so they live in your bundler config (the `runtypes-devtools` plugin) and have no tsconfig key:

```ts
import {defineConfig} from 'vite';
import runtypes from 'runtypes-devtools/vite';

export default defineConfig({
  plugins: [
    runtypes({
      binary: '/path/to/ts-runtypes', // a custom resolver build, for development
      cwd: process.cwd(), // project root, defaults to the bundler root
      tsconfig: 'tsconfig.json', // tsconfig path, relative to cwd
      outDir: 'src/__runtypes', // where generated modules are written
    }),
  ],
});
```

That same plugin also accepts the project options above (`emitMode`, `moduleMode`, and the rest) as a per-build override. Reach for that only when one build needs to differ. The project's home for them is tsconfig.

## Resolution conditions

To resolve packages under a custom export condition (`worker`, `browser`, or a private one), use TypeScript's own `customConditions`. The compiler reads it natively, so there is no separate plugin key:

```json
{
  "compilerOptions": {
    "customConditions": ["worker"]
  }
}
```

## Custom markers

::note
The scanner recognises the marker types from `ts-runtypes` by a brand carried on each one. A package that re-exports or vendors those types keeps the brand, so its markers are recognised automatically, with nothing to configure. Only a fully custom brand needs the embedded-resolver escape hatch.
::

## Serialization defaults at runtime

The binary serializer's tuning knobs (cold-start buffer size, internal cache sizes) apply at runtime, not at build time. Set them once at startup:

```ts
import {setSerializationOptions} from 'ts-runtypes';

setSerializationOptions({defaultBufferSize: 2 ** 20});
```

With the `dynamic` strategy, the first buffer is sized from a compile-time estimate of your type (tuned by `sizeBias`, `sizeItems` and the rest above), so a cold encode rarely falls back to `defaultBufferSize`. The fallback still applies when there is no estimate, for example a value-first schema built at runtime.

## A misspelt key

::note
An unrecognised key in the plugin entry is ignored, and the build prints a warning that names it. A typo like `emitMdoe` never fails silently, it just tells you.
::
