The suites on the previous pages are hand-written: real types, real values, known answers. They're thorough, but they only try the inputs **we** thought of. Fuzzing covers the rest: the weird, the malformed, and the cases nobody would write by hand.

RunTypes is fuzzed on two levels at once: random **values**, and random **types**. Both run through the real build pipeline, and both are checked against rules that must hold for every input, never a hand-picked expected answer.

## Properties, not examples

A normal test says "validate should return true for this value." A fuzz oracle says something stronger: a property that has to hold no matter what the input is. When a property breaks we have found a bug, even though nobody wrote a test for that exact case.

The rules we check include:

- validate always returns a boolean and never throws, for any input at all.
- validate(x) is true exactly when getValidationErrors(x) is empty. The two can never disagree.
- A value survives a JSON round-trip unchanged, and survives a binary round-trip unchanged.
- The JSON wire and the binary wire decode to the same value. Two independent encoders, one answer.
- Every JSON strategy (clone, mutate, direct, compact) and the binary codec round-trip the same value to the same result. One type, five codecs, one answer.
- Every serializer in a family agrees on whether a given type can be serialized at all.

None of these is a hand-written answer. They are laws the library promises to uphold, and the fuzzer tries thousands of random ways to break each one.

## The full list of rules

The list above is the short version. Below is the complete set, naming the functions each rule exercises, so you can see in one place every promise the library keeps. Each rule has a short label, so when a run reports a break it points straight at the rule that broke.

The functions under test are the ones you call yourself: `createValidate<T>()` and `createGetValidationErrors<T>()` for checking a value, `createJsonEncoder<T>()` / `createJsonDecoder<T>()` and `createBinaryEncoder<T>()` / `createBinaryDecoder<T>()` for the two save formats, and `createMockType<T>()` for building a sample value of a type.

**About a value and its type.** Every generated value is run through these.

| ID | Rule | What must always be true |
|----|------|--------------------------|
| O1 | Mock values pass | A value from `createMockType<T>()` always passes `createValidate<T>()`. The generator and the validator agree on what a `T` is. |
| O2 | Broken values fail | Take a valid value, break it at one spot so it no longer matches the type, and `createValidate<T>()` rejects it. |
| O3 | Validate never crashes | `createValidate<T>()` returns a plain true or false for any input at all, and never throws. |
| O4 | Validate matches its errors | `createValidate<T>()` returns true exactly when `createGetValidationErrors<T>()` returns an empty list. The boolean and the error list can never disagree. |
| O5 | JSON round-trip is stable | Encode with `createJsonEncoder<T>()`, read it back with `createJsonDecoder<T>()`, then encode again: the second JSON string matches the first. |
| O6 | Binary round-trip is stable | The same round-trip with `createBinaryEncoder<T>()` and `createBinaryDecoder<T>()`, and the two byte arrays match exactly. |
| O7 | Encoding never throws by surprise | On a valid value, `createJsonEncoder<T>()` and `createBinaryEncoder<T>()` never fail with an unexpected error. (A deliberate "this type cannot be encoded" refusal is allowed, and O10 and O14 cover that case.) |
| O10 | A refusal comes with a reason | When both `createJsonEncoder<T>()` and `createBinaryEncoder<T>()` refuse a type, the build has reported an error explaining why. It never refuses silently. |
| O12 | JSON and binary round-trip alike | Send a value through the binary round-trip (`createBinaryEncoder<T>()` then `createBinaryDecoder<T>()`), re-encode the result as JSON, and it matches the value encoded straight to JSON. The two formats never disagree about the data. |
| O14 | Both encoders accept the same types | `createJsonEncoder<T>()` and `createBinaryEncoder<T>()` always agree on whether a `T` can be encoded at all. Neither accepts a type the other refuses. |

**About the build itself.** Every generated type is run through these as it compiles.

| ID | Rule | What must always be true |
|----|------|--------------------------|
| TR1 | Clean compile | A sound type compiles with no crash and no false error from the resolver. |
| TR2 | Every call is matched | Every `createValidate<T>()`, `createJsonEncoder<T>()`, and sibling call in your code is matched to its compiled output, with none left unresolved. |
| TR3 | Generated code runs | Every module the compiler emits (the code behind those factories) is valid JavaScript that runs, with no missing references. |
| TR4 | The functions build | `createValidate`, `createGetValidationErrors`, the encoders, and the decoders all build successfully from the compiled data. |

## Two levels of randomness

**Random values.** For a fixed set of types, generate a flood of conforming values, deliberately broken values, and pure junk. Check every rule against every one.

**Random codecs.** A type does not have one serializer, it has several. The JSON encoder runs in four modes (a safe clone, an in-place transform, a single pass, and a compact positional form), each paired with the decoder that reads its wire, and there is a separate binary codec. The fuzzer builds all of them for the same random type and sends one value through every one. They must all agree: same value in, same value out, and they must agree on whether the type can be serialized at all. A mode that mishandles a shape the others get right has nowhere to hide.

**Random types.** This is the part most libraries never attempt. We generate the types themselves: arbitrary, valid but strange TypeScript. Unions, intersections, recursive interfaces, tuples, index signatures, Map and Set, classes, and the awkward non-data members like symbols, functions, methods, and typed arrays. Each generated type is fed through the entire build pipeline, which produces a real validator and real encoders and decoders, and those are then fuzzed like any other.

A single run might invent a type like this, build everything for it, and confirm it all behaves:

```ts
type Account = {
  id: bigint;
  labels: Set<string>;
  status: {kind: 'active'; since: Date} | {kind: 'closed'};
  refresh: () => void; // a non-data member
  [extra: string]: unknown; // an index signature
};
```

## How a run is built

Each iteration does four things, all driven from a single seed so any finding replays exactly:

1. **Invent a type.** A recursive generator builds a random type across the widest shape space we can express.
2. **Compile it for real.** The type goes through the actual resolver and plugin, the same path a user's build takes. If the pipeline crashes, hangs, or emits code that will not run, that alone is a finding.
3. **Build a real value.** The value comes from our own mock generator, the same one users call, walking the compiled type to produce a conforming value. Routing the real mock through the real serializers tests the whole stack from end to end, not a simplified stand-in.
4. **Check the rules.** Run the oracles. Any broken property reports the seed that broke it.

## Why this finds what others miss

Three choices do the heavy lifting.

We fuzz the **types**, not only the values. A library that only fuzzes values can never discover that a particular shape of type compiles to a broken function. We generate the shapes.

We feed values from the **real mock generator**, not a hand-tuned helper. That means the mock, the validator, and both serializers are all exercised together. A disagreement anywhere shows up.

We check **metamorphic properties** (round-trip stability, cross-wire agreement, family agreement) instead of expected answers. These catch bugs nobody would write a test for, because nobody imagined the input.

There is one more thing this style is uniquely good at. RunTypes serializes the data-only projection of a type: a method, a symbol, or a typed array carries no JSON-shaped value, so it is dropped where that is safe and refused where it is not. The fuzzer mints values that deliberately contain those non-data members, then confirms each one is handled the right way: dropped from a property, kept across a round-trip where the surrounding data survives, or rejected outright when a whole value could never be represented. That contract is far too large to cover by hand. The fuzzer covers it automatically.

## The bugs it caught

This is the part that pays for itself. Every bug below was found by one of the rules above, then fixed and pinned with a permanent regression test. The "Caught by" column names the rule that flagged it, so you can trace each fix back to the promise it protects.

| The bug | Caught by | What happened |
|---------|-----------|---------------|
| Binary buffer overflow | O6, O7 | The binary encoder overran its output buffer on a valid value, but only after a run of smaller values had sized the buffer a certain way. No hand-written test would line up that sequence. |
| Index rule leaked onto named fields | O12 | On an object mixing a catch-all index signature with named properties (much like the `Account` type above), the binary encoder applied the index rule to the named properties too and fell over, while JSON encoded the same value correctly. |
| A union branch that should have serialized | O10 | A union whose branch held a single non-data property (a lone symbol) was refused as a whole, though the same object on its own would drop that property and serialize the rest. The two paths now agree. |
| Callable interfaces treated two ways | O1 | An interface you can call like a function was a function to the validator and a plain object to the serializers, so no single value could satisfy both. A callable interface now counts as a function everywhere. |
| A hard error that should have been a warning | O5, O6 | The build raised a hard error on a value that round-trips cleanly, because the part it objected to (a non-data property such as a symbol or a typed array) was going to be dropped anyway. A dropped non-data property is now a warning; a type that genuinely cannot be dropped still fails loudly. |
| A codec that lost nulls from a list | O5 | The single-pass JSON encoder joined its array parts together and a null element vanished in the join, so a list of numbers-or-nulls came out as broken JSON. The same gap hit undefined and void elements inside a Set. |
| A Map mistaken for a plain record | O12 | A Record validator accepted a Map (a Map has no plain string keys, so the per-key check passed by default); inside a union the Map then matched the record branch and was serialized as an empty object on every wire. |
| A single-pass encoder that wrote invalid JSON | O5 | On an all-optional object that also carries a catch-all index signature, the compact single-pass strategy emitted a stray comma and produced JSON that would not parse back. |
| A key processed twice | O5, O7 | When a catch-all key allowed several kinds at once, the JSON codec swept each dynamic entry once per kind, so the second pass re-read an already-transformed value and threw. It now sweeps each entry once. |
| A sweep that wandered into named fields | O5, O7 | The catch-all pass also touched the object's named fields, walking the characters of a text field and tripping a safety limit. The pass now skips named fields, which are handled on their own. |
| A union that re-encoded to a different wire | O5 | A discriminated union with overlapping members could serialize one value two ways, so the JSON wire was not byte-stable across a round-trip (the binary format was unaffected). The encoder now picks each shared field's form from the discriminator, which survives the round-trip. |
| getValidationErrors disagreed with validate on records | O4 | The getValidationErrors counterpart of the Map-as-record bug above: a Record checker built by `createGetValidationErrors` reported zero errors for an array (and for Date, Map, Set) while `createValidate` correctly rejected it. The plain-object guard is now shared by both. Pinned by `verr-record-array-disagreement.test.ts`. |

The fuzzer is still earning its keep. That it keeps surfacing the next corner, each smaller than the last, is exactly the point of leaving it running.

Every finding arrives with a seed. Replay the seed, watch it fail, fix it, then keep the seed as a permanent test. That loop is why a feature can change and we still trust that thousands of strange types and values keep behaving.
