Speed only matters if the answer is right. Before comparing how fast each library runs, this page checks whether they even agree on what counts as a valid value.

Every library is given the exact same test data ts-runtypes is built against, and each cell counts the samples that library treats differently. A value of `0` means the library agrees with ts-runtypes everywhere on that case. A higher number means the library accepts values that ts-runtypes rejects. `n-a` means the library does not support that case at all. ts-runtypes is the reference column, so it is always `0`. Hover any row to see the schema each library authored for it.

## What the differences mean

Every difference below is a library being more permissive than ts-runtypes, never the other way around. On each one, ts-runtypes sits with at least one other mainstream library, so it is never the surprising choice.

**Non-finite numbers.** ajv and typia accept `NaN`, `Infinity`, and `-Infinity` as valid numbers. ts-runtypes rejects them (and so do zod and TypeBox), because none of them can be written to JSON and they are almost never what an API or a database wants.

**Invalid dates.** typia accepts a `Date` built from bad input (for example `new Date('invalid')`, whose time value is `NaN`). ts-runtypes checks that the date is real, as do zod and TypeBox.

**Plain objects.** For an object whose fields are all optional, TypeBox accepts class instances like `Date`, `Map`, `Set`, and `RegExp`, because structurally they fit a shape with no required fields. ts-runtypes requires a plain object, and zod matches that with a custom guard.

**Formats.** Every library ships its own patterns for things like email, UUID, and IP addresses, so a string one library accepts another may reject. The shared test data uses ts-runtypes' built-in patterns, so a stricter or looser competitor pattern shows up here.
