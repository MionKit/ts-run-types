Speed only matters if the answer is right. Before comparing how fast each library runs, this page checks whether they even agree on what counts as a valid value.

::tip
Across the whole suite, ts-runtypes and zod agree on every sample, with zero divergences. If you validate with zod today, ts-runtypes accepts and rejects exactly the same values, so on correctness it is a one-to-one replacement.
::

## What the differences mean

Every difference below is a library being more permissive than ts-runtypes, never the other way around, and on each one ts-runtypes sits with at least one other mainstream library.

- **Non-finite numbers.** ajv and typia accept `NaN` and `Infinity`; ts-runtypes rejects them (with zod and TypeBox), since none can be written to JSON.
- **Invalid dates.** typia accepts a `Date` built from bad input like `new Date('invalid')`; ts-runtypes checks the date is real, as do zod and TypeBox.
- **Plain objects.** For an all-optional object, TypeBox accepts class instances like `Date` and `Map`; ts-runtypes requires a plain object, matching zod.
- **Formats.** Each library ships its own email, UUID and IP patterns, so a string one accepts another may reject. The shared data uses ts-runtypes' patterns.

::note
The full cross-library methodology and findings live in the [alignment report](https://github.com/mionkit/ts-run-types/blob/main/docs/cross-library-validation-alignment-report.md) in the repository.
::

Every library is given the exact same test data ts-runtypes is built against, and each cell counts the samples that library treats differently. A value of `0` means the library agrees with ts-runtypes everywhere on that case. A higher number means the library accepts values that ts-runtypes rejects. `n-a` means the library does not support that case at all. ts-runtypes is the reference column, so it is always `0`. Hover any row to see the data tested and each library's validator.
