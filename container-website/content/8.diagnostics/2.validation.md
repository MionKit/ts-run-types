---
title: Validation
description: Diagnostics from createValidate and createGetValidationErrors.
---

`createValidate<T>()` writes a type guard for `T`, and `createGetValidationErrors<T>()` writes a function that reports what failed. Both check the data in your type, the part that survives JSON. That focus is why most codes here are warnings: a single member has no data form, so it is left out. A few are errors, for when the whole type has no data form and the generated function would throw. See [build-time diagnostics](/diagnostics/overview) for how the two differ, and the [validation guide](/guide/validation) for the same contract from the type side.

Codes come in pairs. The `VL` code is from `createValidate`, and the `VE` code is the same case from `createGetValidationErrors`.

## Root-position errors

These stop the build. The type at the root has no data form, so the generated function is replaced by one that throws when it is called.

### A non-serializable built-in at the root

**Error** · `VL001` `VE001`

```text
Cannot validate `Map`.
```

The type you asked to validate is a built-in that carries runtime state, like `Map`, `Set`, `WeakMap`, a typed array, or `Promise`. None of these survive a JSON round trip, so a guard that returned `true` for one would be claiming a safety it cannot deliver. Validate a plain shape instead, or convert before you validate:

```ts
// instead of validating a Map directly
const asObject = Object.fromEntries(myMap);
const isData = createValidate<Record<string, number>>();
```

### A bare symbol at the root

**Error** · `VL002` `VE002`

```text
Cannot validate `symbol`.
```

Every `symbol` has its own runtime identity, and `Symbol() !== Symbol()` even with the same description. That identity is gone the moment a value crosses a network or process boundary, so a check that asserts "this is a symbol" cannot mean anything to a caller on the other side. Use a stable string union:

```ts
type Status = 'pending' | 'active' | 'done';
```

## Member-level warnings

These keep the build going. One member of an object has no data form, so it is left out of the validated shape. The rest of the type is checked as normal.

### A function-valued property is dropped

**Warning** · `VL010` `VE010`

```text
Property `onClick` is a function — validate does not handle function values, so this property is silently not validated.
```

A property whose type is a function carries no data, so it cannot be part of a JSON-shaped check. The guard validates the rest of the object and ignores this property. Given `interface User { name: string; onClick: () => void }`, the generated guard checks `name` only, so a value with a wrong `onClick` still passes. If that is not what you want, drop the property from the type, or replace it with the data it would produce.

### A method is dropped

**Warning** · `VL011` `VE011`

```text
Method `save` is silently not validated by validate — methods aren't data.
```

Methods are behavior, not data, so they are not part of the serializable shape. If you wanted the result of a method checked, expose it as a data property instead.

### A static member is dropped

**Warning** · `VL012` `VE012`

```text
Static member `MAX` is silently not validated by validate — statics aren't part of instance data.
```

Static members live on the class, not on an instance. Validation works on instance shape, so statics are left out.

### A symbol-keyed property is dropped

**Warning** · `VL013` `VE013`

```text
Symbol-keyed property `tag` is silently not validated by validate — symbol keys aren't JSON-representable.
```

JSON has string keys only, so a symbol-keyed property has nowhere to land in the serialized form. Use a string key if the property is real data:

```ts
interface Item {
  id: string; // instead of [Symbol.for('id')]: string
}
```

### A union member with no data form is dropped

**Warning** · `VL014`

```text
Union member(s) of type `symbol` can't be represented as data — validate drops them, so the union is validated as its remaining members.
```

A union is validated as the members that have a data form. `Date | symbol` validates as `Date`, the same projection as `DataOnly<Date | symbol>`. The dropped members carry no data (a symbol, a function, a Promise, or a built-in like `Map`). If every member of the union has no data form the projection is `never`, and validation throws at build time instead. `createGetValidationErrors` reuses this case, so it reports `VL014` too.

### A property whose value type has no data form is dropped

**Warning** · `VL015` `VE015`

```text
Property `id` has a non-serialisable value type (symbol, Promise, or a non-serialisable built-in) — validate drops it, so this property is silently not validated.
```

This is the value-level twin of the function-property drop. A property whose value is a symbol, a Promise, or a non-serializable built-in has no data form, so `{ id: symbol }` validates as `{}`, matching `DataOnly<{ id: symbol }>`. Note the line between this and a value that is only structurally unserializable, like `id: symbol[]` or `id: Map<string, symbol>`. Those cannot be dropped without changing the shape, so validation throws at build time rather than warning.

### any or unknown accepts every value

**Warning** · `VL021` `VE020`

```text
`validate` on `any` / `unknown` always returns true — the validator accepts every value.
```

`any` and `unknown` describe anything, so a structural check has nothing to compare against. `createValidate` returns a guard that is always `true`, and `createGetValidationErrors` returns an empty error list every time. Narrow the type to the shape you actually expect:

```ts
const isUser = createValidate<User>(); // instead of <unknown>
```
