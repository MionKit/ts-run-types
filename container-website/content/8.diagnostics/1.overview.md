---
title: Build-time diagnostics
description: What every build-time diagnostic code means and how to act on it.
---

ts-runtypes resolves your types and writes out the matching functions while your project builds, so it can flag problems before any code runs. When it finds one it prints a short code, like `VL010` or `PJ001`, with a one line message. This section is the reference for those codes: what set each one off, why the rule is there, and what to change.

## Warning or error

Every code is one of two outcomes, and the difference tells you whether you need to act.

::note
**Warning** is an expected, safe drop, and the build keeps going. ts-runtypes works on the data in your type, the part that survives JSON, so anything with no data form (a method, a function valued property, a symbol key) is left out of the generated function. The code names what was left out, so it is never a surprise. Most of the time this is exactly what you want and there is nothing to fix.
::

::note
**Error** means the generated function would throw the moment you call it, so the build stops. This happens when a whole value has no data form: a type that resolves to `never`, a bare `symbol`, or a built-in like `Map` at the position being serialized. Change the type, or pick a shape that can be represented as data.
::

The short version: a warning is a drop you should know about and is usually fine, an error would throw at runtime so the build fails. The [validation guide](/guide/validation) covers the same contract from the type side.

## Reading a code

The letters tell you which feature produced the diagnostic, and the number tells you where it sits.

The **prefix** maps to the API:

- `VL` is `createValidate`, `VE` is `createGetValidationErrors`.
- `PJ`, `PJS`, `RJ`, `SJ` are the JSON family, `TB` and `FB` are the binary family.
- `MKR`, `CTA`, `PFN`, `TMP` cover the type markers and call-site shape.
- `PFE9` is `registerPureFnFactory`, `FMT` is `TypeFormat`, `OVR` is custom overrides.

The **number** follows a convention inside each family. `001` through `009` are root-position errors, the cases where the whole type has no data form. `010` and up are member-level warnings, the silent drops made visible.

A printed diagnostic looks like this:

```text
[VL010] Property `onClick` is a function — validate does not handle function values, so this property is silently not validated.
```

The code in brackets is what you search for on these pages.

## Families

Each family gets its own page, and they all share the warning and error contract above. Once you have read one the rest follow the same shape.

- [Validation](/diagnostics/validation): `createValidate` and `createGetValidationErrors`.
- Serialization: the JSON and binary families (`prepareForJson`, `restoreFromJson`, `toBinary`, and friends).
- Unknown keys: `hasUnknownKeys`, `stripUnknownKeys`, and the rest.
- Markers and call sites: the type markers, `CompTimeArgs`, Temporal types.
- Pure functions: the purity rules for `registerPureFnFactory`.
- Formats: `TypeFormat` pattern and sample constraints.
- Overrides: custom per-type function overrides.
