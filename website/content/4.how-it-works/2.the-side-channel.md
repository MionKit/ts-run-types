Because the door we'd use is gone. TypeScript 7 locked it. So we cut a new one.

## Why a Go binary

For years, type-first libraries had a clean trick. The old `tsc` was a Node program, so you could load a Node plugin into it, walk the syntax tree with the type checker right there, and generate runtime code before the types got erased. Typia, typescript-rtti, and friends all lived on this. It worked great.

Then TypeScript 7 rewrote the compiler in Go — one statically-linked binary called **tsgo**. About ten times faster, which is wonderful. But two things broke at once:

- **You can't load a Node plugin into a Go binary.** That's just not a thing.
- **The old custom-transformer API was never ported** — see [microsoft/typescript-go#516](https://github.com/microsoft/typescript-go/issues/516), no roadmap.

So the door type-first libraries walked through doesn't exist anymore. (The full story is on the [reflection gap page](/why/the-reflection-gap).)

That leaves two options. Wait for an official plugin API — timeline unknown, could be years. Or build a native side-channel into tsgo's checker. We did the second one: a separate Go process that reads the same type information tsgo has, and hands it back to our Vite plugin.

One thing this is **not**: a replacement for `tsc`. We don't type-check your code and we don't emit your JavaScript — Vite's normal pipeline still does all of that. The Go binary does exactly one job: answer "what type is this, right here?" It writes no `.js` of your app.

## Reading the tsgo checker

Here's the catch. The good stuff inside tsgo — the checker, the parser, all of it — lives under Go's `internal/` folders. Go enforces a hard rule: nobody outside the project can import an `internal/` package. The TypeScript team did that on purpose, so they can refactor freely.

So how do we get in?

Through a **shim** — a thin, publicly-importable wrapper that re-exports the symbols hiding inside `internal/`. We didn't invent it; we reuse the [`oxc-project/tsgolint`](https://github.com/oxc-project/tsgolint) shim, the same one oxc's TypeScript linter uses for the same reason. It mostly leans on Go type aliases (which carry a type's whole method set for free), with a sprinkle of link-time tricks for the few helpers that stay private.

Two things worth knowing:

- **It costs nothing at runtime.** Aliases and link-time wiring are resolved when the binary is built, not when it runs.
- **It's a temporary shape, not a hack we're stuck with.** When issue #516 lands an official API, we swap the shim for it. The marker, the caches, the Vite plugin, and your code all stay exactly as they are — only a few files deep in the binary change.

The key payoff: because we bootstrap from **your** tsconfig, the type we resolve as `User` is the *exact* type tsgo resolves. No second type system, no "close enough." What your editor thinks `User` is, the generator thinks too.

## The one tiny marker

We don't keep a hard-coded list of function names to look for. That would be brittle, and it'd stop you wrapping our helpers in your own. Instead, the whole thing pivots on one marker:

```ts
export type InjectRunTypeId<T> = string & {readonly __brand?: T};
```

Any function that wants compile-time type info just declares that as its **last parameter**:

```ts
function validate<T>(value: unknown, id?: InjectRunTypeId<T>): boolean {
  // the build fills in `id` at every call site
}

validate<User>(payload); // becomes validate<User>(payload, 'mNr4Vw')
```

The binary walks every call in your file, asks the checker for each call's resolved signature, and when the last parameter is `InjectRunTypeId<T>` with a concrete `T`, it records the site and generates for that `T`. Our own helpers — `getRunTypeId`, `reflectRunTypeId`, the whole `createX` family — use this marker. Your wrappers can too. Same machinery, no special-casing.

Two small but important details:

- **The marker is matched by name *and* by package.** It only counts if it's named `InjectRunTypeId` *and* declared in `@mionjs/ts-go-run-types`. Declare your own type called `InjectRunTypeId` somewhere else and nothing happens — no accidental rewrites.
- **A still-generic `T` is skipped.** Inside a generic function body, `T` isn't a real type yet, so there's nothing to generate. The injection happens later, at the places that actually call your wrapper with a concrete type.

That's the side-channel: a native peek at the real checker, gated by one tiny marker. Next, where all that generated code goes and why your bundle stays small — [caches and tree-shaking](/how-it-works/caches-and-tree-shaking).
