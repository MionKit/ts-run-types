# Caches & Tree-Shaking

A library that generates code for your types could easily bloat your bundle — generate everything, ship everything, hope the bundler sorts it out. This one does the opposite. The rule is simple:

> You ship only what you actually call.

Here's how that holds up.

## Demand-driven caches

The generator doesn't sit there producing every possible function for every type in your project. It only makes what your call sites ask for. We call that *demand-driven*.

Each kind of function — validators, JSON encoders, binary decoders, and so on — has its own cache, and that cache only fills with the types its own call sites requested. They don't bleed into each other.

So:

- Call `createValidate<User>()` and you get a validator for `User`. You do **not** get a JSON encoder for `User`, because you never asked for one.
- A file whose only type call is `getRunTypeId<User>()` ships **zero** function code. It wanted an id, it gets an id. No validator, no codec, nothing.

You pay for what you order. Nothing comes with a side of code you didn't want.

## One module per entry

Every cache entry is its own little ES module.

That's the trick that makes bundlers do the work for free. To your bundler, each generated function looks like a normal import from a normal module. So normal code-splitting and tree-shaking apply — no special plugin step, no custom dead-code pass. If nothing imports an entry, it's dropped, exactly like any other unused import.

When the plugin rewrites your call, it adds a small import for the entry that call needs, and drops a tiny id into the call itself. Your bundler takes it from there.

(There's one tuned exception: the reflection *data* for your types rides in a single shared bundle rather than one module per node, because there can be a lot of small nodes and one bundle splits better. It doesn't change the "ship only what you use" story — reflection data is still generated only for types you actually reflect on. If you'd rather split it anyway, that's what [`moduleMode`](/how-it-works/plugin-options) is for.)

## Deduped twice

Two call sites, same shape — do you ship that function twice? No.

Types get collapsed two different ways before anything is generated:

- **Same type object, two paths.** If the checker hands back the literal same type reached through different routes, it's one entry.
- **Two different types, identical shape.** `{ id: number }` written in two unrelated files still hashes to the same structural id, so it's one entry.

Either way, equivalent types fold into a single cache entry. Generate once, reuse everywhere. The output is also stable across builds, which keeps your diffs and your caches calm.

## What gets dropped

Putting it together, here's what *doesn't* end up in your bundle:

- **Functions you didn't call.** Demand-driven means uncalled is ungenerated.
- **Duplicate functions.** Double dedup means equivalent types share one entry.
- **Unreferenced entries.** One-module-per-entry means your bundler tree-shakes the leftovers.
- **The plugin's own dependencies.** There aren't any — `vite-plugin-runtypes` ships **zero runtime dependencies**. The Go binary is a build-time tool and never reaches the browser.

The end result: a file that reflects one id ships a few bytes, and a file that builds five validators ships five validators — not the whole library, not your whole type graph, just the work you actually asked for.

Want to change how entries are grouped into modules, or how they're shipped? That's what the [plugin options](/how-it-works/plugin-options) are for.
