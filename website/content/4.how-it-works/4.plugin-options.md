Here's the whole setup most people ever write:

```ts
// vite.config.ts
import {defineConfig} from 'vite';
import runtypes from 'vite-plugin-runtypes';

export default defineConfig({
  plugins: [
    runtypes({
      binary: '/abs/path/to/bin/ts-go-run-types',
    }),
  ],
});
```

That's it. Everything past `binary` is optional. The three knobs worth knowing are `moduleMode`, `inlineMode`, and `emitMode`.

## moduleMode — how entries group into modules

This decides how generated entries get packed into virtual modules. More modules means finer-grained tree-shaking; fewer modules means fewer requests in dev.

| Value | What it does | When to reach for it |
| --- | --- | --- |
| `default` | Reflection data rides one shared bundle; every function entry is its own module. | Leave it. Best chunk-splitting for production builds. |
| `allSingle` | Bundle everything — one module per function family, one bundle for pure fns, facades folded in. | Fewest modules and requests. Handy if a dev server is drowning in tiny modules. The trade: a family bundle re-fetches whole when a type in it changes. |
| `allModules` | Split everything, including one module per reflection node. | Escape hatch. Measurably slower on dense reflection graphs — only if you're chasing a specific splitting behaviour. |

Default: `default`.

## inlineMode — how children fold into parents

When a type contains other types, this decides which ones get their own reusable entry and which get inlined straight into their parent.

| Value | What it does | When to reach for it |
| --- | --- | --- |
| `default` | The name rule: unnamed shapes (arrays, tuples, object literals, unions, classes) inline into their parent; named types (interfaces, aliases) and circular types stay external so they can be shared and deduped. | Leave it. Good balance of dedupe and locality. |
| `allInternal` | Name-blind: everything except circular types inlines. One function per call-site type, per family. | If you'd rather duplicate shapes than chase shared entries — e.g. you want each call site fully self-contained. Costs bytes when shapes are shared across roots. |

Default: `default`.

::note
`moduleMode` and `inlineMode` never share a disk cache — the cache fingerprint folds both in, so switching modes can't read stale artifacts from another mode.
::

## emitMode — what each function entry ships

Generated functions can ship as a string of code that the runtime rebuilds, as a ready-made live function, or both. This only matters if your runtime has rules about running code it builds on the fly.

| Value | What it ships | When to reach for it |
| --- | --- | --- |
| `code` | Just the function body as a string; the runtime rebuilds it with `new Function` on first use. | The default. Smallest output. |
| `functions` | A live, ready-to-call function — no `new Function` at runtime. | Runtimes that ban `new Function`: Cloudflare Workers, sandboxed iframes, a strict CSP without `unsafe-eval`. |
| `both` | The body string *and* the live function. | Niche: a runtime that bans `new Function` yet also reads the `.code` string. Our own test suite uses this to exercise both paths. |

Default: `code`. If your bundle runs under a Content-Security-Policy without `unsafe-eval`, this is the one option you'll actually want to flip — set it to `functions`.

## A couple more, for completeness

- **`cacheDir`** — where the on-disk artifact cache lives. Defaults to `<root>/node_modules/.cache/ts-go-run-types` (the standard spot your `clean` scripts already wipe). Pass a string to relocate it, or `false` to disable caching entirely.
- **`tsconfig`** / **`cwd`** — point the binary at the right tsconfig and project root. Both default to sensible things (`tsconfig.json`, Vite's root).
- **`parallelScan`** / **`parallelRender`** — set either to `false` to force the binary's serial paths. Same output, just slower; these exist for benchmarking and debugging.

## Driving the binary directly

The plugin spawns the Go binary for you, so you rarely touch it. But it's a normal CLI, and it runs in two modes.

**One-shot** reads newline-delimited requests on stdin and writes the result to stdout — perfect for a CI codegen step or a quick poke:

```bash
bin/ts-go-run-types --one-shot --tsconfig tsconfig.json < requests.jsonl > cache.json
```

```json
{"op":"scanFiles","files":["src/app.ts"]}
{"op":"dump"}
```

**Daemon** keeps a process alive on a Unix socket so a long-running Vite/HMR session reuses one warm checker instead of re-spawning:

```bash
bin/ts-go-run-types --daemon --tsconfig tsconfig.json --socket /tmp/ts-go-run-types.sock
```

Every plugin option has a matching flag (`--module-mode`, `--inline-mode`, `--emit-mode`, `--cache-dir`, and friends), so anything you can set in `vite.config.ts` you can also pass on the command line.
