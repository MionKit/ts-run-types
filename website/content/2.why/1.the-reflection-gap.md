# The Reflection Gap

You wrote the types. You were careful with them. Then TypeScript compiled your code and threw every single one of them in the bin.

This isn't a bug. It's the whole design. And it's the hole this library exists to fill.

## TypeScript's original sin

TypeScript types are a compile-time fiction. They exist while the checker is looking at your code, and the moment it emits JavaScript they vanish without a trace. `User`, `Order`, that 40-line discriminated union you're so proud of — all gone. What ships is plain JS that has no idea any of it ever existed.

The TypeScript team is admirably upfront about this. "No runtime impact" is a stated **non-goal** — they will never add a feature that changes how your code runs. Types check your work and then get out of the way. Effectively, TypeScript decided it's a linter with great branding.

> Your types describe your program perfectly, right up until the program actually runs.

Which is fine, until your program meets the outside world. A JSON body off the wire. A row from the database. A config file. A message from another service. TypeScript will happily let you call it `User`, and at runtime it's whatever the network felt like sending. The one moment you'd actually like to know "is this really a `User`?", your types have already left the building.

## The reflection that never shipped

Plenty of languages keep type information around at runtime so you can ask questions about it. That's reflection. TypeScript has flirted with it for years — `emitDecoratorMetadata`, the long-running "types as values" conversations, every "can we please get runtime types" thread on the issue tracker.

None of it ever became a real, general runtime-reflection system. It was never going to, because shipping types to runtime is the exact thing TypeScript promised not to do.

So an entire ecosystem grew up around coping. Two strategies, broadly:

- **Write your types twice.** Describe the shape once as a schema object the runtime can see, and accept that your "real" type is now a second copy that drifts. (The whole [schema-first tax](/why/stop-writing-types-twice).)
- **Patch the compiler.** Hook into `tsc`'s custom-transformer API, read the real types from the checker, and generate runtime code before the types get erased. Clever. It worked.

That second trick was the good one. It read your actual TypeScript types — no second copy, no drift. Libraries like Typia and typescript-rtti built whole runtimes on it.

## Then TS7 pulled the ladder up

Then TypeScript 7 happened.

TS7 rewrites the compiler in Go — a single, statically-linked binary called **tsgo**. It's roughly ten times faster, which is genuinely great news. It also quietly demolished the compiler-patching trick.

The old `tsc` was a Node program, so a Node plugin could load itself in and walk the AST with the checker right there. tsgo is a Go binary. You can't load a Node plugin into a Go binary. And the legacy custom-transformer API was **not ported** — see [microsoft/typescript-go#516](https://github.com/microsoft/typescript-go/issues/516), which has no roadmap.

So the score is now:

- Schema-first libs still work — and you're still writing your types twice.
- Type-first libs that patched `tsc`? The thing they patched no longer exists.

> TypeScript erased your types. The community patched the compiler to get them back. Then TS7 deleted the compiler they were patching.

The entire type-first category needs a new way in. Not a monkey-patch on a binary that won't load plugins — a proper, native side-channel into tsgo's checker.
