## The schema-first tax

This is the schema-first deal: Zod, TypeBox, Valibot, AJV. You build a schema object the runtime *can* see, and your TypeScript type is whatever that schema infers.

It's a real, genuinely clever approach. It's also your type written backwards.

```ts
// The schema you write...
const User = z.object({
  id: z.number(),
  name: z.string(),
  email: z.string().email(),
});

// ...and the type you actually wanted, derived from it.
type User = z.infer<typeof User>;
```

Read that top-down and it's fine. Read it as "I had a `User` type in my head and had to re-encode it as a builder call to make it real," and the tax shows up. You're not describing a `User` anymore — you're describing a `User`-shaped construction kit, and hoping `z.infer` hands back what you meant.

Now scale it. Your schemas import each other, mirror your domain types, and live a parallel life next to your "actual" types. Two things that must agree forever:

- Add a field to the type, forget the schema — validation silently lets bad data through.
- Add a field to the schema, forget the type — the compiler nags you in a place unrelated to the bug.
- Refactor a union and now you're editing it in two languages at once: TypeScript, and Schema-Builder-ese.

> The schema **is** your type, written backwards. And two copies of one truth will drift. Not *might* — will.

There's a runtime cost too: schemas are objects you construct every time the module loads, sitting in memory, doing reflection-y work while your app runs. But honestly, the duplication is the part that wears you down.

## Type-first to the rescue (almost)

The other camp said the obvious thing: *why write the type twice — just read the one I already wrote.*

Type-first libraries (Typia, typescript-rtti) read your **real** TypeScript type straight from the compiler and generate the validator from it. One source of truth. No second copy, no drift. When it works, it's the dream.

It came with asterisks:

- **It patched `tsc`.** These libraries hooked the compiler's custom-transformer API to read types and emit code before erasure. Then TS7 shipped tsgo — a Go binary that loads no plugins and never ported that API ([the reflection gap](/why/the-reflection-gap)). The hook they relied on is gone.
- **Ahead-of-time only.** The transform runs in your build. No build step, no types — which is the right trade, but it *is* a trade.
- **Functions are a blind spot.** Typia, for one, can't reflect a function's signature — it reads the shape of data, not the shape of a callable.

So: the right idea, hobbled by *how* it had to reach the compiler.

## Why not just pick one?

The usual advice is "pick a lane." Type-first or schema-first, choose your trade-offs, live with them.

We thought that was a false choice, so we didn't make it.

**ts-run-types does both.** Write a plain TypeScript type and we read it natively — no patching `tsc`, because we don't go through `tsc` at all. We talk to tsgo's checker through a native side-channel, driven by one tiny marker. Prefer the builder feel? Reach for the `RT.*` schema builders and get the Zod-style ergonomics.

Here's the punchline: **both paths converge to the exact same cache entry.** A type and its equivalent schema resolve to one shared definition under the hood. No lock-in, no two-worlds problem — mix them in the same file if you like.

> Type-first when you want zero ceremony. Schema-first when you want the builder feel. Same result either way.

One source of truth. Pick the syntax, not the lane.
