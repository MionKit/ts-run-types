# What We're Not

Every library pitch is a highlight reel. Here's the part nobody shows you: where ts-run-types stops, what it deliberately won't do, and the trade you sign up for. None of this is a gotcha. It's the bill, and we'd rather you read it now than discover it at 2am.

## Not a plain `tsc` plugin

ts-run-types needs a build step. There's no magic import that makes types appear at runtime in a vanilla `tsc` setup — that's the very thing TypeScript [refuses to do](/why/the-reflection-gap).

What we actually need is a moment, during your build, to read your real types out of the checker and write the matching code. Today that moment is a **Vite plugin** that drives a native Go binary talking to tsgo. So:

- You build with Vite (or something that runs Vite plugins). No build pipeline, no generated functions.
- It runs **ahead of time**, in your build — not in a `tsc --watch` daydream and not at runtime. Same trade the type-first crowd always made; we're just honest about it.

The upside of paying that toll: there's nothing left to pay later. No schema objects to construct on import, no reflection cost while your app runs. The work happened at build time and your runtime [just looks things up](/how-it-works/overview).

> The build step isn't a tax we forgot to remove. It's where the entire trick happens.

## The serializable-data contract

This is the big one, so read it twice. Validators and serializers operate on **serializable data** — the JSON-shaped projection of your type — **not** the full TypeScript type.

Concretely: functions, methods, symbols, symbol-keyed properties, and getters/setters with no backing data get **dropped**. JSON was going to throw them off the wire anyway, so we validate and serialize the shape that actually survives a round-trip. For the typical job — RPC, persistence, network IO — that's the shape you meant.

The surprise lives in a corner. Take this:

```ts
interface User {
  name: string;
  onClick: () => void;
}
```

The generated validator only checks `name`. Pass `{ name: 'x', onClick: 'not-a-function' }` and it returns `true` — the `onClick` field simply isn't part of the contract. That's intended, but if you didn't expect it, it looks like a bug.

So we tell you. At build time. Loudly, but proportionately:

- **Warning = an expected drop.** A non-serializable *property* vanishes and we log it (VL010 and friends). Fine. Carry on. Do **not** treat it as an error.
- **Error = this would throw at runtime, so the build stops.** When a non-serializable type shows up somewhere it can't be quietly dropped — a root value, an array element, a tuple slot, a union member, a function param or return — we won't pretend. We emit a factory that throws, with an Error-severity diagnostic that **halts the build**.

> Warning means "we dropped something harmless, FYI." Error means "this will explode at runtime, fix it now." We never blur the two.

One consequence worth flagging: **decoders hand back the data-only projection.** `createJsonDecoder<T>()` and `createBinaryDecoder<T>()` return `DataOnly<T>`, not bare `T`. A value rebuilt from JSON or bytes can only ever hold data — promising you methods it doesn't have would be a lie the type system lets you call. On a clean DTO, `DataOnly<T>` is just `T`, so you won't notice. The full rules live in [the validate contract](/guide/validate-contract).

## Known limitations

A few more honest edges, so nothing ambushes you:

- **The type graph has a footprint.** Not every exotic corner of TypeScript maps onto a runtime check. Builtin classes (`Date`, `Map`, `Set`, `RegExp`, `Temporal`) are recognized atomically, not walked field by field. The full list of what's covered — and what we deliberately flatten — is in [type support](/reference/type-support).
- **It's experimental, and it tracks the bleeding edge.** ts-run-types rides on tsgo, which is TypeScript 7, which isn't GA yet. We track it closely; we also move when it moves. Treat this as early days.
- **It's a side-channel, not a fork of TypeScript.** We read the checker; we don't change the language. If TypeScript can't express a type, neither can we.

None of these is a dealbreaker for the thing this library is *for*: knowing, at runtime, whether the data in your hands really is the type you said it was. We'd just rather you choose it with eyes open.
