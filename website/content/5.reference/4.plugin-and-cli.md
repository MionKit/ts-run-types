## Vite plugin options

```ts
import runtypes from 'vite-plugin-runtypes';

export default defineConfig({
  plugins: [runtypes({ binary: './bin/ts-go-run-types' })],
});
```

| Option | Type | Default | What it does |
|--------|------|---------|--------------|
| `binary` | `string` | — *(required)* | Absolute path to the compiled `ts-go-run-types` binary. |
| `cwd` | `string` | Vite's `root` | Project root — where `tsconfig.json` lives. |
| `tsconfig` | `string` | `"tsconfig.json"` | Path to tsconfig, relative to `cwd`. |
| `emitMode` | `'code' \| 'functions' \| 'both'` | `'code'` | What each fn entry ships. `code`: body string, rebuilt via `new Function`. `functions`: live closure (for CSP / no-`eval` runtimes like Cloudflare WorkerD). `both`: both. |
| `cacheDir` | `string \| false` | `node_modules/.cache/ts-go-run-types` | On-disk artifact cache. `false` disables it entirely. |
| `moduleMode` | `'default' \| 'allSingle' \| 'allModules'` | `'default'` | How cache entries group into virtual modules. `default`: runtype data bundle + per-entry fn modules (best chunk-splitting). `allSingle`: one module per family (fewest requests). `allModules`: split everything (escape hatch). |
| `inlineMode` | `'default' \| 'allInternal'` | `'default'` | Child-inlining policy. `default`: unnamed compounds inline, named/circular types stay external & shared. `allInternal`: everything except circular types inlines. |
| `parallelScan` | `boolean` | `true` | Set `false` to force the serial marker scan (`--no-parallel-scan`). For benchmarking / debugging. |
| `parallelRender` | `boolean` | `true` | Set `false` to force serial cache rendering (`--no-parallel-render`). For benchmarking / debugging. |

::note
`moduleMode` and `inlineMode` never share disk caches — the cache fingerprint folds both in, plus `emitMode`, so switching modes never cross-reads stale artifacts.
::

## CLI flags

The binary runs in **one-shot** (stdio JSON) or **daemon** (Unix socket, used by Vite/HMR) mode.

```bash
# one-shot
bin/ts-go-run-types --one-shot --tsconfig tsconfig.json < requests.jsonl > cache.json

# daemon
bin/ts-go-run-types --daemon --tsconfig tsconfig.json --socket /tmp/ts-go-run-types.sock
```

### Core

| Flag | Default | What it does |
|------|---------|--------------|
| `--one-shot` | — | Read requests from stdin until EOF, emit the dump to stdout. |
| `--daemon` | — | Listen on a Unix socket for persistent serving. |
| `--socket PATH` | `/tmp/ts-go-run-types.sock` | Socket path (daemon mode). |
| `--tsconfig PATH` | `./tsconfig.json` | tsconfig to load. |
| `--cwd PATH` | `$PWD` | Working directory. |
| `-h`, `--help` | — | Show help. |

### Output (one-shot)

| Flag | Default | What it does |
|------|---------|--------------|
| `--out-json PATH` | — | After stdin drains, write the cache as JSON to `PATH`. |
| `--out-modules DIR` | — | After stdin drains, write every per-entry virtual module to `DIR`. |

### Behaviour

| Flag | Default | What it does |
|------|---------|--------------|
| `--emit-mode MODE` | `code` | `code` \| `functions` \| `both` — fn-entry code/factory slots (mirrors the plugin's `emitMode`). |
| `--module-mode MODE` | `default` | `default` \| `allSingle` \| `allModules` — virtual-module grouping (mirrors `moduleMode`). |
| `--inline-mode MODE` | `default` | `default` \| `allInternal` — child-inlining policy (mirrors `inlineMode`). |
| `--cache-dir PATH` | — | Base directory for the on-disk RT artifact cache. |
| `--hash-length N` | `7` | Short-id length for type hashes. |

### Performance / debugging

| Flag | Default | What it does |
|------|---------|--------------|
| `--no-parallel-scan` | off | Disable the parallel marker scan. |
| `--no-parallel-render` | off | Disable parallel cache renders. |
| `--single-threaded` | off | Force single-checker mode (useful for tests). |
| `--inline-sources-stdin` | off | Read `{"sources":{relpath:content}}` from stdin (no tsconfig glob, no disk reads). |
| `--inline-server` | off | Persistent inline-sources server (for long-lived test daemons). |
| `--pprof-cpu PATH` / `--pprof-heap PATH` | — | Write Go pprof profiles. |

::note
The marker set is fixed — there are no `--marker-name` / `--marker-module` flags (retired in the marker migration). To use a custom marker you embed the Go resolver directly and pass `marker.Options{Specs: […]}`.
::
