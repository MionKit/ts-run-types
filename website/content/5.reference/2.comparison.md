## Approach comparison

### Schema-first (Zod, Valibot, ArkType, AJV, TypeBox)

You describe the shape with a builder API, then derive the TypeScript type from it:

```ts
const User = z.object({ id: z.number(), name: z.string() });
type User = z.infer<typeof User>;
```

**Pros:** no build step, runs everywhere, the schema is a value you can pass around.
**Cons:** the schema and any hand-written type can drift; the schema object exists at runtime even when you only needed a yes/no check.

### Type-first (ts-run-types, Typia)

The TypeScript type *is* the source of truth тАФ there's nothing to keep in sync:

```ts
interface User { id: number; name: string }
const isUser = createValidate<User>();
```

**Pros:** one source of truth, zero duplication, the validator is a specialized function generated at build time (no schema object to construct at runtime).
**Cons:** needs the compiler integration (the Vite plugin + Go resolver).

### Best of both

ts-run-types also ships the `RT.*` schema builders, so you can author schema-first when you like the Zod/TypeBox feel тАФ both styles compile to the *same* validator and can be mixed in one file. See [the homepage](/) for the side-by-side.

## The function-reflection edge

The thing that sets ts-run-types apart even from Typia (the other type-first library) is **function reflection**. Typia cannot extract a type *from* a function signature тАФ you must declare the type separately and pass it as a generic. ts-run-types reads the signature directly:

```ts
// Reflect a function value тАФ params and return type are inferred:
const id = reflectRunTypeId(myHandler);

// Or pull a type straight out of a signature with the std-lib utilities:
const isReturn = createValidate<ReturnType<typeof myHandler>>();
const areArgs  = createValidate<Parameters<typeof myHandler>>();
```

This is what lets a router automatically validate a handler's parameters and return value with no separate type declarations, no generic parameters, and no risk of the types drifting out of sync with the handler.

## When to use ts-run-types

::stylish-list{type=check}
- TypeScript types should be the single source of truth тАФ no schema duplication.
- You need JSON serialization that round-trips `Date`, `BigInt`, `Map`, `Set`, `RegExp` or `Temporal`.
- You need compact, fast **binary** serialization.
- You want **mock data** generated straight from your types.
- You want validators that **tree-shake to the bone** тАФ only the functions you call ship.
- You're reflecting over **function signatures** (RPC, routers, codegen).
::

## Consider an alternative when

::stylish-list{type=info}
- You need to share schemas with **non-TypeScript** consumers тАФ reach for **AJV** / **TypeBox** (JSON Schema) or **Zod** (schema as a value).
- You're shipping a library consumed by **vanilla JavaScript**, or you can't add a build step тАФ **Zod**, **Valibot** or **ArkType** run with no toolchain.
- You define schemas **at runtime** (dynamic shapes that don't exist as TypeScript types) тАФ schema-first libraries are the right tool.
- Bundle size in a no-build context is paramount тАФ **Valibot** is built around tree-shakeable, modular validators.
::
