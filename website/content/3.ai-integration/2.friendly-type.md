`FriendlyType<T>` is a combined, human-readable map for a type: a **label** and a set of **error-message templates** per field. It's pure data ‚Äî authored once, [committed](/ai-integration/workflow-and-commands) next to the type's definition, and validated against the type on every build. Today it renders validation errors into readable messages; later it powers form-building UI.

## The node model

One recursive node. Every node is `{ $label?, $errors?, ...children }` ‚Äî `$`-prefixed keys are meta, every other key is a child field. Leaf nodes simply have no children, so nesting is uniform with no `fields:` wrapper.

```ts
import type * as TF from 'ts-runtypes/formats';

// models/user.ts
export interface User {
  name: TF.String<{ minLength: 2; maxLength: 60 }>;
  age: TF.Number<{ min: 0; max: 120 }>;
  isActive: boolean;
  tags: string[];
  profile: {
    email: TF.Email;
    score: TF.Number<{ min: 0; max: 100 }>;
  };
}
```

The authored map mirrors the shape of `User`. It is validated against `User` at scan time ‚Äî a stale key or a structural mismatch is a build-time Error:

```ts
import type { FriendlyType } from 'ts-runtypes';
import type { User } from './user';

export const userFriendly: FriendlyType<User> = {
  $label: 'User account',

  name: {
    $label: 'Full name',
    $errors: {
      type: '$[label] must be text',
      minLength: '$[label] needs at least $[val] characters',
      maxLength: '$[label] allows at most $[val] characters',
    },
  },
  age: {
    $label: 'Age',
    $errors: {
      type: '$[label] must be a number',
      min: '$[label] must be at least $[val]',
      max: '$[label] must be no more than $[val]',
    },
  },
  isActive: { $label: 'Active?' },

  tags: {
    $label: 'Tags',
    $items: { $errors: { type: 'each tag must be text' } },   // element node
  },

  profile: {                          // nested object ‚Äî same node shape, recursively
    $label: 'Profile',
    email: { $label: 'Email', $errors: { pattern: 'Enter a valid email address' } },
    score: { $label: 'Score', $errors: { min: 'min $[val]', max: 'max $[val]' } },
  },
};
```

Container meta keys follow the type's shape: arrays and tuples use `$items` for the element node; objects nest as the same node, recursively.

## Error keys = the failed constraint

Each `$errors` key names the failed sub-constraint. This isn't an invented key set ‚Äî it maps **1:1 onto what `createGetValidationErrors<T>()` actually emits**. A failure carries a `format` detail, and the `$errors` key is the **tail of that failure's `formatPath`**; `type` is the base type-shape failure (the value wasn't even the right kind).

| Failure | `$errors` key | `$[val]` resolves to |
| --- | --- | --- |
| base type-shape (wrong kind) | `type` | ‚Äî |
| string `minLength` / `maxLength` | `minLength` / `maxLength` | the bound (`2`, `60`) |
| string `pattern` | `pattern` | ‚Äî |
| number `min` / `max` | `min` / `max` | the bound |
| number `lt` / `gt` | `lt` / `gt` | the bound |
| number `integer` | `integer` | `true` |
| date/time `date` / `time` | `date` / `time` | ‚Äî |
| `uuid` version | `version` | `'4'` |

**Constraint granularity is bounded by the type.** A bare `name: string` can only fail as `type`; you get `minLength` / `maxLength` keys only because the field is `TF.String<{ minLength; maxLength }>`. The richness of the friendly map is a function of how richly the type is annotated.

::note
Errors **accumulate**. A value that violates both `minLength` and `pattern` produces two failures, not one ‚Äî so the data form yields **one message per violated constraint** (a list). To join them into a single sentence, reach for the function escape hatch below.
::

## The placeholder DSL

Templates are plain strings with `$[‚Ä¶]` tokens the renderer substitutes (and the compiler validates):

| Token | Resolves to |
| --- | --- |
| `$[label]` | the node's `$label`, falling back to the raw field name |
| `$[val]` | the failed constraint's bound (e.g. `2` for `minLength: 2`) |
| `$[path]` | the dotted path to the field (`profile.email`) |
| `$[index]` | the array element index, for `$items` failures |

::note
`$[value]` ‚Äî the *actual received value* ‚Äî is out of scope for v1: the error carries no input value, so threading it into the renderer is deferred.
::

## The function escape hatch

Any `$errors` entry may be an **inline arrow** instead of a template record, for logic the data form can't express ‚Äî joining constraints, pluralization, i18n lookups. It receives a synthesized `failed` bag (this field's failures, grouped and keyed by the same constraint names), so one call yields one message per field:

```ts
name: {
  $label: 'Full name',
  $errors: (failed) => {
    const parts: string[] = [];
    if (failed.minLength) parts.push(`at least ${failed.minLength.val} characters`);
    if (failed.maxLength) parts.push(`at most ${failed.maxLength.val} characters`);
    return parts.length ? `Name must be ${parts.join(' and ')}` : 'Invalid name';
  },
},
```

The function must be an **inline** expression (no external reference); its body is opaque to the compiler and runs at runtime, so it can call i18n machinery freely. The trade: the data form gets compile-time placeholder/constraint validation; a function doesn't.

## Rendering at runtime ‚Äî `createFriendly`

Error rendering needs only `(map, errors)` ‚Äî no type id, no registry lookup. `createFriendly<T>(map)` returns a renderer with two methods:

```ts
import { createFriendly } from 'ts-runtypes';
import { userFriendly } from './models/user.rt';

const friendly = createFriendly<User>(userFriendly);

// label(path) ‚Äî dotted string or a raw path-segment array
friendly.label('profile.email');
// ‚Üí 'Email'

// errors(errs) ‚Äî render a createGetValidationErrors result into messages
friendly.errors(getUserErrors(badInput));
// ‚Üí [{ path: 'profile.email', label: 'Email', message: 'Enter a valid email address' }]
```

`errors()` groups failures by path, walks each `path` into the map, picks the template by the failed constraint, and interpolates the `$[‚Ä¶]` tokens. Each result is a `FriendlyMessage`:

```ts
interface FriendlyMessage {
  path: string;     // dotted path to the field ('profile.email'); '' for the root
  label: string;    // the field's $label, or its raw last path segment as fallback
  message: string;  // the interpolated message
}
```

A data-form `$errors` yields one message per violated constraint; a function-form `$errors` yields one message per field. When a field has no map entry, the renderer falls back gracefully ‚Äî the raw field name as the label, and a generic "is invalid" message.

::note
Error rendering is pure data and works today. **Form-building UI is deferred** ‚Äî enumerating every field of `T` (labelling the mapped ones, falling back to raw names for the rest, in declaration order) needs the reflection node, which arrives through the registry accessors on the [workflow page's](/ai-integration/workflow-and-commands) v2 roadmap.
::
