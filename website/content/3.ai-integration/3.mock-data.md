`MockData<T>` is the realistic-sample enrichment map for a type: per-field **value pools**, **ranges**, and element/length/optionality hints. It feeds the existing [`createMockType<T>()`](/guide/mocking) generator — the AI supplies believable values, the mechanical generator stays deterministic. Like [`FriendlyType<T>`](/ai-integration/friendly-type) it's pure data, [committed](/ai-integration/workflow-and-commands) next to the type's definition, and validated against the type on every build.

`createMockType` already supports custom pools and per-property overrides; `MockData<T>` is the typed, validated form of those.

## The node model

The map mirrors the shape of `T`. Each field carries the knobs that make sense for its kind:

```ts
import type { MockData } from 'ts-runtypes';
import type { User } from './user';

export const userMock: MockData<User> = {
  name: { pool: ['Alice Martin', 'Liang Wei', 'Fatima Noor', /* …50+ */] },
  age: { min: 18, max: 95 },
  tags: { $items: { pool: ['urgent', 'beta', 'vip'] }, $length: [1, 4] },
  profile: {
    email: { pool: ['alice@example.com', 'liang@corp.io', /* … */] },
    score: { min: 0, max: 100 },
  },
};
```

The per-field knobs, by kind:

| Knob | Applies to | What it does |
| --- | --- | --- |
| `pool` | strings, numbers, booleans, bigints, … | draw the value from this list |
| `min` / `max` | numbers, dates | bound the generated range |
| `$items` | arrays, tuples | the element node (a `MockData` node for `T[number]`) |
| `$length` | arrays | fixed length, or a `[min, max]` range |
| `$optional` | objects | present-probability (0–1) for optional members |

Then feed it to the generator through the `data` option:

```ts
import { createMockType } from 'ts-runtypes';
import { userMock } from './models/user.rt';

const mockUser = createMockType<User>({ data: userMock });

mockUser();
// → { name: 'Liang Wei', age: 41, isActive: true,
//     tags: ['beta', 'vip'],
//     profile: { email: 'liang@corp.io', score: 73 } }
```

::note
The `data` option threads each field's `MockData` node into the existing mock walker — strictly additive: omit it and the generator behaves byte-for-byte as it does today. The `MockData<T>` DSL type and the `createMockType<T>({ data })` integration are shipped today; the build-time pool-value validation (MD003) is still in development.
::

## The pool-validation superpower (MD003)

Because RunTypes can **validate**, the compiler checks that **every pool and range value actually satisfies its field's type and format** — at parse time, not at test runtime. An LLM that hallucinates a malformed address into the `email` pool, or a `score` of `150`, is caught by the build:

```ts
export const userMock: MockData<User> = {
  profile: {
    email: { pool: ['alice@example.com', 'not-an-email'] },   // MD003 Error — 'not-an-email' fails TF.Email
    score: { min: 0, max: 150 },                              // MD003 Error — 150 exceeds the field's max: 100
  },
};
```

No other mock library can do this — it falls straight out of the existing validator. (`min > max` or an inverted `$length` is a separate `MD004` Warning.)

## Format-aware by default

Even without a pool, the generator already respects [type formats](/guide/type-formats) — an `email` field gets a real-looking address, a `uuidv4` field a real-looking UUID. `MockData<T>` is for when *mechanical-but-valid* isn't enough and you want *believable*: real names in `name`, your own domains in `email`, realistic ranges in `age`. Anything you don't pin falls back to the format-aware mechanical generator.

::note
Mock pools tend to be large (50+ items) and almost never belong in a production bundle. Reach for `MockData<T>` in tests and seed scripts — the same place `createMockType` itself lives. The future registry path gates mock registration to dev/test for exactly this reason (see the [workflow page](/ai-integration/workflow-and-commands)).
::
