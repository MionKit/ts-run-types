Everything RunTypes emits by default — validators, JSON/binary codecs, the reflection bundle — is a pure function of your type: regenerated every build, never committed, correct by construction. There's no "sync" problem because the artifact *is* the type, recomputed.

**Enrichment is a different species.** [`FriendlyType<T>`](/ai-integration/friendly-type) (human labels + error messages) and [`MockData<T>`](/ai-integration/mock-data) (realistic sample pools) carry content a generator can't invent — a good label, a clear message, a believable name. So they're authored *once* (with an LLM's help), **committed** to your repo, and validated against the type forever after.

| | Generated cache (today) | Enrichment |
| --- | --- | --- |
| Determinism | `f(type)` — deterministic | An LLM authored it |
| Content | Glue (validators, codecs) | The value *is* the content |
| Cost | Cheap — recompute every build | Expensive (an LLM call) — generate once |
| Storage | Ephemeral, gitignored | **Committed** to the repo |
| Drift | Impossible | **Possible** — the type can change under it |

These are **satellite artifacts keyed by a type**. They are *not* a code-emit family like `validate` / `json` / `binary` — there's no Go emitter and no runtime codegen. The compiler's only jobs are to **validate** the authored literal against the live type, and later route it through the registry by id.

## Validation rides the normal build

The marker *is* the type annotation. The scanner already parses every file for markers and already resolves each type's graph — recognizing a declaration typed `FriendlyType<T>` or `MockData<T>` is one more marker arm. It walks the object literal against the resolved type and emits diagnostics on the same channel as today's validation warnings, so they surface in Vite/HMR exactly like `VL0xx` codes. No separate watcher, no separate pass.

The diagnostics catch the two failure modes enrichment introduces — a stale key (a field was renamed or removed) and a value that no longer fits its field:

| | Friendly | Mock |
| --- | --- | --- |
| Key isn't a field of `T` | `FT002` Error | `MD001` Error |
| Structural mismatch (object where `T` is scalar, or vice-versa) | `FT004` Error | `MD002` Error |
| A pool / range value **fails validation** | — | `MD003` Error |
| Field has no label (renders the raw name) | `FT001` Info | — |
| Structural drift since authored | `FT010` Info | `MD010` Info |

::note
Validation is **always-on during the build**. The CLI verbs below are for CI and for agents driving generation out-of-band — they run the same checks standalone.
::

## The agent loop — the compiler as a tool

Generation is an explicit, opt-in action — **never an LLM call inside a build**. Builds stay pure and deterministic; any LLM-backed authoring happens out-of-band, writes a reviewable diff, and is committed and human-reviewed.

The loop is two CLI ops the agent drives:

```
describe ./models/user.ts#User  ──►  agent writes/patches user.rt.ts  ──►  check --file user.rt.ts --json
        ▲                                                                          │
        └──────────────────────────  loop until clean  ◄────────────────────────────┘
                                            │
                                 human reviews the diff, commits
```

`describe` is the prompt context; `check --file --json` is the ground-truth correctness check. The Go binary already runs as a long-lived process under the plugin, so `check --file` is a cheap incremental op — exactly what you'd expose as **MCP tools** so any agent (Claude Code, Cursor, your own) can drive generation, vendor-neutral.

## CLI surface

| Command | Purpose |
| --- | --- |
| `ts-runtypes check [glob]` | Run FT/MD validation standalone; non-zero exit on an Error. CI / pre-commit. |
| `ts-runtypes check --file <p> --json` | Validate **one file**, structured JSON out — the agent's tight feedback tool. |
| `ts-runtypes describe <file>#<Type> --format prompt\|json` | Emit the type's shape (names, kinds, optionality, formats, literals) as LLM prompt context. |
| `ts-runtypes gen <file> [--mock] [--friendly] [--check]` | Generate / refresh the `.rt.ts` sibling (below). |

::note
The CLI commands above (`describe` / `check` / `gen`) and the always-on build-time diagnostics are in active development and describe the intended workflow as the design specifies it. Shipped today: the `FriendlyType<T>` / `MockData<T>` DSL types (type-checked against `T` by your own compiler), the pure-data [`createFriendly`](/ai-integration/friendly-type) renderer, and the `createMockType<T>({ data })` integration.
::

## The `.rt.ts` sibling — anchored at the definition

Enrichment is stored in a committed sibling of the file where the **type is defined** — not where it's consumed. This mirrors the cache's core rule (one canonical entry per structural id, app-wide, demand discovered at scattered call sites): the `.rt.ts` sibling is the committed analog — **one enrichment home per type, at its definition**, however many files consume it.

```
src/models/user.ts        interface User { … }              ← definition
src/services/userApi.ts   createMockType<User>()            ← consumer
src/test/fixtures.ts      createMockType<User>()            ← consumer
                          ──────────────────────────────────
gen ⇒  src/models/user.rt.ts   export const userMock: MockData<User> = { … }
                               (ONE sibling, at the definition — not at the call sites)
```

Demand is discovered **globally** (scan every call site), but output lands at the **definition**: for each demanded named type, resolve it to its declaration's source file (following re-exports to the original) and write or refresh `<file>.rt.ts`. One `.rt.ts` per source file holds one export per enriched type defined there:

```ts
// src/models/user.rt.ts — GENERATED, COMMITTED, hand-editable.  rt-id: 9f3a (User structural hash)
import type { User, Post } from './user';

export const userMock: MockData<User> = { /* … */ };
export const userFriendly: FriendlyType<User> = { /* … */ };
export const postMock: MockData<Post> = { /* … */ };
```

This is the **first committed RunTypes artifact** — every other output is gitignored cache. `gen` is **create-only**: it appends missing entries and never clobbers existing ones, because these files are hand-editable and AI-authored.

::note
What gets scaffolded differs by artifact. `FriendlyType<T>` is generated for **any** type referenced by **any** RunTypes marker (it also serves a future form-building UI, so it scaffolds eagerly). `MockData<T>` is generated only for types consumed by a **`createMockType`** call — demand-driven by the mock consumer.
::

## External / library types — a resolution order

A type defined in `node_modules` can't get a sibling there (the directory is ephemeral and wiped on install), and a dependency might **already ship** enrichment for its own types — so we never blindly generate over it. Enrichment for a type resolves by precedence, **first match wins**:

1. **`T` defined in your source** → its definition-file sibling `*.rt.ts`. *(generated by `gen`)*
2. **`T` from a dependency that ships enrichment** → the library's `*.rt.ts` / named enrichment exports. *(read-only, vendored — we consume, never generate)*
3. **`T` from a dependency, user opted in** via a `@rtEnrich` JSDoc tag → a user-authored override in a configured dir (`rt-overrides/`). *(opt-in only)*
4. **No match** → emit nothing; the factories fall back — mock to the mechanical `createMockType`, friendly to raw field names.

So **by default we emit nothing for external types** — they land at #4 unless the library provides (#2) or you explicitly opt in (#3). Built-ins (`Date`, `Map`, …) are #4 by nature; `createMockType` already mocks them mechanically.

This makes enrichment **part of a library's public API**: a library that wants to support it adds `ts-runtypes` (for the `FriendlyType` / `MockData` types) and ships `customer.rt.ts` next to `customer.d.ts` — the resolver finds it by the same sibling convention, no special-casing. The opt-in tag wants an anchor; the cleanest is a tagged local re-export, which gives both the signal and a stable import:

```ts
/** @rtEnrich */
export type { Customer } from 'stripe';
```
