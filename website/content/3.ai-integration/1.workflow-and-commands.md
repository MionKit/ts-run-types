Most of what RunTypes makes — validators, JSON and binary codecs, mock data — it works out from your type automatically, every build. But a few things can't be guessed: a clear field label, a friendly error message, a believable sample name. Those have to be *written*.

That's where an AI agent comes in. **The agent writes them; the compiler checks them.** You commit the result next to your code, and from then on the compiler keeps it honest — if your type changes, it tells you exactly what no longer fits.

## How it works

It's a simple loop between the agent and the compiler:

1. **Describe** — the agent asks the compiler what a type looks like: its fields, their kinds, any format rules. That's the agent's context.
2. **Write** — the agent writes the labels, messages, or sample data into a committed file that lives next to your code.
3. **Check** — the compiler validates what the agent wrote against the real type and reports anything wrong. The agent fixes it and checks again, until it's clean.
4. **Review & commit** — you review the diff like any other change, and commit it.

```
describe a type  ──►  agent writes the data  ──►  check against the type
       ▲                                                   │
       └────────────────  fix and repeat  ◄────────────────┘
                                  │
                        you review and commit
```

The compiler never calls an AI model during a build — builds stay fast and predictable. Writing the data is a separate, opt-in step the agent runs on its own, and the result is always a reviewable, committed diff.

## The compiler keeps it in sync

Because this data lives alongside a type that can change, the compiler watches for two things and flags them as you work:

- a label or field that **no longer exists** on the type (it was renamed or removed), and
- a value that **no longer fits** its field — say, a sample that stopped being a valid email.

When a type changes you don't regenerate the whole file. The compiler updates only what changed and keeps everything you (or the agent) hand-tuned.

## Commands

A small CLI lets an agent — or your CI — drive the loop. The same checks also run automatically during your normal build.

| Command | What it does |
| --- | --- |
| `describe <file> <Type>` | Print a type's shape as context for the agent. |
| `check [files]` | Validate the authored data against your types. Fails on a real error — good for CI and pre-commit. |
| `gen <file> <Type>` | Scaffold a starting file for a type (committed, hand-editable). |
| `gen … --update` | Re-sync a file after a type change, keeping your hand-written values. |
| `gen … --prune` | Clean up entries left behind by deleted types or fields. |

These map cleanly onto **MCP tools**, so any agent — Claude Code, Cursor, your own — can drive the whole thing.
