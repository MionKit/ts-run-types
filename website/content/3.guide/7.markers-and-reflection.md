# Markers & Reflection

TypeScript erased your types, so it can't tell you anything about them at runtime. ts-run-types reads them at build time and gives you back the one thing the compiler refused to: **reflection** — a stable id for any type, plus a way to wrap that machinery into your own helpers.

This page is the consumer-important one. There's really just a single marker you'll type by hand: `InjectRunTypeId<T>`.

## A stable id for any type

Two functions, same job — get the id of a type:

- `getRunTypeId<T>()` — static. You bring the type, you get its id. No value needed.
- `reflectRunTypeId(value)` — reflection. `T` is inferred from a value. The value is only read for its type; at runtime it's ignored.

<code-import path="packages/examples/src/guide/markers-reflection.ts" lang="ts" />

The id is a short, stable hash of the type's shape. Two types with the same shape resolve to the **same** id — `getRunTypeId<{id: number}>()` and a value of that shape land on the same string. That's the whole point: a structural fingerprint you can use as a cache key, a registry key, a discriminator, whatever you need.

::note
These are build-time calls. With the Vite plugin off, there's no id to inject — `getRunTypeId` throws rather than hand you a wrong answer. [Here's how the build fills it in.](/how-it-works/overview)
::

## InjectRunTypeId — the one you'll type

Here's the star. Every `createX` factory in ts-run-types — `createValidate`, `createJsonEncoder`, all of them — works because of one trick: a trailing parameter branded `InjectRunTypeId<T>` that the build fills in at each call site.

The good news: **you can use the exact same trick.** Declare a trailing `id?: InjectRunTypeId<T>` on your own generic function and it opts into the toolchain. The build injects the resolved id at every call site — you never pass it yourself.

<code-import path="packages/examples/src/guide/markers-wrap-helper.ts" lang="ts" />

That's it. `describe<{id: number}>()` looks like a normal generic call, but the build quietly fills the `id` slot with the type's hash. Inside, `id` is just that string — look it up, branch on it, do whatever your helper needs.

::tip
The marker is the public extension point. If you're tempted to ask "can ts-run-types do X?", the answer is often "wrap it yourself with `InjectRunTypeId<T>`" — that's exactly how the built-in factories are built.
::

### Why your own `InjectRunTypeId` won't fire

The toolchain matches the marker by **name AND declaring module** (`@mionjs/ts-go-run-types`). So if you happen to have your own `type InjectRunTypeId<T>` declared somewhere else, it's inert — nothing gets injected. No spooky action at a distance.

<code-import path="packages/examples/src/guide/markers-not-triggered.ts" lang="ts" />

::warning
A marker only fires when `T` is concretely known at the call site. Inside a generic body, where `T` is still your function's own type variable, there's nothing to resolve — so the build skips it. Propagate by declaring `id?: InjectRunTypeId<T>` on your function and let the injection happen at *its* call sites (which is what the wrapper examples above do).
::

## The rest of the marker family (FYI)

You almost certainly won't type these — the `createX` factories use them for you — but here's the cast, so nothing's a mystery:

- **`InjectTypeFnArgs<T, Fn>`** — `InjectRunTypeId`'s cousin for the factories. It injects a `[typeId, fnId]` pair so the build emits only the function you actually asked for. Every `createX` already declares it; you don't.
- **`CompTimeArgs<T>`** — brands a parameter as "must be a literal at the call site" (or a `const` of literals). It's how options bags get read at build time. Static check only.
- **`CompTimeFnArgs<T>`** — like `CompTimeArgs`, but the literal also *selects* which variant of a factory you get (e.g. the JSON encoder strategy). Static check only.
- **`PureFunction<F>`** — brands a function argument as "inline definition that passes the purity rules". Used for [pure functions](/guide/custom-serializers-and-pure-fns).

That's the whole family. One you'll type, four that quietly do their job.
