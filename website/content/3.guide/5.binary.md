# Binary Encode / Decode

JSON is great until you're sending a lot of it. Field names repeat on every record, numbers become strings, and it all adds up. `createBinaryEncoder<T>()` and `createBinaryDecoder<T>()` pack your value into a compact binary buffer instead — no field names on the wire, numbers stored as numbers.

The encoder returns an `ArrayBuffer`; the decoder takes one back. And like JSON, the codec knows your type — so a `Date` round-trips cleanly instead of getting lost.

  icon: i-lucide-braces

  A human or a third party reads the payload, you need it `curl`-able, or you're talking to something you don't control. Readability wins.
  ::::
::

## Reuse the buffer in hot loops

By default each encode allocates a fresh backing buffer. In a tight loop that adds up — so build a `DataViewSerializer` once with `createDataViewSerializer` and hand it to the encoder, and it'll reuse the same buffer. `createDataViewDeserializer` does the same on the way back.

## The decoder returns `DataOnly<T>`

Just like JSON, `createBinaryDecoder<T>()` returns `DataOnly<T>` — the serializable projection of your type. A value rebuilt from bytes can't carry methods or symbols, so the return type leaves them out. See the [Validate contract](/guide/validate-contract) for the full reasoning.
