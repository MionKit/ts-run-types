# JSON Encode / Decode

`JSON.stringify` has a dirty secret: it can't round-trip half the things you put in an object. A `Date` goes out as a string and never comes back. A `Map` turns into `{}`. A `BigInt` throws. `createJsonEncoder<T>()` and `createJsonDecoder<T>()` know your type, so they fix all of that.

## Three encoder strategies

`createJsonEncoder` takes a `strategy`. They differ in how they walk your value and what they do with undeclared keys.

<code-import path="packages/examples/src/guide/json-strategies.ts" lang="ts" commentStart="// start-strategies" commentEnd="// end-strategies" />

| Strategy | Allocates a clone? | Mutates input? | Undeclared keys |
| --- | --- | --- | --- |
| `clone` *(default)* | Yes — built from the declared shape | No | Dropped (for free) |
| `mutate` | No | **Yes** | Kept on the wire |
| `direct` | No | No | Dropped |

::tip
`clone` is the safe default — it never touches your input and drops stray keys by building a fresh value from the type. Reach for `mutate` only in a hot path where the allocation actually shows up in a profile, and you're fine mutating the object you pass in.
::

## The decoder returns `DataOnly<T>`

`createJsonDecoder<T>()` doesn't return `T` — it returns `DataOnly<T>`. A decoded value was rebuilt from JSON, so it can only hold serializable data. Anything that can't ride JSON (methods, for one) was never on the wire, so TypeScript drops it from the return type and stops you from reaching for it.

<code-import path="packages/examples/src/guide/json-dataonly.ts" lang="ts" commentStart="// start-dataonly" commentEnd="// end-dataonly" />

This isn't a runtime cost — it's just the type telling the truth about what came back. More on the why in the [Validate contract](/guide/validate-contract).
