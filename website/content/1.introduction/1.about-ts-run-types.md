TypeScript decided it's "just a linter" and throws your types away before your code ever runs. **ts-run-types** reads them first — at build time — and hands the runtime back what it lost: validators, JSON + binary (de)serializers, mock data and reflection.

Plenty of libraries generate runtime checks. This page isn't a feature tour — it's the handful of **design decisions** that make ts-run-types what it is.

## Driven by TypeScript, nothing else

The runtime model is exactly what the TypeScript type system can express — no more, no less. The goal is to support the *whole* type system: atomics, objects, arrays, tuples, unions, intersections, template literals, generics, the utility types, `Map`/`Set`/`Promise`/`Date`/`Temporal`, recursive and circular shapes.

The flip side is just as deliberate: **if something isn't part of TypeScript's type system, it's out of scope by design.** We don't invent a parallel "schema dialect" with concepts TypeScript doesn't have. Your types *are* the schema — there's no second language to learn and nothing to keep in sync.

That principle is why the two ways to describe a shape are really one. A plain type and an `RT.*` schema builder are two doors into the same engine — they resolve to the identical structural representation:

## One type, one id

Every type collapses to a single, stable **structural id** — a short hash of its shape. Two types with the same shape resolve to the *same* id, and ts-run-types deduplicates them twice over: once by pointer identity (the same type reached two ways) and once by structural id (two distinct types with identical shapes). Both fold into a single cache entry.

<code-import path="packages/examples/src/introduction/one-type-one-id.ts" lang="ts" />

This idempotency is the quiet engine behind everything else:

- **One source of truth, no drift.** The validator and the serializer are generated from the same shape, so they can never disagree about what your type means. Change the type, every generated function changes with it.
- **Stable, reproducible output.** The same type always emits the same id and the same code, build after build.
- **Ship only what you use.** Caches are *demand-driven*: a function family holds only the types its own call sites actually request. A file that just reflects an id ships zero validation code.
- **Native tree-shaking.** Every cache entry is its own module, so bundlers code-split and drop what you never call — no central registry to defeat them.

## Build time, not run time

TypeScript 7 ships the compiler as a compiled **Go binary**. The legacy custom-transformer hook that runtime-reflection libraries used to plug into ([microsoft/typescript-go#516](https://github.com/microsoft/typescript-go/issues/516)) was never ported, and the compiler can no longer be monkey-patched from Node.

So ts-run-types reads your types through a **native Go side-channel** straight into tsgo's type checker — what we resolve as type `T` is exactly what tsgo resolves. The work older libraries did *lazily, at first call* becomes ahead-of-time Go emit: each `createValidate<T>()` is a specialized function written out during the build.

::note
There's no reflection happening when your app is live, and the published Vite plugin carries **zero runtime dependencies**. The only thing in your bundle is the small `@mionjs/ts-go-run-types` runtime — and the functions you actually called.
::

## Fast by construction

The performance comes from the architecture, not micro-tuning:

- **A Go compiler underneath.** Type resolution rides the same native `Program` view tsgo uses.
- **No first-call cost.** Everything is emitted ahead of time, so there's nothing to compile when your code runs.
- **Byte-offset rewrites with real source maps**, content-addressed immutable modules, and a compact binary codec for the paths where every byte counts.

## Proven by its test suite

The port is backed by a deep test suite — a Go suite (70+ test files) and a TypeScript suite (90+ test files) that spawns the real binary and asserts the full round-trip, covering everything from atomic kinds to recursive and circular types.

A couple of rules keep it honest: every marker API is tested in **both** call shapes — static `getRunTypeId<T>()` and value-first `getRunTypeId(value)` — with a hash-equivalence assertion that the two agree; and the "this transform is a no-op" optimisation has a one-directional soundness contract pinned by a corpus test.

And you don't have to take our word for it. The [**Suites**](/suites/validation) on this very site are generated *straight from that test suite* — every case, the type it came from, the equivalent schema, and the actual JavaScript the build emits. The [**Benchmarks**](/benchmarks/validation) show each competitor's real source, so you can judge the comparison yourself.

## A deliberate contract: serializable data

`createValidate`, the JSON codec and the binary codec all operate on the **serializable projection** of your type. Functions, methods and symbol keys are dropped (with a build-time warning), because they don't survive JSON anyway.

That projection has a name in the types, too: **`DataOnly<T>`** — a type-level mapping that strips the non-serializable members from `T`. It isn't just a runtime convention. `createValidate<T>()` narrows to `DataOnly<T>`, and `createJsonDecoder<T>()` / `createBinaryDecoder<T>()` **return `DataOnly<T>`** — so a value that passes validation, or comes back from a decoder, is strongly typed as exactly the data. The dropped methods and functions are gone at compile time, not just at runtime.

That's a design choice, not a shortcoming: it matches how these tools are actually used — RPC, persistence, network IO — and it keeps the cache bound to *shapes*, not live JavaScript values, which is exactly the coupling we left `tsc` to escape.
