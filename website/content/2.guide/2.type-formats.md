## What's in the box

Formats come in four families. Here are the named ones (there are more ‚Äî these are the ones you'll reach for):

| Family | A few of the named formats |
| --- | --- |
| **String** | `email`, `uuidv4`, `uuidv7`, `url`, `ipv4`, `ipv6`, `domain`, `alpha`, `numeric`, `lowercase` |
| **Number** | `int8`, `int16`, `int32`, `uint8`, `uint16`, `uint32`, `integer`, `float`, `positive`, `negative` |
| **BigInt** | `bigInt64`, `bigUInt64`, `bigPositive`, `bigNegative`, `bigPositiveInt`, `bigNegativeInt` |
| **Date / time** | three representations ‚Äî string-encoded (`stringDate`, `stringTime`, `stringDateTime`), native `date`, and the TC39 `temporal.*` types ‚Äî all sharing the same bounds (see below) |

Type-first, the alias name is `Format` + the capitalized name: `FormatEmail`, `FormatInt32`, `FormatBigInt64`, `FormatStringDate`. Schema-first, it's `RT.email()`, `RT.int32()`, `RT.bigInt64()`, `RT.stringDate()`.

::note
The fixed-width number/bigint formats (`int32`, `bigUInt64`, ‚Ä¶) aren't just validation ‚Äî they also tell the [binary codec](/guide/serialization) how many bytes to pack. A `FormatUInt8` field rides the wire in a single byte.
::

## Dates, times and Temporal

A date-ish value shows up in three shapes, and there's a format family for each ‚Äî all taking the **same** `min` / `max` / `gt` / `lt` bounds, so a constraint means the same thing whichever representation you pick:

- **String-encoded** ‚Äî `FormatStringDate`, `FormatStringTime`, `FormatStringDateTime` check an ISO string in a fixed layout (`'2020-01-01'`, `'08:30'`).
- **Native `Date`** ‚Äî `FormatDate` checks a real JS `Date`; `FormatDateFuture` / `FormatDatePast` are ready-made (a `Date` that's `>= now` / `<= now`).
- **TC39 Temporal** ‚Äî `FormatTemporalPlainDate`, `FormatTemporalZonedDateTime`, `FormatTemporalInstant`, `FormatTemporalPlainTime`, `FormatTemporalPlainDateTime`, `FormatTemporalPlainYearMonth` check an actual `Temporal.*` instance. Opt in from the dedicated `ts-runtypes/formats/temporal` subpath, so consumers who don't use Temporal never pull in its lib.

```ts
import type {FormatStringDate} from 'ts-runtypes/formats';
import type {FormatTemporalPlainDate} from 'ts-runtypes/formats/temporal';

// The same "2020 or later" bound, two representations:
type DateString = FormatStringDate<{min: '2020-01-01'}>;       // an ISO string
type PlainDate = FormatTemporalPlainDate<{min: '2020-01-01'}>; // a Temporal.PlainDate
```

The bound literal is written in the type's own ISO form ‚Äî a `PlainDate` takes `'2020-01-01'`, an `Instant` takes `'2020-01-01T00:00:00Z'`. (`Temporal.PlainMonthDay` and `Temporal.Duration` have no min/max ordering, so they're validated by identity only, with no bounds.)

Schema-first, the same builders live under `RT.date(...)` and the `RT.temporal.*` namespace:

```ts
import * as RT from 'ts-runtypes/schema';

RT.date({max: 'now'});                       // a Date in the past
RT.temporal.plainDate({min: '2020-01-01'});  // a Temporal.PlainDate, 2020 on
RT.temporal.instant({min: 'now-PT1H'});      // an Instant within the last hour
```

::note
Temporal values are validated by **identity** (`instanceof`), not by structural shape ‚Äî and they survive `DataOnly` whole, so `DataOnly<Temporal.PlainDate>` stays a `Temporal.PlainDate`.
::

## Relative bounds with `now`

A bound doesn't have to be an absolute date. Write `now`, or `now` plus or minus an **ISO-8601 duration** (`now+P‚Ä¶` / `now-P‚Ä¶`), and the build resolves it against the current time every time it validates:

<code-import path="packages/examples/src/guide/type-formats-relative.ts" lang="ts" />

The `P‚Ä¶` tail is a full ISO-8601 duration ‚Äî `P1Y2M10D` (1 year, 2 months, 10 days), `P1W` (1 week), `PT12H30M` (12 hours, 30 minutes). The build checks the units fit the field: a date-only format takes only date units, a time-only format only time units.

| Family | Relative units |
| --- | --- |
| Date-only (`stringDate`, `temporal.plainDate`, `temporal.plainYearMonth`) | date ‚Äî `Y` `M` `W` `D` |
| Time-only (`stringTime`, `temporal.plainTime`, `temporal.instant`) | time ‚Äî `H` `M` `S` |
| Date + time (`stringDateTime`, native `date`, `temporal.plainDateTime`, `temporal.zonedDateTime`) | both |

`min` / `max` are inclusive; `gt` / `lt` are the exclusive twins. Give each edge as one or the other, never both.

## Branded (nominal) types

By default a format is still structurally a `string` or `number` ‚Äî handy, but it won't stop you from passing a raw string where a `UserId` belongs. Add a **brand name** (the second type argument) and the format becomes *nominal*: nothing else is assignable to it without an explicit `as` cast.

<code-import path="packages/examples/src/guide/type-formats-branded.ts" lang="ts" />

::tip
The `as` cast is a feature, not a chore. It marks the exact line where you've decided a raw value is now a `UserId` ‚Äî usually right after you've validated it. Everywhere else, the type system keeps your ids, cents and timestamps from getting mixed up.
::

Schema-first, brand with the `brand()` tag: `RT.string({minLength: 1}, RT.brand('UserId'))`.

## Custom formats

No named format fits? `FormatString`, `FormatNumber` and `FormatBigInt` are the escape hatches ‚Äî pass your own params.

<code-import path="packages/examples/src/guide/type-formats-custom.ts" lang="ts" />

The common params:

| Format | Params |
| --- | --- |
| `FormatString` | `minLength`, `maxLength`, `length`, `pattern`, `allowedChars`, `allowedValues`, ‚Ä¶ |
| `FormatNumber` | `min`, `max`, `gt`, `lt`, `integer`, `float`, `multipleOf` |
| `FormatBigInt` | `min`, `max`, `gt`, `lt`, `multipleOf` (all `bigint` literals, e.g. `0n`) |

`min`/`max` are inclusive; `gt`/`lt` are their exclusive twins. A bound is one or the other, never both ‚Äî the build rejects `{min: 0, gt: 0}`.

::note
A `pattern` always needs `mockSamples` ‚Äî canonical valid values the mock generator can draw from (a bare regex with no samples is rejected). Reusing a pattern across types? Register it once with [`registerFormatPattern`](/guide/pure-functions) and reference it by `typeof`.
::
