## The full story — `createGetValidationErrors`

Same checks, but instead of a boolean you get an array of what failed. Each entry is `{path, expected}` — where it broke and what was expected there.

<code-import path="packages/examples/src/guide/validation-basics.ts" lang="ts" commentStart="// start-errors" commentEnd="// end-errors" />

An empty array means the value is valid. Every error carries a `path` (the keys / indices to the bad value) and an `expected` string (the type that should've been there). Format failures add a `format` detail too.

## Three ways to call it

Every factory takes your type three different ways. They all resolve to the same generated function — use whichever fits the call.

<code-import path="packages/examples/src/guide/validation-three-forms.ts" lang="ts" commentStart="// start-forms" commentEnd="// end-forms" />

- **Type-first** — you supply `<T>`, no value needed.
- **Value-first** — `T` is inferred from a value you already have lying around.
- **Schema-first** — pass an `RT.*` schema; `T` comes from the schema.

## Tuning with `ValidateOptions`

`createValidate` and `createGetValidationErrors` take an options object as a literal. The build reads the literal and routes you to a specialized variant — nothing is parsed at runtime.

<code-import path="packages/examples/src/guide/validation-options.ts" lang="ts" commentStart="// start-options" commentEnd="// end-options" />

| Option | What it does |
| --- | --- |
| `noLiterals` | Literal checks degrade to their base type — `literal 'a'` accepts any string, `literal 2` accepts any finite number. |
| `noIsArrayCheck` | Skips the leading `Array.isArray()` guard on array validators, for when you've already proven the value is an array upstream. |

::warning
Pass the options as an **object literal at the call site**. It's read at build time, so a variable won't work — the build can't see its value.
::

## One thing to know: validators check serializable data

`createValidate` validates the **serializable** projection of your type, not every last TypeScript member. Functions, methods and symbol keys get silently dropped — they don't survive JSON anyway, so the validator ignores them. A `{name: string; onClick: () => void}` produces a validator that only checks `name`.

This is on purpose, and the build tells you when it happens (a `VL010` warning, not an error). The full rules live in the [Validate contract](/guide/validate-contract).
