The rule: **validators and serializers operate on serializable data, not the full TypeScript type.** Functions, methods, symbols, symbol-keyed props, and getters/setters with no backing data aren't data — JSON drops them on the wire anyway — so ts-run-types drops them too.

This applies to `createValidate`, `createGetValidationErrors`, and the JSON / binary families alike.

## Serializable data only

Why this and not "check the whole type"? Because the typical job is RPC, persistence, network IO — and the moment your object hits `JSON.stringify`, every function and symbol on it vanishes. The data-only projection of `T` *is* what actually crosses the wire. Validating that is validating the thing you care about.

So `createValidate<User>()` checks the data shape of `User`. The methods aren't part of the contract.

`onClick` is a function, so it's not data, so the validator never looks at it. `isUser({name: 'Ada', onClick: 'not-a-function'})` returns `true` — because as far as serializable data goes, it's a valid `User`.

At build time you'll see a `VL010` **Warning** telling you `onClick` was dropped. That's the system working as intended — not an error to chase.

## Warning vs Error

One clean line splits the two outcomes:

::note
**Warning** = an expected drop. Fine. Carry on.
**Error** = it will throw at runtime. The build halts so you fix it.
::

Which one you get depends on *where* the non-serializable member sits:

- **At a property** (like `onClick` above) → it's silently dropped with a **Warning** (`VL010` / `VE010` and friends). The rest of the object validates normally.
- **At a root, or a position that propagates** — an array element, a tuple slot, a union member, a function param or return — there's nothing to drop *into*. So the build emits an **Error**-severity diagnostic and the generated function `alwaysThrow`s at runtime. The build fails, on purpose. A `() => void[]` has no serializable shape to fall back to, and silently "validating" it would be a lie.

So: a non-serializable *field* is a shrug. A non-serializable *whole value* is a hard stop.

## Decoders return DataOnly&lt;T&gt;

The type system stays honest about all this. `createJsonDecoder<T>()` and `createBinaryDecoder<T>()` don't return `T` — they return `DataOnly<T>`, the same projection the runtime produces. The dropped members are gone from the return type too, so you can't accidentally call a method that isn't there.

<code-import path="packages/examples/src/guide/validate-contract-dataonly.ts" lang="ts" />

Encoders are unchanged — you hand them a full `T`. It's the *decoded* value that's data-only, because that's all that survived the trip.

## The one-line summary

Think "is this JSON-safe?" If yes, it's checked. If no, it's dropped (Warning) — unless dropping it would be meaningless, in which case the build stops you (Error). That's the whole contract.
