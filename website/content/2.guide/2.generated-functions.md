Every `createX<T>()` factory turns a type into a purpose-built function the build writes out for you — no schema object, no runtime reflection. They all share one calling convention and one contract, so once you've seen one you've seen them all.

This page covers the **data functions**: validation, unknown-key handling, JSON and binary. ([Mocking](/guide/mocking) and [reflection](/guide/markers-and-reflection) get their own pages.)

## Three ways to call any factory

Every factory takes your type three different ways. They all resolve to the same generated function — use whichever fits the call site.

<code-import path="packages/examples/src/guide/validation-three-forms.ts" lang="ts" commentStart="// start-forms" commentEnd="// end-forms" />

- **Type-first** — you supply `<T>`, no value needed.
- **Value-first** — `T` is inferred from a value you already have.
- **Schema-first** — pass an `RT.*` schema; `T` comes from the schema.

The rest of this page uses whichever form reads best for the example — but every one of them accepts all three.

### Tuning with `ValidateOptions`

Both validators take an options object as a **literal at the call site**. The build reads the literal and routes you to a specialized variant — nothing is parsed at runtime.

<code-import path="packages/examples/src/guide/validation-options.ts" lang="ts" commentStart="// start-options" commentEnd="// end-options" />

| Option | What it does |
| --- | --- |
| `noLiterals` | Literal checks degrade to their base type — `literal 'a'` accepts any string, `literal 2` accepts any finite number. |
| `noIsArrayCheck` | Skips the leading `Array.isArray()` guard on array validators, for when you've already proven the value is an array upstream. |

::warning
Pass the options as an **object literal** at the call site. It's read at build time, so a variable won't work — the build can't see its value.
::

## Unknown keys

Four tools for properties that aren't in your declared type. These check for *extra* keys — to check that the *declared* keys hold the *right types*, that's validation above.

| Factory | What it does |
| --- | --- |
| `createHasUnknownKeys` | Predicate — `true` if the value carries any undeclared key. |
| `createStripUnknownKeys` | Deletes undeclared keys in place, returns the same reference. |
| `createUnknownKeyErrors` | One `{path, expected: 'never'}` per undeclared key — same shape as `createGetValidationErrors`. |
| `createUnknownKeysToUndefined` | Keeps the key but sets it to `undefined` instead of deleting it. |

### Three encoder strategies

`createJsonEncoder` takes a `strategy` (a call-site literal) that decides how it walks your value and what it does with undeclared keys.

<code-import path="packages/examples/src/guide/json-strategies.ts" lang="ts" commentStart="// start-strategies" commentEnd="// end-strategies" />

| Strategy | Allocates a clone? | Mutates input? | Undeclared keys |
| --- | --- | --- | --- |
| `clone` *(default)* | Yes — built from the declared shape | No | Dropped (for free) |
| `mutate` | No | **Yes** | Kept on the wire |
| `direct` | No | No | Dropped |

::tip
`clone` is the safe default — it never touches your input and drops stray keys by building a fresh value from the type. Reach for `mutate` only in a hot path where the allocation shows up in a profile, and you're fine mutating the object you pass in.
::

## Binary encode / decode

`createBinaryEncoder` / `createBinaryDecoder` are the same idea, smaller and faster on the wire. The encoder returns an `ArrayBuffer`; the decoder takes one back — and like JSON, the codec knows your type, so a `Date` round-trips cleanly.

<code-import path="packages/examples/src/guide/binary-basics.ts" lang="ts" commentStart="// start-roundtrip" commentEnd="// end-roundtrip" />

Reach for binary when you're moving lots of records and both ends are yours (WebSockets, game state, IoT). Stick with JSON when a human or a third party reads the payload, or you need it `curl`-able.

### Reuse the buffer in hot loops

By default each encode allocates a fresh backing buffer. In a tight loop that adds up — so build a `DataViewSerializer` once with `createDataViewSerializer` and hand it to the encoder to reuse the same buffer. `createDataViewDeserializer` does the same on the way back.

::note
You only need these helpers for buffer reuse in performance-sensitive code. For everyday encode/decode, the plain `createBinaryEncoder` / `createBinaryDecoder` calls handle their own buffers.
::

## Decoders return `DataOnly<T>`

Both `createJsonDecoder<T>()` and `createBinaryDecoder<T>()` return `DataOnly<T>`, not bare `T`. A value rebuilt from JSON or bytes can only hold serializable data — anything that can't ride the wire (methods, for one) was never there — so the return type leaves it out and stops you from reaching for it.

<code-import path="packages/examples/src/guide/json-dataonly.ts" lang="ts" commentStart="// start-dataonly" commentEnd="// end-dataonly" />

This isn't a runtime cost — it's the type telling the truth about what came back.

## One contract: serializable data only

Validation and both codecs operate on the **serializable projection** of your type, not every last TypeScript member. Functions, methods and symbol keys are silently dropped — they don't survive JSON anyway, so a `{name: string; onClick: () => void}` produces a validator that only checks `name`.

This is on purpose, and the build tells you when it happens — a `VL010`-family **Warning**, not an error. (At a position that would throw at runtime, like a union member or array element, it's escalated to an **Error** and the build fails instead.) The reasoning is in [About ts-run-types](/introduction/about-ts-run-types#a-deliberate-contract-serializable-data).
