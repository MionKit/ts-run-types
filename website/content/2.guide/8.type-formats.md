## What's in the box

Formats come in four families. Here are the named ones (there are more — these are the ones you'll reach for):

| Family | A few of the named formats |
| --- | --- |
| **String** | `email`, `uuidv4`, `uuidv7`, `url`, `ipv4`, `ipv6`, `domain`, `alpha`, `numeric`, `lowercase` |
| **Number** | `int8`, `int16`, `int32`, `uint8`, `uint16`, `uint32`, `integer`, `float`, `positive`, `negative` |
| **BigInt** | `bigInt64`, `bigUInt64`, `bigPositive`, `bigNegative`, `bigPositiveInt`, `bigNegativeInt` |
| **Date / time** | `stringDate`, `stringTime`, `stringDateTime` (string-encoded dates with layout + min/max bounds) |

Type-first, the alias name is `Format` + the capitalized name: `FormatEmail`, `FormatInt32`, `FormatBigInt64`, `FormatStringDate`. Schema-first, it's `RT.email()`, `RT.int32()`, `RT.bigInt64()`, `RT.stringDate()`.

::note
The fixed-width number/bigint formats (`int32`, `bigUInt64`, …) aren't just validation — they also tell the [binary codec](/guide/binary) how many bytes to pack. A `FormatUInt8` field rides the wire in a single byte.
::

## Branded (nominal) types

By default a format is still structurally a `string` or `number` — handy, but it won't stop you from passing a raw string where a `UserId` belongs. Add a **brand name** (the second type argument) and the format becomes *nominal*: nothing else is assignable to it without an explicit `as` cast.

<code-import path="packages/examples/src/guide/type-formats-branded.ts" lang="ts" />

::tip
The `as` cast is a feature, not a chore. It marks the exact line where you've decided a raw value is now a `UserId` — usually right after you've validated it. Everywhere else, the type system keeps your ids, cents and timestamps from getting mixed up.
::

Schema-first, brand with the `brand()` tag: `RT.string({minLength: 1}, RT.brand('UserId'))`.

## Custom formats

No named format fits? `FormatString`, `FormatNumber` and `FormatBigInt` are the escape hatches — pass your own params.

<code-import path="packages/examples/src/guide/type-formats-custom.ts" lang="ts" />

The common params:

| Format | Params |
| --- | --- |
| `FormatString` | `minLength`, `maxLength`, `length`, `pattern`, `allowedChars`, `allowedValues`, … |
| `FormatNumber` | `min`, `max`, `gt`, `lt`, `integer`, `float`, `multipleOf` |
| `FormatBigInt` | `min`, `max`, `gt`, `lt`, `multipleOf` (all `bigint` literals, e.g. `0n`) |

`min`/`max` are inclusive; `gt`/`lt` are their exclusive twins. A bound is one or the other, never both — the build rejects `{min: 0, gt: 0}`.

::note
A `pattern` always needs `mockSamples` — canonical valid values the mock generator can draw from (a bare regex with no samples is rejected). Reusing a pattern across types? Register it once with [`registerFormatPattern`](/guide/custom-serializers-and-pure-fns) and reference it by `typeof`.
::
